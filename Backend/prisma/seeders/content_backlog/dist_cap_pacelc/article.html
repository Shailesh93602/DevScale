Understanding CAP and PACELC Theorems

In the world of distributed systems, you cannot have everything. These two theorems define the fundamental trade-offs that every architect must make when designing globally scalable software.

1. The CAP Theorem

Introduced by Eric Brewer, CAP states that a distributed data store can only provide two out of three guarantees:

  • Consistency: Every read receives the most recent write (or an error).
  • Availability: Every request receives a (non-error) response, without the guarantee that it contains the most recent write.
  • Partition Tolerance: The system continues to operate despite an arbitrary number of messages being dropped (or delayed) by the network between nodes.

Crucial Rule: Because network partitions ARE a reality ("P" is mandatory), you must choose between Consistency (CP) or Availability (AP).

2. PACELC: The Extended Theorem

PACELC takes it further: If there is a partition (P), how does the system trade off Availability (A) and Consistency (C)? Else (E), when the system is running normally, how does it trade off Latency (L) and Consistency (C)?

For example, MongoDB is often configured as **PA/EC**: It prioritizes availability during partitions, but emphasizes consistency during normal operations.

Conclusion

Mastering CAP and PACELC allows you to choose the right database for the job. If you're building a bank, you choose **Consistency**. If you're building a social media feed, you choose **Availability** and **Low Latency**.
