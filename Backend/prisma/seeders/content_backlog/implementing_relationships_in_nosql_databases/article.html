Implementing Database Relationships in NoSQL

While often described as "schema-less," NoSQL databases still have relationships. The primary ways to model these are Embedding and Referencing.

1. Embedding (Denormalization)

Storing related data within a single document. This is ideal for 1:1 and small 1:N relationships where the related data is always queried together.


{
  "_id": "60c...1",
  "name": "Jane User",
  "address": {
    "street": "123 Main St",
    "city": "Nowhere"
  }
}

2. Referencing (Normalization)

Storing the ID of a document from another collection. This is similar to a foreign key in SQL and is better for large 1:N and all M:N relationships.


// Book Document
{
  "_id": "1",
  "title": "Mastering NoSQL",
  "author_id": "8" // Reference to a document in another collection
}

3. Mongoose Population

For referencing, ODMs like Mongoose allow you to "populate" the ID with the actual document during a query.

4. Which one to choose?

  • Use Embedding if the related data is small, doesn't change often, and is always queried with the parent.
  • Use Referencing if the related data can grow large, is updated frequently, or needs the same record to be linked across multiple parents.

Conclusion

Understanding the trade-offs between embedding and referencing is the key to designing high-performance NoSQL database schemas.
