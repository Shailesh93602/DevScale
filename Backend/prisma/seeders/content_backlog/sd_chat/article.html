System Design: Building a Global Chat (WhatsApp Logic)

Designing a real-time chat for 2 billion users requires solving the "Consistency vs. Availability" trade-off while ensuring message delivery with zero latency.

1. Real-Time Connection: WebSockets

Standard HTTP doesn't work for chat because the server can't "push" a message to a client. We use **WebSockets** for a permanent, bi-directional connection. When User A sends a message, it goes to the "Chat Service," which instantly pushes it to User B's open socket.

2. Handling Offline Users

If User B is offline, the message is stored in a **Message Store** (usually a NoSQL database like Cassandra for its high write speed). When User B comes online, the system performs a "Sync" to pull all pending messages.

3. Scalability: The Gateway and Service Discovery

With billions of users, you cannot have one server. You use an **API Gateway** and **Push Clusters**. These servers track which user is connected to which server node (using a Cache like Redis) so the message can be routed across the global network in milliseconds.

Conclusion

Designing a global chat system teaches you the fundamentals of high-concurrency engineering, real-time protocols, and the critical importance of a distributed "Presence" system.
