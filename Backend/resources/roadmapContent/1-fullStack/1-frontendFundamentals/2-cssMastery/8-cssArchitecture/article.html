CSS Architecture & Methodologies

As CSS codebases grow, they become difficult to maintain without a clear organizational strategy. CSS methodologies provide naming conventions and structural rules that keep stylesheets scalable and team-friendly.

BEM — Block, Element, Modifier

BEM is the most widely adopted CSS methodology. It uses a strict naming convention:

/* Block — standalone component */
.card { }

/* Element — part of a Block, uses double underscore */
.card__title { }
.card__image { }
.card__body { }
.card__footer { }

/* Modifier — variant of Block or Element, uses double dash */
.card--featured { }
.card__title--large { }
.card--dark { }

In HTML:

<article class="card card--featured">
  <img class="card__image" src="..." alt="...">
  <div class="card__body">
    <h2 class="card__title card__title--large">Title</h2>
    <p class="card__text">Content...</p>
  </div>
  <footer class="card__footer">...</footer>
</article>

SMACSS — Scalable and Modular Architecture for CSS

SMACSS categorizes CSS rules into 5 types:

  • Base — Element defaults (resets, typography). No classes.
  • Layout — Page structure (header, sidebar, grid). Prefix: l-
  • Module — Reusable UI components (cards, buttons, forms).
  • State — JavaScript-driven states. Prefix: is- (e.g., is-active, is-hidden)
  • Theme — Colour/font overrides for different themes.

CSS Modules

CSS Modules (used in React with Vite/Next.js) scope class names locally — eliminating global class name collisions:

/* Button.module.css */
.button {
  background: var(--color-primary);
  padding: 0.5rem 1rem;
  border-radius: 0.375rem;
}

.button--danger {
  background: #ef4444;
}
/* Button.jsx */
import styles from './Button.module.css';

export function Button({ variant, children }) {
  return (
    <button className={`${styles.button} ${variant === 'danger' ? styles['button--danger'] : ''}`}>
      {children}
    </button>
  );
}

CSS Modules transform class names to something like Button_button__3xkY8 at build time, guaranteeing uniqueness.

Utility-First CSS

The utility-first approach (popularized by Tailwind CSS) uses single-purpose, immutable classes directly in HTML:

<!-- Traditional component-based -->
<button class="btn btn--primary btn--lg">Submit</button>

<!-- Utility-first (Tailwind-style) -->
<button class="bg-indigo-500 hover:bg-indigo-600 text-white font-semibold px-6 py-2 rounded-lg">Submit</button>

Organizing CSS Files

A recommended folder structure for a medium-to-large project:

styles/
  ├── tokens/          # CSS variables (colour, spacing, typography)
  │   └── variables.css
  ├── base/            # Reset, typography defaults
  │   ├── reset.css
  │   └── typography.css
  ├── components/      # Individual UI components
  │   ├── button.css
  │   ├── card.css
  │   └── modal.css
  ├── layouts/         # Page layouts and grid systems
  │   └── grid.css
  └── main.css         # Entry point — imports everything

The 7-1 Pattern

Popular for Sass projects: 7 folders + 1 main file:

sass/
  ├── abstracts/    # Variables, mixins, functions
  ├── base/         # Reset, typography
  ├── components/   # Buttons, inputs, cards
  ├── layout/       # Header, footer, grid
  ├── pages/        # Page-specific styles
  ├── themes/        # Dark mode, brand themes
  ├── vendors/       # Third-party CSS
  └── main.scss     # @use all seven folders

Best Practices

  • Choose one methodology and apply it consistently across your team.
  • BEM + CSS Variables is an excellent combination for most projects.
  • Use CSS Modules or CSS-in-JS in React apps to prevent naming collisions.
  • Group related rules and add comments for context.
  • Avoid deeply nested selectors — they are hard to override and debug.
