Error Handling & Debugging in JavaScript

Robust applications handle failures gracefully. JavaScript provides structured mechanisms for catching errors, creating custom error types, and debugging issues in production.

try / catch / finally

try {
  const data = JSON.parse(invalidJsonString); // throws SyntaxError
  processData(data);
} catch (error) {
  // error is an Error object with: name, message, stack
  console.error(`${error.name}: ${error.message}`);
  // Handle gracefully (show user feedback, log to service)
} finally {
  // Always runs — cleanup code (close connections, hide spinners)
  hideLoadingSpinner();
}

The Error Object

JavaScript has several built-in error types:

  • Error — Generic base type
  • SyntaxError — Invalid JavaScript syntax
  • TypeError — Wrong type (e.g., calling null as a function)
  • ReferenceError — Accessing an undefined variable
  • RangeError — Value out of valid range
  • URIError — Bad URI encoding

Custom Error Classes

class ValidationError extends Error {
  constructor(field, message) {
    super(message);
    this.name = 'ValidationError';
    this.field = field;
  }
}

class NetworkError extends Error {
  constructor(statusCode, message) {
    super(message);
    this.name = 'NetworkError';
    this.statusCode = statusCode;
  }
}

// Usage
function validateAge(age) {
  if (typeof age !== 'number') throw new ValidationError('age', 'Age must be a number');
  if (age < 0 || age > 150) throw new ValidationError('age', 'Age must be between 0 and 150');
}

try {
  validateAge('twenty');
} catch (error) {
  if (error instanceof ValidationError) {
    console.log(`Field "${error.field}" failed: ${error.message}`);
  } else {
    throw error; // re-throw unexpected errors
  }
}

Async Error Handling

// With async/await
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new NetworkError(response.status, `HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    if (error instanceof NetworkError) {
      if (error.statusCode === 404) return null;
      throw error; // re-throw for caller to handle
    }
    throw error;
  }
}

// With Promise chains
fetch('/api/data')
  .then(res => res.json())
  .catch(err => console.error('Fetch failed:', err))
  .finally(() => setLoading(false));

Global Error Handlers

// Catch uncaught synchronous errors
window.addEventListener('error', (event) => {
  console.error('Unhandled error:', event.error);
  // Log to monitoring service (Sentry, DataDog, etc.)
});

// Catch unhandled promise rejections
window.addEventListener('unhandledrejection', (event) => {
  console.error('Unhandled promise rejection:', event.reason);
  event.preventDefault(); // prevent default console warning
});

Debugging with Chrome DevTools

  • console.log / warn / error / table — Basic logging with different severity levels.
  • debugger — Insert a breakpoint in code: debugger;
  • Breakpoints — Click line numbers in Sources panel to pause execution.
  • Watch expressions — Monitor variable values in the Scope panel.
  • console.time / timeEnd — Measure execution time: 
    console.time('sort');
arr.sort();
console.timeEnd('sort'); // sort: 2.034ms
  • console.trace() — Print a stack trace to the current call location.

Defensive Programming Patterns

// Optional chaining — avoid TypeError: Cannot read property of undefined
const city = user?.address?.city ?? 'Unknown';

// Nullish coalescing — safe default values
const limit = config.limit ?? 10;

// Guard clauses — fail fast
function processOrder(order) {
  if (!order) throw new Error('Order is required');
  if (!order.items?.length) throw new ValidationError('items', 'Order must have items');
  // happy path below
}

Best Practices

  • Always handle errors in async functions — never leave a Promise without .catch().
  • Create custom error classes for domain-specific errors.
  • Use instanceof to distinguish error types.
  • Re-throw errors you can't handle locally.
  • Use optional chaining (?.) and nullish coalescing (??) for defensive access.
