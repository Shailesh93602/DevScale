JavaScript Modules & the Browser Event Loop

Two essential concepts for writing scalable JavaScript: ES Modules for organizing code into independent files, and the Event Loop — the engine that makes JavaScript's asynchronous behaviour possible.

ES Modules

// math.js — named exports
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }

// utils.js — default export
export default function formatCurrency(amount, currency = 'USD') {
  return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(amount);
}

// main.js — importing
import formatCurrency from './utils.js';           // default import
import { add, multiply, PI } from './math.js';      // named imports
import * as Math from './math.js';                  // namespace import
import { add as sum } from './math.js';             // renamed import

// Dynamic import (lazy loading)
const { heavyLibrary } = await import('./heavy.js');  // only loads when needed

Module Features

  • Modules are always in strict mode — no need for 'use strict'.
  • Each module has its own scope — top-level variables are not global.
  • Modules are singletons — they execute only once and cache their exports.
  • Use type="module" when referencing modules in HTML: <script type="module" src="main.js"></script>

The JavaScript Event Loop

JavaScript is single-threaded — it can only execute one piece of code at a time. The Event Loop is the mechanism that allows it to handle asynchronous operations without blocking.

// Execution order example
console.log('1. Start');            // synchronous — runs immediately

setTimeout(() => {
  console.log('4. setTimeout');     // goes to Macrotask Queue
}, 0);

Promise.resolve().then(() => {
  console.log('3. Promise');        // goes to Microtask Queue
});

console.log('2. End');              // synchronous — runs immediately

// Output: 1, 2, 3, 4

The Call Stack, Queues, and Web APIs

The Event Loop coordinates three areas:

  • Call Stack — Where all synchronous code executes (LIFO structure).
  • Web/Browser APIs — Where async operations run (setTimeout, fetch, DOM events). These are NOT in JavaScript — they're in the browser.
  • Microtask Queue — High priority. Processes Promises (.then, .catch), queueMicrotask(), and MutationObserver callbacks. Fully drained before next macrotask.
  • Macrotask Queue (Task Queue) — Lower priority. Processes setTimeout, setInterval, I/O callbacks.

Event Loop Algorithm

  1. Execute all code in the Call Stack until it's empty.
  2. Process ALL pending microtasks (Promise callbacks). If microtasks add more microtasks, keep processing.
  3. Render the page if needed.
  4. Pick ONE macrotask (setTimeout/setInterval callback) and execute it.
  5. Go back to step 2.

Why This Matters

// Never do this — blocks the event loop for 3 seconds!
const start = Date.now();
while (Date.now() - start < 3000) {}  // CPU spins, UI freezes

// Do this instead — non-blocking
setTimeout(() => {
  console.log('3 seconds passed');
}, 3000);

Module Bundlers

In production, modules are compiled/bundled by tools that optimize the output:

  • Vite — Fast dev server using native ES modules. Uses Rollup for production builds.
  • Webpack — Industry veteran. Highly configurable, handles all asset types.
  • esbuild — Extremely fast, written in Go. Used internally by many tools.
  • Rollup — Optimized for libraries and tree-shaking.

Best Practices

  • Prefer named exports over default exports — better for refactoring and autocomplete.
  • Use dynamic imports for code splitting and lazy loading large libraries.
  • Avoid blocking the main thread with long synchronous operations — use Web Workers.
  • Understand that Promise callbacks always run after the current synchronous code completes.
