Understanding and Handling CORS

If you've ever built a full-stack application where the frontend and backend are hosted on different domains (or even different ports on localhost), you've likely encountered the dreaded CORS error. Cross-Origin Resource Sharing (CORS) is a vital security mechanism, but it can be a significant hurdle if not properly understood.

What is CORS?

CORS is a browser-based security feature that prevents a web page from making requests to a different domain than the one that served the web page. This is called the "Same-Origin Policy."

  • Same-Origin: Two URLs have the same origin if they have the same protocol (http/https), host (example.com), and port (80/443/3000).
  • Why it exists: To prevent malicious scripts on one site from accessing sensitive data on another site (like your bank's API) without your permission.

How CORS Works

When your frontend makes a "cross-origin" request, the browser first sends a "preflight" request (using the HTTP OPTIONS method) to the server. The server then responds with specific headers indicating whether the cross-origin request is allowed.

  • Access-Control-Allow-Origin: Specifies which domains are allowed to access the resource (e.g., https://myfrontend.com or * for any domain).
  • Access-Control-Allow-Methods: Specifies which HTTP methods are permitted (GET, POST, PUT, DELETE, etc.).
  • Access-Control-Allow-Headers: Specifies which custom headers can be sent in the request.

Handling CORS in Express.js

The easiest and most common way to handle CORS in an Express application is by using the cors middleware package.

const cors = require('cors');
const app = express();

// Allow all origins (not recommended for production)
app.use(cors());

// Allow specific origin and configuration (recommended)
const corsOptions = {
  origin: 'https://myfrontend.com',
  optionsSuccessStatus: 200
}
app.use(cors(corsOptions));

Common CORS Pitfalls

  1. Using the Wildcard (*) in Production: While * allows any domain to access your API, it's a security risk. Always specify exact domains for production.
  2. Forgetting Preflight Requests: Some requests (like those with custom headers or complex body types) trigger a preflight. Ensure your server correctly handles the OPTIONS method.
  3. Credentials and Wildcards: If you need to send cookies or authentication headers (withCredentials: true), you cannot use * for the allowed origin; you must specify an explicit domain.
