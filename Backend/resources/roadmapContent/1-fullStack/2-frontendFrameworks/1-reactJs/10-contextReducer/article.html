Advanced Context & useReducer

As complex state grows, useState can become difficult to manage, and prop drilling creates tightly coupled components. Pairing useReducer with the Context API is React's native solution for scalable component architectures.

The useReducer Hook

useReducer is preferable to useState when you have complex state logic involving multiple sub-values, or when the next state depends on the previous one. It formalizes state updates through an action dispatch model.

import { useReducer } from 'react';

const initialState = { count: 0, error: null };

// Reducer function: pure, predictable
function counterReducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + action.payload, error: null };
    case 'decrement':
      if (state.count === 0) return { ...state, error: 'Cannot go below 0' };
      return { ...state, count: state.count - 1, error: null };
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

function Counter() {
  const [state, dispatch] = useReducer(counterReducer, initialState);

  return (
    <div>
      <p>Count: {state.count}</p>
      {state.error && <p className="error">{state.error}</p>}
      <button onClick={() => dispatch({ type: 'increment', payload: 1 })}>+1</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-1</button>
    </div>
  );
}

The Problem with Singleton Context

When you put both state data and the update function (dispatch) into a single Context, every component that consumes that Context will re-render whenever the state changes — even if that component only needs the dispatch function to trigger actions!

// ❌ ANTI-PATTERN: The massive singleton context
const AppContext = createContext();

function AppProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, initialState);
  // Any change to state causes the value object to be a new reference
  return (
    <AppContext.Provider value={{ state, dispatch }}>
      {children}
    </AppContext.Provider>
  );
}

// Even though this Button only uses dispatch, it re-renders entirely when state changes
function ClearButton() {
  const { dispatch } = useContext(AppContext);
  return <button onClick={() => dispatch({ type: 'clear' })}>Clear</button>;
}

The Solution: Separating State and Dispatch

To optimize performance, separate the state and the dispatch function into two different contexts. Since dispatch is guaranteed by React to have a stable reference, components consuming the Dispatch context will never unnecessarily re-render.

import { createContext, useContext, useReducer } from 'react';

// 1. Create two separate contexts
const TasksStateContext = createContext(null);
const TasksDispatchContext = createContext(null);

// 2. Wrap them together in a Provider Component
export function TasksProvider({ children }) {
  const [tasks, dispatch] = useReducer(tasksReducer, []);

  return (
    <TasksStateContext.Provider value={tasks}>
      <TasksDispatchContext.Provider value={dispatch}>
        {children}
      </TasksDispatchContext.Provider>
    </TasksStateContext.Provider>
  );
}

// 3. Create Custom Custom Hooks for safety
export function useTasks() {
  const context = useContext(TasksStateContext);
  if (context === null) {
    throw new Error('useTasks must be used within TasksProvider');
  }
  return context;
}

export function useTasksDispatch() {
  const context = useContext(TasksDispatchContext);
  if (context === null) {
    throw new Error('useTasksDispatch must be used within TasksProvider');
  }
  return context;
}

Using the Optimized Contexts

// Only re-renders when task list changes
function TaskList() {
  const tasks = useTasks();
  return (
    <ul>
      {tasks.map(task => <li key={task.id}>{task.text}</li>)}
    </ul>
  );
}

// 🚀 Highly optimized: NEVER re-renders when tasks change!
// It only reads the stable dispatch function.
function AddTaskButton() {
  const dispatch = useTasksDispatch();
  return (
    <button onClick={() => dispatch({ type: 'ADD_DEFAULT' })}>
      Add Task
    </button>
  );
}

When Context + Reducer is Overkill

  • If the state is asynchronous (fetching from a server), use React Query or SWR instead.
  • If the state is changing rapidly (like dragging elements at 60fps), use a library like Zustand, Jotai, or simple local state to avoid Context's inherent render propagation limitations.
  • If it's just a simple isOpen bool, Stick to useState and pass it down one or two levels. Context is for widely shared business logic.
