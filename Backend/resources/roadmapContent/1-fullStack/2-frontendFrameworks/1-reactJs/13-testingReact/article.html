Testing React Components

Testing gives you confidence that your components work correctly over time. Modern React testing relies mostly on two tools: Jest (the test runner) and React Testing Library (the rendering and assertion utilities).

Philosophy: React Testing Library (RTL)

Unlike older tools (like Enzyme) that inspected the internal state (useState values) or the component tree, RTL follows a simple rule: "The more your tests resemble the way your software is used, the more confidence they can give you."

You should query the DOM exactly how a user does: by finding text, ARIA roles, or labels, rather than searching for specific CSS classes or component names.

Basic Component Test

import { render, screen, fireEvent } from '@testing-library/react';
import Greeting from './Greeting';

test('renders the greeting and interacts', () => {
  // 1. Render the component in a virtual DOM
  render(<Greeting name="Alice" />);

  // 2. Query the DOM (using user-centric queries)
  const heading = screen.getByRole('heading', { name: /hello alice/i });
  expect(heading).toBeInTheDocument();

  // 3. Interact
  const button = screen.getByRole('button', { name: /change name/i });
  fireEvent.click(button);

  // 4. Assert changes
  expect(screen.getByText(/hello bob/i)).toBeInTheDocument();
});

Queries: Finding Elements

RTL provides several queries. They are ordered by accessibility preference:

  1. getByRole: Best option. Queries by ARIA role (button, heading, link, textbox).
  2. getByLabelText: For form inputs.
  3. getByPlaceholderText: For inputs without labels (less ideal).
  4. getByText: Good for finding non-interactive elements like paragraphs.
  5. getByTestId: The last resort. Use when the element is impossible to find otherwise.

Queries also come in three variants based on expected results:

  • getBy...: Returns the element. Throws an error if not found.
  • queryBy...: Returns the element, or null if not found. Use to assert an element is NOT present.
  • findBy...: Returns a Promise. Use for elements that will appear asynchronously.

Testing User Events

While fireEvent works for simple clicks, the @testing-library/user-event library simulates real user behavior much more accurately (focusing, typing character by character, firing keydown events).

import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import LoginForm from './LoginForm';

test('submitting a form login', async () => {
  // Setup userEvent
  const user = userEvent.setup();
  render(<LoginForm />);

  // Find inputs
  const emailInput = screen.getByLabelText(/email/i);
  const passwordInput = screen.getByLabelText(/password/i);
  const submitButton = screen.getByRole('button', { name: /login/i });

  // Simulate typing (asynchronous)
  await user.type(emailInput, 'test@example.com');
  await user.type(passwordInput, 'secret123');
  
  // Submit
  await user.click(submitButton);

  // Asserting async result (e.g. success message loads)
  const successMsg = await screen.findByText(/welcome back/i);
  expect(successMsg).toBeInTheDocument();
});

Mocking API Calls with MSW (Mock Service Worker)

Mocking fetch or axios directly is brittle. Mock Service Worker (MSW) intercepts network requests at the network level, letting your components fetch data as if they were talking to a real API.

import { render, screen } from '@testing-library/react';
import { setupServer } from 'msw/node';
import { http, HttpResponse } from 'msw';
import UserList from './UserList';

// Setup mock server
const server = setupServer(
  http.get('/api/users', () => {
    return HttpResponse.json([
      { id: 1, name: 'Alice' },
      { id: 2, name: 'Bob' }
    ]);
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test('fetches and displays users', async () => {
  render(<UserList />);
  
  // Initially loading
  expect(screen.getByText(/loading/i)).toBeInTheDocument();

  // Wait for the async API call to resolve
  const aliceItem = await screen.findByText('Alice');
  expect(aliceItem).toBeInTheDocument();
  expect(screen.getByText('Bob')).toBeInTheDocument();
});

Best Practices

  • Stop testing implementation details (e.g. expect(component.state().count).toBe(1)). Test what the user sees in the DOM.
  • Avoid wrapping your render calls in large custom setup functions unless strictly necessary for Providers (Router, Redux, Context).
  • Use screen.debug() generously when tests fail to visually see the virtual DOM state.
  • Always use findBy... for assertions related to asynchronous state updates.
