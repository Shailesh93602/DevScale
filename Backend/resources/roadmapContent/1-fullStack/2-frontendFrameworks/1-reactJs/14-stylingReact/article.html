Styling Ecosystem in React

Modern React applications have moved away from global CSS files toward scoped, component-based styling strategies. Each approach has unique trade-offs regarding performance, developer experience, and scalability.

CSS Modules

CSS Modules are simply CSS files where all class names are hashed at build time locally to the component. This perfectly solves the global namespace collision problem of CSS.

/* Button.module.css */
.primaryButton {
  background-color: blue;
  color: white;
  border-radius: 4px;
}
.disabled {
  opacity: 0.5;
  cursor: not-allowed;
}
// Button.jsx
import styles from './Button.module.css';

export function Button({ disabled, children }) {
  return (
    <button 
      className={`${styles.primaryButton} ${disabled ? styles.disabled : ''}`}
      disabled={disabled}
    >
      {children}
    </button>
  );
}

Pros: Zero runtime overhead, pure CSS support (Sass/SCSS), natively supported in Vite and Next.js.
Cons: Managing multiple classes dynamically can get messy.

CSS-in-JS (Styled Components)

Styled Components allows you to write actual CSS syntax directly inside JavaScript using tagged template literals. The styles are injected into a <style> tag at runtime.

import styled from 'styled-components';

// Creates a React component with hashed class names attached
const Wrapper = styled.button`
  background-color: ${props => props.primary ? 'blue' : 'gray'};
  color: white;
  border-radius: 4px;
  padding: 8px 16px;
  cursor: ${props => props.disabled ? 'not-allowed' : 'pointer'};
  
  &:hover {
    background-color: ${props => props.primary ? 'darkblue' : 'darkgray'};
  }
`;

export function Content() {
  return (
    <div>
      <Wrapper primary>Submit</Wrapper>
      <Wrapper disabled>Loading</Wrapper>
    </div>
  );
}

Pros: Excellent dynamic styling based on React props, completely scoped, easy to read.
Cons: Performance penalty at runtime, massive issues with Next.js Server Components, large bundle size.

Utility-First CSS (Tailwind)

Tailwind CSS provides low-level utility classes that let you build completely custom designs without ever leaving your JSX. It has become the de facto standard in modern React, especially in Next.js.

export function Card({ title, description, active }) {
  return (
    <div className={`p-6 max-w-sm rounded-xl shadow-lg border 
        ${active ? 'bg-blue-50 border-blue-500' : 'bg-white border-gray-200'}
      `}
    >
      <h2 className="text-xl font-bold text-slate-800">{title}</h2>
      <p className="mt-2 text-slate-500">{description}</p>
      <button className="mt-4 px-4 py-2 bg-blue-600 hover:bg-blue-700 text-white rounded">
        Action
      </button>
    </div>
  );
}

Handling Complex Classes with clsx and tailwind-merge

As components get complex, writing ternary operators in template literals becomes unreadable. The clsx utility conditionally joins classes, and tailwind-merge ensures that conflicting Tailwind utilities override correctly.

import { clsx, type ClassValue } from 'clsx';
import { twMerge } from 'tailwind-merge';

// Common utility pattern (often named cn.ts)
export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}

// Usage in Component
export function Badge({ variant = 'default', className }) {
  return (
    <span className={cn(
      "px-2 py-1 rounded text-sm font-medium",
      {
        "bg-blue-100 text-blue-800": variant === 'default',
        "bg-red-100 text-red-800": variant === 'destructive',
        "bg-green-100 text-green-800": variant === 'success',
      },
      className // allow consumer to override
    )}>
      Hey
    </span>
  );
}

Best Practices

  • For Next.js / React Server Components projects, avoid runtime CSS-in-JS (Styled Components, Emotion). Use Tailwind or CSS Modules instead.
  • Use clsx or classnames instead of complex template literals (${foo ? 'a' : 'b'}) for managing class logic.
  • In component libraries (like shadcn/ui), create reusable styled components natively with Tailwind + the cva (class-variance-authority) library.
