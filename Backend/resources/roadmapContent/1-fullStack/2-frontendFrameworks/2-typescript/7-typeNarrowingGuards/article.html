Type Narrowing & Custom Guards

In TypeScript, a variable can have a broad type (like string | number veya unknown). Narrowing is the process of safely refining that type to a more specific one using JavaScript control flow structures so TypeScript can guarantee safety.

Basic Control Flow Narrowing

TypeScript understands standard JavaScript checks like typeof and instanceof. When used in an if block, the type is narrowed inside that block.

function processInput(input: string | number | Date) {
  // Narrowing with typeof
  if (typeof input === "string") {
    // TypeScript knows input is exactly string here
    console.log(input.toUpperCase());
  } else if (typeof input === "number") {
    // TypeScript knows input is exactly number
    console.log(input.toFixed(2));
  } else if (input instanceof Date) {
    // Narrowing with instanceof for classes/objects
    console.log(input.toISOString());
  }
}

The 'in' Operator Narrowing

When dealing with union types of objects, the in operator checks if a property exists on an object, allowing TypeScript to narrow down which object type you have.

type Fish = { swim: () => void };
type Bird = { fly: () => void };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    animal.swim(); // TypeScript knows this is Fish
  } else {
    animal.fly();  // TypeScript knows this is Bird
  }
}

Custom Type Predicate Guards (is)

Sometimes standard JavaScript checks aren't enough, particularly when dealing with complex data fetching where you just get unknown or any. You can write custom functions that explicitly tell TypeScript "if this function returns true, the variable is of type X".

// The return type "pet is Fish" is a Type Predicate
function isFish(pet: Fish | Bird | unknown): pet is Fish {
  // Standard JS check returning a boolean
  return (pet as Fish).swim !== undefined;
}

function feed(pet: unknown) {
  if (isFish(pet)) {
    pet.swim(); // pet is narrowed from unknown to Fish
  }
}

// Excellent for array filtering
const pets = [getFish(), getBird(), getFish()];
// TypeScript doesn't narrow arrays effectively without standard type predicates
const fishes: Fish[] = pets.filter(isFish);

Assertion Functions

Assertion functions throw an error if a condition is not met. After calling an assertion function, TypeScript assumes the rest of the code is safe and the type is narrowed.

// Assertion signature
function assertIsString(val: any): asserts val is string {
  if (typeof val !== 'string') {
    throw new Error('Not a string!');
  }
}

function process(value: unknown) {
  assertIsString(value);
  // execution stops here if it throws
  // therefore, value MUST be string below this line
  console.log(value.toUpperCase());
}

The power of unknown vs any

any turns off type checking completely. unknown means "I don't know what this is yet". You cannot do anything with an unknown variable (call methods, access properties) until you narrow it using a type guard. Always use unknown over any for API responses.

function handleResponse(res: any) {
  res.doSomething(); // No error, but crashes at runtime if method doesn't exist
}

function handleSafeResponse(res: unknown) {
  res.doSomething(); // ❌ Error: Object is of type 'unknown'
  
  if (typeof res === 'object' && res !== null && 'doSomething' in res) {
    (res as { doSomething: () => void }).doSomething(); // ✅ Safe
  }
}

Exhaustiveness Checking with never

The never type represents a state that should not exist. You can use it in switch statements to ensure you have handled every possible case of a union. If you add a new case to the union later but forget to update the switch statement, TypeScript will throw an error!

type Shape = "circle" | "square";

function getArea(shape: Shape) {
  switch (shape) {
    case "circle":
      return Math.PI * 2;
    case "square":
      return 4;
    default:
      // If we add "triangle" to Shape, `shape` will fall through to default.
      // TypeScript will yell because "triangle" cannot be assigned to "never".
      const _exhaustiveCheck: never = shape;
      return _exhaustiveCheck;
  }
}

Best Practices

  • Avoid using type assertions (as string) — they bypass type checks. Use Type Guards instead.
  • Use unknown instead of any when processing data from APIs or user inputs, forcing you to write guards.
  • Use exhaustiveness checking (never) on Redux reducers or state machine transitions to guarantee all states are handled.
