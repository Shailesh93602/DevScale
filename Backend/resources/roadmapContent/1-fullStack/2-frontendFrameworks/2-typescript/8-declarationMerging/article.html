Declaration Merging & Global Augmentation

Declaration merging is a unique feature of TypeScript where the compiler merges two or more separate declarations declared with the same name into a single definition. This is incredibly useful for extending third-party libraries, augmenting global variables, or patching missing types.

Interface Merging

If you declare two interface definitions with the same name in the same scope, TypeScript merges their properties. Note: This does not work with type aliases — they will throw a "Duplicate identifier" error.

interface User {
  id: number;
  name: string;
}

// Later in the same file or a different file
interface User {
  email: string; // Adds to the existing User interface
}

const me: User = {
  id: 1,
  name: "Alice",
  email: "alice@test.com" // Allowed! The interface merged.
};

Augmenting Global Objects (Window, NodeJS)

When you add properties to global objects like window in the browser or process.env in Node, TypeScript will complain because those properties aren't in the default type definitions. You can use declare global to merge your custom properties into the global scope.

// Appends 'analytics' to the global browser Window object
declare global {
  interface Window {
    analytics: {
      track(event: string): void;
    };
  }
}

// Now you can use it anywhere without errors
window.analytics.track("page_view");

For Node.js environments (like expanding process.env):

declare global {
  namespace NodeJS {
    interface ProcessEnv {
      DATABASE_URL: string;
      API_KEY: string;
      NODE_ENV: 'development' | 'production';
    }
  }
}

// Now process.env.DATABASE_URL is strongly typed as string instead of string | undefined
console.log(process.env.DATABASE_URL.toUpperCase());

Module Augmentation (Adding to Third-Party Libraries)

A very common problem is needing to add custom properties to an object defined by a library, such as adding a user property to an Express.js Request object.

import { Request } from 'express';

// Augment the 'express' module
declare module 'express-serve-static-core' {
  // Merge into the Request interface of the module
  interface Request {
    user?: {
      id: string;
      role: 'admin' | 'user';
    };
  }
}

// Inside a middleware or controller
app.get('/profile', (req, res) => {
  // TypeScript now knows req.user exists and provides autocomplete!
  console.log(req.user?.id); 
});

Note: You must augment the exact module name where the interface is originally declared.

Namespace Merging

Namespaces can merge with themselves, but incredibly, they can also merge with Classes and Functions. This is a pattern heavily used in older JS libraries (like jQuery $ being both a function and an object with properties).

// 1. Defining a class
class Album {
  label: Album.AlbumLabel; // Using a type from the merged namespace!
}

// 2. Merging a namespace with the same name
namespace Album {
  // This essentially creates a static property/type on the Album class
  export class AlbumLabel { }
}

// Result: You can use Album as a class, and access Album.AlbumLabel

When NOT to use Declaration Merging

  • Do not use it to blindly fix "Property does not exist" errors on complex objects without understanding where the type comes from. Often, using a generic parameter is the "correct" way (e.g., express.Request<Params, ResBody, ReqBody, ReqQuery>).
  • Avoid interface merging in your own application's core business logic. It makes the code harder to read because the developer has to hunt for multiple interface declarations. Use interface Person extends User {} instead for intentional inheritance.
