Server State vs. Client State

One of the most important architectural decisions in React is understanding that not all state is the same. State has two fundamentally different natures, and using the wrong tool for each is the root cause of most state management complexity.

The Two Types of State

Client State (UI State)

Local, ephemeral information that belongs to the UI:

  • Is the modal open or closed?
  • What text is in the search input?
  • Which tab is active?
  • Is dark mode enabled?
  • Form field values before submission

Tools: useState, useReducer, Zustand, Jotai, Context API

Server State (Remote/Async State)

Data that exists on the server and is fetched asynchronously:

  • The list of users from /api/users
  • The current user's profile
  • Product inventory data
  • Notification count

Characteristics: asynchronous, can become stale, shared across sessions, requires caching, re-fetching, optimistic updates, etc.

Tools: React Query (TanStack Query), SWR, RTK Query

Why You Shouldn't Store Server State in Redux/Zustand

// ❌ Anti-pattern: manual server state in Redux/Zustand
// You end up managing: loading, error, caching, refetching, staleness manually
const useStore = create((set) => ({
  users: [],
  usersLoading: false,
  usersError: null,
  fetchUsers: async () => {
    set({ usersLoading: true });
    try {
      const users = await getUsers();
      set({ users, usersLoading: false });
    } catch (error) {
      set({ usersError: error, usersLoading: false });
    }
  }
}));

// ✅ Correct: server state belongs in React Query
function useUsers() {
  return useQuery({
    queryKey: ['users'],
    queryFn: getUsers,
    staleTime: 5 * 60 * 1000, // consider fresh for 5 minutes
  });
}

Combining React Query + Zustand

// Zustand handles client-only UI state
const useUIStore = create((set) => ({
  selectedUserId: null,
  isCreateModalOpen: false,
  filterRole: 'all',
  setSelectedUser: (id) => set({ selectedUserId: id }),
  openCreateModal: () => set({ isCreateModalOpen: true }),
  closeCreateModal: () => set({ isCreateModalOpen: false }),
  setFilterRole: (role) => set({ filterRole: role }),
}));

// React Query handles server state
function useFilteredUsers() {
  const filterRole = useUIStore(s => s.filterRole);

  return useQuery({
    queryKey: ['users', filterRole],  // re-fetches when filter changes
    queryFn: () => fetchUsers({ role: filterRole }),
  });
}

// Component uses both together cleanly
function UsersPage() {
  const { data: users, isLoading } = useFilteredUsers();
  const { filterRole, setFilterRole, openCreateModal } = useUIStore();

  return (
    <div>
      <select value={filterRole} onChange={e => setFilterRole(e.target.value)}>
        <option value="all">All</option>
        <option value="admin">Admins</option>
      </select>
      <button onClick={openCreateModal}>Create User</button>
      {isLoading ? <Spinner /> : users?.map(u => <UserRow key={u.id} user={u} />)}
    </div>
  );
}

Optimistic Updates with React Query

const queryClient = useQueryClient();

const deleteUser = useMutation({
  mutationFn: (userId: string) => api.deleteUser(userId),
  onMutate: async (userId) => {
    // Optimistically remove from UI immediately
    await queryClient.cancelQueries({ queryKey: ['users'] });
    const previousUsers = queryClient.getQueryData(['users']);
    queryClient.setQueryData(['users'], (old: User[]) =>
      old.filter(u => u.id !== userId)
    );
    return { previousUsers }; // for rollback
  },
  onError: (err, userId, context) => {
    // Rollback on error
    queryClient.setQueryData(['users'], context?.previousUsers);
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['users'] });
  }
});

State Decision Flowchart

Is the state from a server/API?
├── YES → Use React Query / SWR
└── NO → Is it needed by many components?
          ├── YES → Use Zustand / Context
          └── NO → Use useState / useReducer

Best Practices

  • Never store server-fetched data in Zustand/Redux — React Query manages caching and updates.
  • Keep client state as close to the component that uses it as possible (start with useState).
  • Use React Query's staleTime to prevent overfetching.
  • Optimistic updates significantly improve perceived performance.
