State Machines & XState

As user interfaces grow more complex, boolean flags (isLoading, isError, isSuccess) inevitably lead to "impossible states." Finite State Machines (FSM) solve this by formally defining every possible state your app can be in, and strictly controlling the transitions between them.

The Problem with Boolean Flags

// Typical React component state
const [isLoading, setIsLoading] = useState(false);
const [isSuccess, setIsSuccess] = useState(false);
const [isError, setIsError] = useState(false);
const [data, setData] = useState(null);

async function fetchData() {
  setIsLoading(true);
  setIsError(false); // Easy to forget this!
  try {
    const res = await api.get();
    setData(res);
    setIsSuccess(true);
    // Forgot to set isLoading to false!
  } catch (err) {
    setIsError(true);
    setIsLoading(false);
  }
}

// Result: You end up with impossible states, like:
// { isLoading: true, isError: true, isSuccess: true }

Introduction to State Machines

A finite state machine is a mathematical model of computation. It has a finite number of States (Idle, Loading, Success, Error). It can only be in exactly ONE state at a time. It also explicitly defines Transitions: specific events that move the machine from one state to another.

If you are in the Error state, the machine ignores a SUCCESS event because that transition doesn't exist.

XState: Statecharts for JS

// 1. Install xstate and @xstate/react
import { setup, createMachine } from 'xstate';
import { useMachine } from '@xstate/react';

// 2. Define the exact shape of your state machine
const fetchMachine = createMachine({
  id: 'fetch',
  // The starting point
  initial: 'idle',
  
  // Define all possible states
  states: {
    idle: {
      on: { FETCH: 'loading' } // Transition: when FETCH event happens, go to loading
    },
    loading: {
      on: {
        RESOLVE: 'success',
        REJECT: 'error'
      }
    },
    success: {
      type: 'final' // Machine stops here
    },
    error: {
      on: { RETRY: 'loading' }
    }
  }
});

Using XState in React

function DataFetcher() {
  // useMachine connects XState to React's lifecycle
  const [state, send] = useMachine(fetchMachine);

  return (
    <div>
      {/* state.matches() safely checks the current state */}
      
      {state.matches('idle') && (
        <button onClick={() => send({ type: 'FETCH' })}>Start Fetch</button>
      )}

      {state.matches('loading') && <p>Loading data...</p>}

      {state.matches('success') && <p>Data loaded successfully!</p>}

      {state.matches('error') && (
        <div>
          <p>Oh no, an error occurred.</p>
          <button onClick={() => send({ type: 'RETRY' })}>Try Again</button>
        </div>
      )}
    </div>
  );
}

Context: State Machine Data

While standard States define where the app is, Context (in XState terms) defines the quantitative data (like user inputs, fetched arrays, or counts). You use Actions to update Context.

import { assign } from 'xstate';

const userMachine = createMachine({
  id: 'user',
  initial: 'loggedOut',
  context: {
    user: null, // quantitative data
    retries: 0
  },
  states: {
    loggedOut: {
      on: { LOGIN: 'authenticating' }
    },
    authenticating: {
      on: {
        SUCCESS: {
          target: 'loggedIn',
          // Action to update context
          actions: assign({
            user: (context, event) => event.user
          })
        },
        FAILURE: {
          target: 'error',
          actions: assign({
            retries: (context) => context.retries + 1
          })
        }
      }
    },
    // ...
  }
});

Why use XState?

  • Visual Editing: XState generates visual charts of your logic structure. You can literally show the UI logic to a product manager before writing React code.
  • No Impossible States: Bugs where isLoading and isError are true simultaneously disappear forever.
  • Decoupling: The business logic (the machine) is completely separated from the UI component framework.
