Signals & Fine-Grained Reactivity

For years, React's primary execution model has relied on the Virtual DOM. When state changes at the top of an app, React re-executes the entire component tree below it to figure out what changed. This top-down model works but leads to heavy memoization requirements (useMemo, useCallback, React.memo).

Signals represent a paradigm shift back to fine-grained reactivity, allowing specific parts of the DOM to update directly without triggering widespread component re-renders.

What is a Signal?

A Signal is an intelligent wrapper around a piece of data. While it holds a value, it also tracks who is listening to it. When the value changes, the Signal notifies its subscribers directly.

import { signal } from "@preact/signals-react";

// The signal is defined OUTSIDE the component.
// It acts as a global, reactive variable.
const count = signal(0);
const multiplier = signal(2);

The Magic: Render Bypassing

When you use standard React useState, changing state causes the component function to run again. When you use a Signal deeply embedded in the DOM, standard implementations (like Preact Signals) bypass the Virtual DOM entirely. React's render function does not run again.

import { signal, computed } from "@preact/signals-react";

const count = signal(0);
// Computed signals automatically track dependencies
const doubled = computed(() => count.value * 2);

function Counter() {
  console.log("Component Rendered!"); // This only runs ONCE!

  return (
    <div>
      {/* 
        When count.value changes, the signal reaches directly into the 
        real DOM to update this text node. The component does NOT re-render!
      */}
      <p>Count: {count}</p>
      <p>Doubled: {doubled}</p>
      
      {/* Mutating a signal is as easy as assigning to .value */}
      <button onClick={() => count.value++}>Increment</button>
    </div>
  );
}

Signals vs. React State

Compare the simplicity of Signals to the complexity of standard React state management.

Traditional React

// ❌ Needs Context API to avoid prop drilling
// ❌ Re-renders entire App tree when name changes
function App() {
  const [name, setName] = useState("Alice");
  return (
    <UserContext.Provider value={{ name, setName }}>
      <Header />
      <Profile />
    </UserContext.Provider>
  );
}

With Signals

// ✅ No Context Needed. No prop drilling.
const name = signal("Alice");

// ✅ Mutates state globally. Does not re-render wrapper components.
function ProfileControls() {
  return <input value={name.value} onChange={e => name.value = e.target.value} />;
}

// ✅ Updates independently.
function Header() {
  return <h1>Welcome, {name}</h1>;
}

Effect Tracking

Signals use specialized effects that automatically subscribe to the signals they read. You don't have to provide dependency arrays.

import { effect } from "@preact/signals-react";

// Logs immediately, and then automatically runs again ONLY when count.value changes
effect(() => {
  console.log(`The count is now ${count.value}`);
});

Ecosystem Impact

Signals are the driving philosophy behind modern frameworks like SolidJS and Vue 3 (ref/reactive). Their addition to the React ecosystem (via libraries like @preact/signals-react or MobX) offers an alternative to React Context and Redux for global state, specifically emphasizing performance and developer ergonomics.

Best Practices

  • Use Context/React State for data intrinsically tied to the UI tree (like specific modal states).
  • Use Signals for global business logic, heavy performance-sensitive pages, or rapidly changing state (like animation data or drag-and-drop coordinates).
  • Treat Signal variables like standard JS variables: assign with =, read with .value. Pass the signal itself into JSX (<p>{count}</p>) to leverage direct DOM updates.
