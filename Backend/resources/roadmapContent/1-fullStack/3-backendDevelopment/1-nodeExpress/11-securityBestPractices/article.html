Express Security Best Practices

Modern backend APIs are under constant attack from automated bots and malicious actors. Securing an Express application means defending against the OWASP Top 10 vulnerabilities, including XSS, Injection, and DDoS attacks. Relying on default Express settings is highly dangerous.

1. Helmet: Secure HTTP Headers

By default, Express sends headers that identify the server (X-Powered-By: Express), which tells hackers exactly which exploits to try. Express also lacks dozens of modern security headers (like Content Security Policy, X-XSS-Protection, and Strict-Transport-Security).

The Helmet middleware automatically fixes all of this by setting 15+ secure HTTP headers instantly.

const express = require('express');
const helmet = require('helmet');
const app = express();

// Must be the VERY FIRST middleware in your app!
app.use(helmet());

2. CORS (Cross-Origin Resource Sharing)

By default, if you have an API running at api.example.com, a hacker can build a malicious website at evil.com that makes Fetch/XHR requests to your API using the victim's browser (and cookies!). Configuring CORS restricts which domains are allowed to call your API from a browser.

const cors = require('cors');

// ❌ DANGEROUS: Allows ANY website in the world to access your API
// app.use(cors()); 

// ✅ SECURE: Only allows requests from your specific frontend domains
const corsOptions = {
  origin: ['https://www.myfrontend.com', 'https://admin.myfrontend.com'],
  optionsSuccessStatus: 200, 
  credentials: true // Required if you are sending HttpOnly cookies!
};

app.use(cors(corsOptions));

3. Rate Limiting (Preventing Brute Force / DDoS)

If you don't limit how many requests a user can make, a hacker can easily write a script to guess 10,000 passwords per second on your /login route, or crash your server by requesting heavy database queries over and over (Denial of Service).

const rateLimit = require('express-rate-limit');

// 1. General Rate Limiter for all routes
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes window
  max: 100, // Limit each IP to 100 requests per window
  message: "Too many requests from this IP, please try again after 15 minutes"
});
app.use(globalLimiter);

// 2. Strict Rate Limiter specifically for authentication routes
const authLimiter = rateLimit({
  windowMs: 60 * 60 * 1000, // 1 hour
  max: 5, // Limit each IP to 5 failed login requests per hour
  message: "Too many login attempts, please try again after an hour"
});
app.use('/api/auth/login', authLimiter);

4. Preventing NoSQL Injection (MongoDB)

SQL Injection is famous, but NoSQL databases like MongoDB are also vulnerable. If you allow users to pass objects directly into a query without sanitization, hackers can use query operators (like $gt, $ne) to bypass authentication.

// ❌ THE VULNERABILITY
// If the hacker sends req.body as { "email": { "$gt": "" }, "password": { "$gt": "" } }
// The query evaluates to "Find a user where email is greater than nothing", which returns the first user (often an Admin) and logs them in!
app.post('/login', async (req, res) => {
  const user = await User.findOne(req.body); 
});

// ✅ THE FIX: Use express-mongo-sanitize
const mongoSanitize = require('express-mongo-sanitize');

// Removes any keys in req.body, req.query, or req.params that begin with a '$' or '.'
app.use(mongoSanitize());

// Also, strictly extract expected strings rather than passing the whole req.body object
app.post('/login', async (req, res) => {
  const email = String(req.body.email);
  const password = String(req.body.password);
  const user = await User.findOne({ email, password }); 
});

5. Preventing Cross-Site Scripting (XSS)

If a user enters <script>alert('Hacked!');</script> as their username, and you display that username to other users, their browsers will execute the malicious code. The server must sanitize all incoming user data.

const xss = require('xss-clean');

// Intercepts all req.body/query/params and strips out HTML/script tags
app.use(xss());

6. Environment Variables & Secrets

Never hardcode API keys, database URLs, or JWT Secrets in your source code. Use the dotenv package to load them from a .env file, and ensure .env is added to your .gitignore so it is never pushed to GitHub.

Best Practices

  • Keep your dependencies updated using npm audit to find and fix known vulnerabilities in third-party packages.
  • Avoid using eval() or setTimeout('string') at all costs; they execute arbitrary strings as code.
  • Always run app.disable('x-powered-by') if you aren't using Helmet to obscure that you are running Express.
