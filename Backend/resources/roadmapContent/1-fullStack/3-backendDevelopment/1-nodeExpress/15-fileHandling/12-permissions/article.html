Managing File Access Permissions

In a multi-user application, not all files should be accessible to everyone. Proper file access permissions ensure that a user can only view, download, or delete the files they own or have been explicitly granted access to.

Permissions for Local Files (UNIX Permissions)

On a traditional file system, permissions are managed through owners, groups, and access modes (read, write, execute). In Node.js, you can use fs.mode to check and fs.chmod() to change these permissions.

  • Read (r): 4
  • Write (w): 2
  • Execute (x): 1

Using fs.chmodSync('./secret.txt', 0o600) would make the file readable and writable only by the owner.

Permissions in Cloud Storage (S3 Policies)

In cloud storage like Amazon S3, permissions are much more granular and are managed through IAM Policies and Bucket Policies. You can define exactly who (which IAM user or role) can perform what action (e.g., s3:GetObject) on specific objects.

Implementing User-Level Permissions in Your App

For most web applications, you'll need to implement your own permission logic in the backend. This is best done by storing a "user_id" on the file record in your database.

app.get('/files/:id', authenticate, async (req, res) => {
  const file = await FileModel.findById(req.params.id);
  
  // 1. Check if the file exists
  if (!file) return res.status(404).json({ error: "File not found" });

  // 2. Check if the authenticated user is the owner (ACL)
  if (file.userId !== req.user.id && req.user.role !== 'ADMIN') {
    return res.status(403).json({ error: "Access denied" });
  }

  // 3. Serve the file
  res.sendFile(file.path);
});

Public vs. Private Access

  • Public Files: CSS, JS, and generic images that can be cached on a CDN and served directly to any visitor.
  • Private Files: Personal documents, private photos, or internal logs that must only be served through an authenticated API endpoint or using Pre-signed URLs.

Best Practices Summary

  1. Deny by Default: Unless a file is explicitly marked as public, access should be blocked for unauthenticated or unauthorized users.
  2. Avoid ID Guessing: Don't use sequential IDs for files in your URLs (e.g., /files/1, /files/2). Use UUIDs or other long, random strings to prevent "Insecure Direct Object Reference" (IDOR) attacks.
  3. Keep Metadata Private: Even the metadata of a file (the name, the size, the fact that it exists) can be sensitive and should be protected with permissions.
