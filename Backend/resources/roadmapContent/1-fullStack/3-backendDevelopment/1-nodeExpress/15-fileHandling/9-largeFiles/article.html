Handling Large Files in Node.js

Loading a large file (e.g., a 1GB video) entirely into memory using fs.readFile() would likely crash your Node.js process. To handle large files efficiently, you must use Streams.

The Power of Streams

Streams allows you to process data piece by piece (chunk by chunk) as it's being read or written. This keeps your application's memory usage constant and very low, regardless of the size of the file being processed.

const fs = require('fs');

// Create a readable stream from a large file
const src = fs.createReadStream('./large-video.mp4');

// Pipe the stream directly to the response (writable stream)
app.get('/video', (req, res) => {
  res.setHeader('Content-Type', 'video/mp4');
  src.pipe(res);
});

Memory-Efficient File Uploads

When users upload large files, you should avoid storing the entire file in a memory buffer. Instead, use Multer's diskStorage or, even better, stream the file directly to cloud storage like AWS S3 using multer-s3 or busboy.

Processing Data on the Fly

Sometimes you need to process a large file as you're reading it (e.g., parsing a 2GB CSV file). You can use Transform streams for this:

const { Transform } = require('stream');

const upperCaseTransform = new Transform({
  transform(chunk, encoding, callback) {
    this.push(chunk.toString().toUpperCase());
    callback();
  }
});

fs.createReadStream('input.txt')
  .pipe(upperCaseTransform)
  .pipe(fs.createWriteStream('output.txt'));

Timeout and Error Handling

Large file operations can take a long time. You must ensure your server doesn't time out the request and that you handle potential stream errors (e.g., the user disconnects mid-upload) to prevent memory leaks or hanging processes.

Best Practices for Large Files

  1. Avoid fs.readFile(): Never load a large or unknown-size file completely into memory.
  2. Use pipe(): It's the most efficient way to connect input and output streams.
  3. Implement Backpressure: If you're building custom streams, ensure you handle "backpressure"—where the source is sending data faster than the destination can process it.
  4. Offload to Worker Threads: If you're doing heavy processing on the file data (like video transcoding), offload it to a Worker Thread to keep the main event loop responsive.
