Middleware & Error Handling in Express

Middleware functions are the core architectural pattern of Express.js. They are functions that have access to the request object (req), the response object (res), and the next middleware function in the application's request-response cycle.

Understanding the Middleware Stack

When a request hits an Express server, it passes through a pipeline of middleware functions. Each middleware can execute code, make changes to the request/response objects, end the response cycle (by sending a response), or pass control to the next() middleware.

const express = require('express');
const app = express();

// 1. Application-level middleware (Runs on every request)
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next(); // MUST call next() to pass control to the next function
});

// 2. Built-in middleware (Parses incoming JSON payloads)
app.use(express.json());

// 3. Route specific middleware
const requireAuth = (req, res, next) => {
  if (!req.headers.authorization) {
    // Ending the cycle early (client receives error, next is never called)
    return res.status(401).json({ error: "Unauthorized" });
  }
  next();
};

app.get('/dashboard', requireAuth, (req, res) => {
  res.json({ message: "Welcome to your dashboard" });
});

Modifying Request/Response Objects

Middleware is often used to validate data, decode tokens, or fetch user profiles, attaching the result directly to the req object so downstream controllers can use it without duplicating work.

const decodeUser = async (req, res, next) => {
  try {
    const token = req.headers.authorization.split(' ')[1];
    const user = await jwt.verify(token, process.env.SECRET);
    
    // Attach user to request!
    req.user = user; 
    next();
  } catch (err) {
    res.status(403).json({ error: "Invalid token" });
  }
};

app.get('/profile', decodeUser, (req, res) => {
  // Controller safely uses req.user
  res.json({ email: req.user.email }); 
});

Global Error Handling Middleware

In standard routes, if an error happens, you usually send a 500 response. However, if you forget a try/catch, your Node server might crash. Express provides a dedicated Error-Handling Middleware.

It acts identically to normal middleware, except it has exactly Four arguments (err, req, res, next). Express recognizes this signature and reserves it for error handling.

// Normal route throws an error
app.get('/crash', (req, res, next) => {
  const err = new Error("Database connection failed");
  err.status = 503;
  // Passing an argument to next() tells Express: "An error occurred! Skip all normal middleware and go straight to the Error Handler."
  next(err); 
});

// ... all other routes ...

// Global Error Handler MUST be the last app.use() in the file!
app.use((err, req, res, next) => {
  console.error("GLOBAL ERROR CATCHER:", err.message);
  
  const status = err.status || 500;
  res.status(status).json({
    error: {
      message: err.message || "Internal Server Error",
      // Only show stack trace in development mode
      stack: process.env.NODE_ENV === 'production' ? '🥞' : err.stack
    }
  });
});

Async Error Wrapping (Express 4 vs 5)

In Express 4, if a Promise rejects inside an async route handler, Express does not catch it automatically, leading to Unhandled Promise Rejections. You must wrap them or use a library like express-async-errors.

// Wrapper utility
const asyncHandler = (fn) => (req, res, next) => {
  return Promise.resolve(fn(req, res, next)).catch(next);
};

app.get('/users', asyncHandler(async (req, res) => {
  const users = await db.getUsers(); // If this throws, asyncHandler catches it and calls next(err)
  res.json(users);
}));

Note: Express 5 (currently in beta/rc) natively supports returning Promises from handlers and catches them automatically.

Best Practices

  • Always remember to either call res.send()/res.json() to end the request, or call next(). If you do neither, the client request will hang indefinitely until it times out.
  • Place your Global Error Handler dead last, below all other app.use() and route definitions.
  • Use a library like morgan for logging middleware and helmet for security header middleware rather than writing them from scratch.
