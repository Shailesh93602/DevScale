Async Patterns & Event Loop

Node.js is famous for being "single-threaded" yet highly concurrent. It achieves this non-blocking I/O through the V8 engine combined with the libuv library, which implements the Event Loop.

The Node.js Event Loop Phases

Understanding the strict order of event loop phases is critical for advanced performance profiling. When Node.js starts, it initializes the event loop, processes the provided input script, and then enters the loop.

  1. Timers: Executes callbacks scheduled by setTimeout() and setInterval().
  2. Pending Callbacks: Executes I/O callbacks deferred to the next loop iteration (rare internal OS errors).
  3. Idle, Prepare: Internal use only.
  4. Poll: Retrieves new I/O events (file reading, network requests); executes I/O related callbacks. (This is where the loop spends most of its time).
  5. Check: Executes callbacks registered via setImmediate().
  6. Close Callbacks: Executes close events, e.g., socket.on('close', ...).

Microtasks vs Macrotasks

Between every single phase of the event loop, Node.js empties the Microtask Queue. Microtasks are given the highest priority.

  • Microtasks: process.nextTick() (highest priority), and Promise callbacks (.then()/.catch()).
  • Macrotasks: Everything in the main phases (setTimeout, setImmediate, I/O callbacks).
console.log('1. Start');

setTimeout(() => console.log('2. setTimeout (Timers Phase)'), 0);
setImmediate(() => console.log('3. setImmediate (Check Phase)'));

Promise.resolve().then(() => console.log('4. Promise (Microtask)'));

process.nextTick(() => console.log('5. nextTick (Highest Priority Microtask)'));

console.log('6. End');

/* Execution Order:
1. Start (Main thread)
6. End (Main thread ending)
5. nextTick (Microtask queue processed immediately after main thread)
4. Promise (Microtask queue processed)
2. setTimeout (Timers Phase - Macrotask)
3. setImmediate (Check Phase - Macrotask)
*/

setImmediate vs setTimeout(0)

They sound similar, but their execution order differs depending on context.

  • If called in the main module (globally), the order is non-deterministic (depends on machine performance).
  • If called inside an I/O callback (like inside fs.readFile), setImmediate will ALWAYS run before setTimeout. Why? Because the event loop is currently in the Poll phase. The next phase is the Check phase (where setImmediate sits), while the Timers phase has already passed and won't be reached until the next full loop.

Don't Block the Event Loop

Because there is only one V8 thread executing JavaScript, expensive synchronous operations will freeze the entire server. E.g., computing a massive array loop, synchronous cryptography (bcrypt.hashSync()), or reading huge JSON files via JSON.parse().

app.get('/hash', (req, res) => {
  // ❌ EVIL: Blocks the entire server for 500ms. No other users can connect!
  const hash = crypto.pbkdf2Sync('password', 'salt', 100000, 512, 'sha512');
  res.send(hash);
});

app.get('/hash-safe', async (req, res) => {
  // ✅ GOOD: Offloads work to the thread pool (libuv) via async. Main thread is free!
  crypto.pbkdf2('password', 'salt', 100000, 512, 'sha512', (err, hash) => {
    res.send(hash);
  });
});

Worker Threads for CPU Bound Tasks

If you absolutely must process heavy CPU tasks (image resizing, machine learning, heavy math), do not do it in the main event loop. Node 10+ introduced Worker Threads to spin up completely separate V8 isolates.

const { Worker, isMainThread, parentPort } = require('worker_threads');

if (isMainThread) {
  // We are in the main server loop
  app.get('/heavy-math', (req, res) => {
    const worker = new Worker(__filename); // Pass this file to the worker
    worker.on('message', (msg) => res.json({ result: msg }));
    worker.postMessage('start'); // Trigger the worker
  });
} else {
  // We are running inside the Worker entirely separate from the web server!
  parentPort.on('message', () => {
    let count = 0;
    for (let i = 0; i < 10000000000; i++) count++;
    parentPort.postMessage(count); // Send back to main thread
  });
}
