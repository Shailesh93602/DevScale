WebSockets & Socket.io

Traditional HTTP represents a unidirectional, request-response model: the client asks for data, the server replies, and the connection closes. If the server has new data (like a chat message), it cannot initiate contact with the client. WebSockets solve this by establishing a persistent, bi-directional, real-time connection.

WebSockets vs HTTP Polling

Before WebSockets, applications used "Long Polling" — writing a setInterval on the frontend to ping the server every 2 seconds asking "Any new messages?". This creates massive overhead (HTTP headers, TCP handshakes) and strains the server. WebSockets keep a single continuous TCP pipe open, transmitting tiny, lightweight data frames instantly.

Socket.io over Native WebSockets

While browsers have a native WebSocket API, the community standard is Socket.io. Socket.io sits on top of WebSockets and provides crucial features that bare WebSockets lack:

  • Automatic Reconnection: If the user's WiFi drops, Socket.io automatically manages reconnecting and buffering messages.
  • Fallbacks: If a corporate firewall completely blocks WebSocket traffic (port 80/443 upgrade), it gracefully falls back to HTTP Long Polling.
  • Rooms & Namespaces: Easily target messages to specific groups of users (e.g., "Chat Room A" only).

Basic Socket.io Setup in Express

Because WebSockets rely on upgrading the core HTTP server, you cannot attach Socket.io directly to the Express app object. You must attach it to the underlying Node HTTP server.

const express = require('express');
const { createServer } = require('http'); // Native Node.js module
const { Server } = require('socket.io');

const app = express();
// 1. Wrap Express app inside native HTTP server
const httpServer = createServer(app);
// 2. Bind Socket.io to the HTTP server, configuring CORS for frontend
const io = new Server(httpServer, {
  cors: { origin: "http://localhost:3000" }
});

// 3. Listen for incoming connections
io.on('connection', (socket) => {
  console.log(`User connected: ${socket.id}`);
  
  // Listen for a custom event from this specific client
  socket.on('send_message', (data) => {
    console.log("Message received:", data);
    
    // Broadcast message to EVERYONE (including sender)
    io.emit('receive_message', data);
    
    // Broadcast to everyone EXCEPT the sender
    // socket.broadcast.emit('receive_message', data);
  });

  socket.on('disconnect', () => {
    console.log(`User disconnected: ${socket.id}`);
  });
});

// IMPORTANT: Listen on httpServer, NOT app!
httpServer.listen(3001, () => {
  console.log('Server running on port 3001');
});

Rooms: Grouping Connections

Rooms are arbitrary channels that sockets can join or leave. They are perfect for chat rooms, game lobbies, or specific document editing sessions.

io.on('connection', (socket) => {
  
  socket.on('join_room', (roomId) => {
    socket.join(roomId);
    console.log(`User ${socket.id} joined room: ${roomId}`);
    
    // Notify others in the room
    socket.to(roomId).emit('notification', `A new user joined!`);
  });

  socket.on('send_room_message', ({ roomId, message }) => {
    // Emits specifically to clients in that room ONLY
    io.to(roomId).emit('receive_room_message', message);
  });

});

Authenticating Socket Connections

It is dangerous to allow anyone to connect to your WebSockets. You should use a middleware in Socket.io to verify JWT tokens before allowing the connection to establish.

const jwt = require('jsonwebtoken');

// Socket.io middleware runs before 'connection' is finalized
io.use((socket, next) => {
  // Clients pass the token inside the socket "auth" object during initialization
  const token = socket.handshake.auth.token;
  
  if (!token) return next(new Error("Authentication error: No Token"));

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    // Attach user payload to the socket object!
    socket.user = decoded; 
    next();
  } catch(err) {
    next(new Error("Authentication error: Invalid Token"));
  }
});

io.on('connection', (socket) => {
  console.log(`Authenticated user connected: ${socket.user.userId}`);
});

Scaling WebSockets (The Redis Adapter)

Stateful WebSockets clash with stateless horizontally-scaled servers. If you have 3 Node.js servers behind a Load Balancer, User A might connect to Server 1, and User B to Server 2. If User A emits a message to a room, Server 1 doesn't know User B exists on Server 2!

To solve this, you must use the Socket.io Redis Adapter. It utilizes a Redis database as a Pub/Sub message broker. When Server 1 emits a message, it sends it to Redis, which silently forwards it to all other servers, ensuring User B gets the message seamlessly.
