OAuth 2.0 Scopes and Permissions

When you use OAuth 2.0 to log in to an application using your Google or GitHub account, you've likely seen a screen asking for permission to "view your email address" or "access your public repositories." These specific permissions are handled through Scopes.

What are Scopes?

Scopes are a mechanism in OAuth 2.0 to limit an application's access to a user's account. An application can request one or more scopes, and the "access token" it receives will be limited to only those specific permissions.

  • Least Privilege: Scopes allow for the principle of least privilege. An app that only needs to see your username shouldn't be given permission to delete your files.
  • User Consent: The authorization server shows the requested scopes to the user, who must explicitly agree to grant them.

Common Scope Examples

Different identity providers use different naming conventions for their scopes, but common patterns include:

  • openid: (OIDC) Requests that the user's ID be returned in the ID token.
  • profile: Requests access to the user's basic profile information (name, picture, etc.).
  • email: Requests access to the user's primary email address.
  • repo: (GitHub) Requests full access to private repositories.
  • https://www.googleapis.com/auth/drive.readonly: (Google) Requests read-only access to a user's Google Drive files.

The Scope Request Flow

  1. Authorization Request: The client application redirects the user to the authorization server, including a scope parameter in the URL (e.g., scope=user:email repo:status).
  2. Consent Screen: The authorization server authenticates the user and shows a screen listing the requested permissions.
  3. Token Issuance: If the user consents, the server issues an access token. This token internally "knows" which scopes were granted.
  4. Resource Access: When the client uses the token to call an API, the Resource Server (the API) checks the token's scopes to ensure the requested action is permitted.

Implementing Scopes in Your API

If you're building an API that acts as an OAuth Resource Server, you must verify scopes on your protected endpoints. In Express.js, this is often done using middleware:

const checkScopes = (requiredScopes) => {
  return (req, res, next) => {
    const tokenScopes = req.user.scopes; // Assume scopes were added to req.user by auth middleware
    const hasAccess = requiredScopes.every(s => tokenScopes.includes(s));
    
    if (!hasAccess) {
      return res.status(403).json({ error: "Insufficient scope" });
    }
    next();
  };
};

app.get('/api/private-data', checkScopes(['read:data']), (req, res) => {
  res.json({ data: "Sensitive info" });
});

Best Practices

  • Be Specific: Only request the minimum scopes your application actually needs.
  • Explain Why: If you're requesting sensitive scopes, explain to the user within your app's UI why those permissions are necessary.
  • Handle Revocation: Users can revoke scopes at any time through the identity provider's settings. Your application should gracefully handle "Insufficient Scope" errors.
  • Security Audits: Regularly review the scopes your application requests to ensure they are still necessary and appropriate.
