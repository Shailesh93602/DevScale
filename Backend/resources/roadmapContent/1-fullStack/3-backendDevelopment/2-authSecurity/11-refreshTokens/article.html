Understanding Refresh Tokens

In token-based authentication (like JWT), security is often improved by using very short-lived access tokens. However, forcing a user to log in every 15 minutes would be a terrible user experience. This is where Refresh Tokens come into play.

The Problem with Short-Lived Access Tokens

An access token is used to authenticate every request to the API. If an access token is stolen, the attacker has full access to the user's account. By making tokens expire quickly (e.g., in 15 minutes), we limit the window of opportunity for an attacker. But this creates the problem of frequent user re-authentication.

The Refresh Token Solution

A refresh token is a special, long-lived token that is issued alongside the access token. Its only purpose is to request a new access token when the current one expires.

  • Access Token: Short-lived (15 mins), used for API requests, stored in memory or a short-lived cookie.
  • Refresh Token: Long-lived (7 days, 30 days, or indefinite), used only to get new access tokens, stored in a secure httpOnly cookie.

The Refresh Flow

  1. Initial Login: User logs in; server sends back an Access Token and a Refresh Token.
  2. Standard Request: Client sends the Access Token with every API request.
  3. Token Expiration: The Access Token expires. The next request returns a 401 Unauthorized error.
  4. Refresh Request: The client detects the 401 error and sends the Refresh Token to a specific /refresh endpoint on the server.
  5. New Token Issuance: The server verifies the Refresh Token. If valid, it sends back a new, fresh Access Token.
  6. Retry: The client retries the originally failed request with the new Access Token. All of this happens invisibly to the user.

Security Benefits of Refresh Tokens

  • Revocation: Unlike stateless access tokens, refresh tokens are usually stored in a database on the server. This allows the server to "blacklist" or delete a refresh token, effectively logging the user out globally and immediately preventing them from getting new access tokens.
  • Rotating Refresh Tokens: Each time a refresh token is used, the server can issue a new refresh token and invalidate the old one. This helps detect token theft; if an old refresh token is reused, it's a sign of a compromise, and the server can invalidate all tokens for that user.

Where to Store Tokens?

For maximum security:

  1. Access Tokens: Store in application memory (variable). This way, they are deleted if the page is refreshed or the tab is closed.
  2. Refresh Tokens: Store in a httpOnly, secure, SameSite=Strict cookie. This protects it from XSS and CSRF attacks.
