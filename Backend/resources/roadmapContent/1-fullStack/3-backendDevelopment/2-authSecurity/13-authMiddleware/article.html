Implementing Authorization Middleware in Express

In a large application with many routes, writing a permission check for every single endpoint is tedious and error-prone. In Express.js, the most elegant and scalable solution is to use Middleware to handle authorization.

What is Authorization Middleware?

Middleware in Express is a function that has access to the request (req) and response (res) objects and the next middleware function (next). Authorization middleware specifically checks if a user is allowed to proceed to the route handler based on their authentication status and permissions.

Example: Authentication Middleware

Before you can authorize a user, you must authenticate them. This middleware verifies a JWT and attaches the user data to the request object.

const jwt = require('jsonwebtoken');

const authenticate = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  
  if (!token) return res.status(401).json({ error: "No token provided" });

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded; // Attach user info (including role) to req
    next();
  } catch (err) {
    res.status(403).json({ error: "Invalid token" });
  }
};

Example: RBAC Authorization Middleware

This middleware checks if the authenticated user has one of the required roles to access a specific route.

const authorize = (allowedRoles) => {
  return (req, res, next) => {
    if (!req.user || !allowedRoles.includes(req.user.role)) {
      return res.status(403).json({ error: "Forbidden: Insufficient permissions" });
    }
    next();
  };
};

// Usage in routes
app.get('/admin/users', authenticate, authorize(['ADMIN']), (req, res) => {
  // Only admins can see this
});

app.get('/posts', authenticate, authorize(['ADMIN', 'USER']), (req, res) => {
  // Both admins and users can see this
});

Benefits of This Approach

  • DRY (Don't Repeat Yourself): Write the permission logic once and reuse it across hundreds of routes.
  • Readability: Looking at a route definition immediately tells you who has permission to access it.
  • Gating: You can easily gate entire sections of your API by applying middleware at the router level.

Best Practices

  1. Always Authenticate First: Ensure your authenticate middleware runs before any authorize middleware.
  2. Deny by Default: If a user doesn't have an explicitly allowed role, they should be forbidden by default.
  3. Fail Softly: Return clear HTTP status codes (401 for unauthenticated, 403 for unauthorized) so your frontend can handle them appropriately (e.g., redirecting to login or showing a "no access" message).
