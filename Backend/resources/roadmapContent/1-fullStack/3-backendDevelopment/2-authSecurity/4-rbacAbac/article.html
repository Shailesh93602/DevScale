Authorization: RBAC vs ABAC

Once a user is Authenticated (logged in), the server must determine what they are Authorized to do. "Can this user delete this post?" Solving this problem at scale efficiently usually involves either Role-Based Access Control (RBAC) or the more advanced Attribute-Based Access Control (ABAC).

Role-Based Access Control (RBAC)

In RBAC, permissions are assigned to Roles (e.g., Admin, Editor, Viewer), and users are assigned to those roles. This is the simplest and most common form of authorization.

Instead of checking "Can Alice delete posts?", you check "Is Alice an Admin? If yes, allow."

Implementing RBAC Middleware in Express

// 1. Definition of Roles and Permissions
const roles = {
  ADMIN: 'admin',
  EDITOR: 'editor',
  VIEWER: 'viewer'
};

// 2. High-order middleware ensuring the user has the required role
const requireRole = (...allowedRoles) => {
  return (req, res, next) => {
    // req.user is set by the earlier Authentication (JWT) middleware
    if (!req.user || !allowedRoles.includes(req.user.role)) {
      // 403 Forbidden: You are logged in, but lack permission
      return res.status(403).json({ error: "Access Denied: Insufficient Role" });
    }
    next();
  };
};

// 3. Applying the middleware to routes
// Only admins can delete users
app.delete('/api/users/:id', requireAuth, requireRole(roles.ADMIN), deleteUser);

// Editors AND Admins can create posts
app.post('/api/posts', requireAuth, requireRole(roles.ADMIN, roles.EDITOR), createPost);

The Problem with RBAC

RBAC fails when permissions depend on context.

For example, "A User can edit a post, BUT ONLY if they are the author of that post." In RBAC, being a "User" role means you can edit all posts or none. You can't easily express ownership.

Attribute-Based Access Control (ABAC)

ABAC solves the context problem by evaluating Attributes (properties of the User, the Resource, or the Environment) against boolean policies.

  • User Attributes: Department, Age, Security Clearance.
  • Resource Attributes: Document Owner, Document Status (Draft vs Published).
  • Environment Attributes: Current Time, IP Address, Device Type.

Implementing ABAC

ABAC policies are usually written as code functions or JSON policies. For complex Node apps, libraries like CASL are the industry standard for ABAC.

import { defineAbility } from '@casl/ability';

// 1. Define ABAC policy using CASL
function defineRulesFor(user) {
  return defineAbility((can, cannot) => {
    // Admins can manage everything
    if (user.role === 'admin') {
      can('manage', 'all');
    } else {
      // Regular users can only read Published posts
      can('read', 'Post', { status: 'published' });
      
      // Crucially: Users can ONLY update or delete posts where the authorId matches their own ID!
      can(['update', 'delete'], 'Post', { authorId: user.id });
    }
  });
}

// 2. Checking authorization in the controller
app.delete('/api/posts/:id', async (req, res) => {
  // 1. Fetch the resource from DB
  const post = await db.posts.findById(req.params.id);
  
  // 2. Generate the user's abilities
  const ability = defineRulesFor(req.user);

  // 3. Evaluate the ABAC rule!
  // It automatically checks if post.authorId === user.id based on our definition
  if (ability.cannot('delete', post)) {
    return res.status(403).json({ error: "You do not own this post" });
  }

  await post.delete();
  res.json({ message: "Post deleted" });
});

Hybrid Approach

Most modern applications use a Hybrid approach: RBAC to grant sweeping permissions across the organization (Administrators vs Guests), and ABAC to handle fine-grained row-level security (Ownership checks, temporal checks like "Cannot approve after 5:00 PM").

Status Codes: 401 vs 403

  • 401 Unauthorized: "I don't know who you are. Please log in." (Failed Authentication)
  • 403 Forbidden: "I know exactly who you are, but you aren't allowed to do this." (Failed Authorization)
