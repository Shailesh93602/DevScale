Stateful Session Management

While stateless JWTs (JSON Web Tokens) are popular for mobile apps and distributed microservices because they require no database lookups, they have a massive flaw: They cannot be revoked instantly. If a user's JWT is stolen, or if you ban a user, their JWT remains valid until it mathematically expires. For highly secure applications (banking, healthcare, admin panels), Stateful Sessions are the industry standard.

How Stateful Sessions Work

Instead of sending the client a cryptographic puzzle containing their identity (JWT), the server gives the client a random, meaningless string (a Session ID).

  1. The user logs in with their password.
  2. The server generates a secure, random string: sess_abc12345XYZ.
  3. The server saves this string in a fast database (like Redis) alongside the user's ID: { "sess_abc12345XYZ": { userId: 5 } }.
  4. The server sends the string to the browser inside an HttpOnly Cookie.
  5. On the next request, the browser automatically sends the cookie. The server looks up sess_abc12345XYZ in Redis, sees it belongs to User 5, and authorizes the request!

The Redis Advantage

Why use Redis instead of PostgreSQL/MongoDB for sessions? Because the server must lookup the session on every single HTTP request. Querying a hard drive database 100 times a second per user is incredibly slow. Redis is an In-Memory database, meaning lookups happen in microseconds.

Implementing with Express-Session and Redis

const express = require('express');
const session = require('express-session');
const RedisStore = require('connect-redis').default;
const { createClient } = require('redis');

const app = express();

// 1. Initialize Redis Client
const redisClient = createClient({ url: 'redis://localhost:6379' });
redisClient.connect().catch(console.error);

// 2. Configure Session Middleware
app.use(session({
  // Tell Express to save sessions in Redis, not RAM!
  store: new RedisStore({ client: redisClient }), 
  
  secret: process.env.SESSION_SECRET, // Signs the cookie so the user can't tamper with it
  resave: false,
  saveUninitialized: false, // Don't create a session until the user actually logs in
  
  cookie: {
    secure: process.env.NODE_ENV === 'production', // true requires HTTPS!
    httpOnly: true, // Crucial! Prevents frontend JS (XSS attacks) from reading the cookie
    maxAge: 1000 * 60 * 60 * 24 * 7 // 1 Week
  }
}));

// 3. Login Route
app.post('/login', async (req, res) => {
  const user = await authenticateUser(req.body.email, req.body.password);
  
  if (user) {
    // Magic happens here: this permanently attaches data to this user's session in Redis!
    req.session.userId = user.id;
    req.session.role = user.role;
    res.json({ message: "Logged in successfully" });
    // Express automatically sends the 'set-cookie' header back to the browser
  } else {
    res.status(401).json({ error: "Invalid credentials" });
  }
});

Protecting Routes

Protecting routes is trivial because the session middleware automatically attaches the req.session object if the browser sends a valid cookie.

const requireLogin = (req, res, next) => {
  if (!req.session.userId) {
    return res.status(401).json({ error: "Unauthorized" });
  }
  next();
};

app.get('/dashboard', requireLogin, (req, res) => {
  res.json({ message: `Welcome User ${req.session.userId}` });
});

Instant Revocation (The Killer Feature)

If you suspect an account is compromised, or the user clicks "Log Out", you can instantly destroy the session. The very next request they make will fail, even if they perfectly copied their Session Cookie.

app.post('/logout', (req, res) => {
  req.session.destroy((err) => {
    if (err) return res.status(500).json({ error: "Could not log out" });
    
    // Tell the browser to delete the cookie
    res.clearCookie('connect.sid'); 
    res.json({ message: "Logged out globally" });
  });
});

Best Practices

  • Never put secure: true in development unless you are running HTTPS locally. If you do, Express will refuse to set the cookie over HTTP, and your logins will silently fail!
  • By storing sessions in Redis, you ensure your Express API remains stateless. If you run 5 Node.js servers behind a load balancer, all 5 servers look up the exact same Redis database to authorize the user.
