Passwordless Auth & WebAuthn

Passwords are fundamentally insecure. Users reuse them, forget them, and fall for phishing attacks. Modern applications are moving towards a Passwordless future using technologies like Magic Links, OTPs (One-Time Passwords), and WebAuthn (Passkeys).

1. Magic Links (Email Authentication)

A magic link is a unique, one-time-use URL sent to a user's email. Clicking the link authenticates the user. This removes the need for a password entirely and relies on the security of the user's email provider.

How to Implement Magic Links

  1. User enters their email into the login form.
  2. The backend generates a secure, random Token (e.g., crypto.randomBytes(32)) and stores it in the database with a short expiration time (e.g., 15 minutes).
  3. The backend uses a service like SendGrid or AWS SES to email the link: https://myapp.com/verify?token=abc123XYZ
  4. When the user clicks the link, the backend verifies the token exists in the database. If valid, the backend deletes the token (so it cannot be reused) and logs the user in (issues a JWT or Session Cookie).
// 1. Generate and Email the Token
app.post('/auth/magic-link', async (req, res) => {
  const { email } = req.body;
  const user = await db.users.findOne({ email });
  
  if (!user) return res.status(404).json({ error: "User not found" });

  const token = crypto.randomBytes(32).toString('hex');
  const expiresAt = Date.now() + 15 * 60 * 1000; // 15 mins

  await db.magicTokens.create({ userId: user.id, token, expiresAt });

  // Send email (Pseudo-code)
  await mailer.send(email, `Click here: https://myapp.com/verify?token=${token}`);
  res.json({ message: "Check your email!" });
});

// 2. Verify the Token and Login
app.get('/verify', async (req, res) => {
  const { token } = req.query;
  const magicToken = await db.magicTokens.findOne({ token });

  if (!magicToken || magicToken.expiresAt < Date.now()) {
    return res.status(401).send("Invalid or expired link");
  }

  // Found it! Log them in and DESTROY the token immediately.
  await db.magicTokens.deleteOne({ _id: magicToken._id });
  
  // Issue your real session cookie / JWT here
  res.redirect('/dashboard'); 
});

2. OTPs via SMS (One-Time Passwords)

Used widely in FinTech and mobile apps. Instead of an email link, the user receives a 6-digit code via SMS (usually via Twilio or an equivalent service). The backend stores the hashed 6-digit code with an expiration. When the user submits the code in the UI, the backend verifies it.

Security Note: SMS is vulnerable to SIM-swapping attacks. Authenticator Apps (like Google Authenticator using TOTP) are much more secure for highly sensitive data.

3. WebAuthn & Passkeys (The Future)

WebAuthn is a web standard allowing servers to register and authenticate users using public-key cryptography instead of passwords. It delegates authentication to the user's device (e.g., Apple FaceID, Windows Hello, Android Fingerprint, or YubiKey physical tokens). A "Passkey" is an implemention of this standard that syncs across the user's devices (like their iCloud Keychain).

The WebAuthn Architecture

  1. Registration: The server sends a "challenge" to the browser. The browser prompts the user for their fingerprint. The device generates a private/public key pair securely inside its hardware enclave. The browser sends the Public Key back to the backend. The backend stores it. The Private Key NEVER leaves the user's phone.
  2. Login: The server sends a new "challenge". The browser prompts the user for their fingerprint. The device uses its hidden Private Key to cryptographically sign the challenge. The server receives the signature and verifies it mathematically using the stored Public Key.

Because the Private Key never leaves the phone's secure hardware chip, Phishing is mathematically impossible! Even if a user visits a fake website, the device will refuse to sign the challenge for the wrong domain.

Implementing WebAuthn in Node.js

Implementation is highly complex, utilizing libraries like @simplewebauthn/server. You must manage challenges on the server and use the navigator.credentials.create() API on the frontend.

// High level overview using simplewebauthn
const { generateAuthenticationOptions, verifyAuthenticationResponse } = require('@simplewebauthn/server');

// 1. Get Login Options
app.get('/generate-authentication-options', async (req, res) => {
  // Generate a random challenge and fetch the user's stored Public Key
  const options = await generateAuthenticationOptions({
    allowCredentials: [{ id: user.publicKeyId, type: 'public-key' }],
    userVerification: 'preferred',
  });
  
  // Save challenge to the session to verify later
  req.session.currentChallenge = options.challenge;
  res.json(options);
});

// 2. Verify the Signed Assertion
app.post('/verify-authentication', async (req, res) => {
  // The frontend sends the signed payload back
  const body = req.body;
  const expectedChallenge = req.session.currentChallenge;

  const verification = await verifyAuthenticationResponse({
    response: body,
    expectedChallenge,
    expectedOrigin: 'https://myapp.com', // Prevents Phishing perfectly!
    authenticator: user.savedAuthenticator,
  });

  if (verification.verified) {
    req.session.loggedIn = true;
    return res.json({ success: true });
  }
});
