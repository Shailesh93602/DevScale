Access Control Lists (ACL)

While Role-Based Access Control (RBAC) is excellent for managing broad permissions across groups of users, some scenarios require a more granular approach. Access Control Lists (ACLs) allow you to define permissions for specific users on specific resources.

What is an Access Control List?

An ACL is a table or list that tells a computer system which access rights each user has to a particular object, such as a file, a database record, or an API endpoint. Think of it like a guest list for a private event: even if someone has a "General Admission" role (RBAC), the ACL determines if they are allowed into a specific VIP room.

RBAC vs. ACL

Understanding when to use each is key to building a flexible security system:

  • RBAC: "All users with the 'Editor' role can edit any article."
  • ACL: "Only user 'JohnDoe' can edit the article with ID 123, even though other users might have the 'Editor' role."

In many complex applications, both systems are used together. RBAC handles general categorization, while ACL handles the specific ownership and fine-grained permissions.

The Components of an ACL

A typical ACL entry (often called an Access Control Entry or ACE) includes three main pieces of information:

  1. The Subject: Who is requesting access? (A specific user ID or a group ID).
  2. The Object: What are they trying to access? (A specific file, record, or resource ID).
  3. The Operation: What do they want to do? (Read, Write, Delete, Execute).

Implementing ACL in a Database

For a web application, an ACL is often represented as a database table:

| resource_id | user_id | permission |
|-------------|---------|------------|
| doc_101     | user_1  | READ       |
| doc_101     | user_2  | WRITE      |
| doc_202     | user_1  | DELETE     |

When a request comes in, the server queries this table to see if the authenticated user_id has the required permission for the requested resource_id.

Best Practices for ACLs

  • Deny by Default: If there is no specific entry in the ACL for a user and resource, access should be denied.
  • Inheritance: In some systems, it's useful to allow permissions to be inherited (e.g., if you have access to a folder, you automatically have access to all files inside it).
  • Audit Logs: Always keep a record of whenever an ACL is modified, as these changes can have significant security implications.
  • Performance: Because ACL checks happen on almost every resource access, ensure your ACL table is properly indexed to prevent becoming a bottleneck.
