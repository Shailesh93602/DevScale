Mutations & Input Types

While Query in GraphQL is equivalent to a REST GET request, Mutations act as your POST, PUT, and DELETE. Mutations modify data on the server and then return a value, all within the exact same HTTP request.

Defining a Mutation

In your GraphQL schema, Mutations are defined under a special root type called Mutation. Just like queries, mutations define what they accept as arguments and what they return.

# The Schema (TypeDefs)
type Mutation {
  # Accepts two string arguments, and returns the modified User object
  updateUserName(id: ID!, newName: String!): User
}

The resolver for this mutation looks identical to a query resolver, handling the database logic.

// The Resolvers
const resolvers = {
  Mutation: {
    // parent, args, context, info
    updateUserName: async (_, args) => {
      const user = await db.users.findById(args.id);
      if (!user) throw new Error("User not found");
      
      user.name = args.newName;
      await user.save();
      
      // Return the updated object so the frontend UI can refresh instantly
      return user; 
    }
  }
};

Input Types (The Clean Way)

When creating a complex object (like a new User), passing 10 arguments directly into the mutation signature (createUser(name: String, email: String, age: Int...)) becomes messy and unreadable.

GraphQL solves this with Input Types. An input is a special object type strictly used for passing complex data as an argument.

# The Schema
input CreateUserInput {
  name: String!
  email: String!
  age: Int
  role: String = "USER" # Default value
}

type Mutation {
  # Now the mutation accepts a single clean object argument
  createUser(input: CreateUserInput!): User!
}
// The Resolver
const resolvers = {
  Mutation: {
    createUser: async (_, { input }) => {
      // { input } is perfectly formatted as { name, email, age, role }
      const newUser = await db.users.create(input);
      return newUser;
    }
  }
};

Standardized Mutation Payload Patterns

Returning just the modified `User` object works, but what if the mutation fails? Returning `null` isn't very helpful for the frontend to display an error. A best practice in Enterprise GraphQL is returning a Payload Object that contains a status, an error message, AND the modified data.

# The Schema
type UserPayload {
  success: Boolean!
  message: String
  user: User
}

type Mutation {
  deleteUser(id: ID!): UserPayload!
}
// The Resolver
deleteUser: async (_, { id }) => {
  try {
    const deletedUser = await db.users.findByIdAndDelete(id);
    if (!deletedUser) {
      return { success: false, message: "User not found", user: null };
    }
    return { success: true, message: "Successfully deleted", user: deletedUser };
  } catch (err) {
    return { success: false, message: err.message, user: null };
  }
}

Executing Mutations on the Client

When executing mutations from the frontend (using a library like Apollo Client or fetch), you write a GraphQL Document specifying exactly what data you want back after the mutation succeeds.

mutation CreateNewUser($userData: CreateUserInput!) {
  createUser(input: $userData) {
    # What I want the server to return after creating
    id
    name
    email
  }
}

Best Practices

  • Always use Input Types when a mutation takes more than 2 or 3 arguments to keep your schema readable.
  • Always return the newly modified object (with its ID) from a mutation. Apollo Client uses this ID to automatically update its local cache, preventing you from having to write manual Redux state updates!
