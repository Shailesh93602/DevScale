Context & Authentication in GraphQL

Unlike REST APIs which have many separate endpoints (/users, /posts) where authentication middleware can be attached selectively, a GraphQL API has exactly ONE endpoint (/graphql). You cannot easily block the entire endpoint because some queries (like getPublicPosts) must be accessible to anonymous users.

Solving authentication in GraphQL relies on the Context Context Object.

The Global Context Object

In GraphQL resolvers, the third argument is the context: (parent, args, context, info). The Context is a plain JavaScript object that is created once, globally, for every single incoming HTTP request. It is shared across all resolvers.

This makes it the perfect place to execute authentication logic: extract the JWT from the HTTP headers, verify it, and put the user's data into the context for any resolver to access if needed.

Setting up Authentication with Apollo Server

When you initialize Apollo Server, you define a context function. This function receives the Express req object.

const { ApolloServer } = require('@apollo/server');
const jwt = require('jsonwebtoken');

const server = new ApolloServer({
  typeDefs,
  resolvers,
});

// Using the Express middleware integration
app.use('/graphql', expressMiddleware(server, {
  // This function runs on EVERY incoming GraphQL request!
  context: async ({ req }) => {
    // 1. Extract the Authorization header
    const token = req.headers.authorization?.split(' ')[1] || '';
    
    // 2. Default state: not logged in
    let user = null;

    if (token) {
      try {
        // 3. Verify the token mathematically
        const decodedPayload = jwt.verify(token, process.env.JWT_SECRET);
        // 4. Optionally fetch the full user from the DB if needed
        user = await db.users.findById(decodedPayload.id); 
      } catch (err) {
        // Token is invalid/expired. We don't throw an error here, 
        // because maybe they are requesting a public resource!
        console.warn("Invalid token provided");
      }
    }

    // 5. Return the Context object. 
    // This object is injected as the 3rd argument into EVERY resolver!
    return { user, db, req };
  },
}));

Authorizing Resolvers

Now that the context.user is populated globally, you can protect specific queries and mutations simply by checking if the user exists inside the resolver.

const resolvers = {
  Query: {
    // 🟢 Public Endpoint: We don't care if context.user is null
    getPublicPosts: async (_, args, context) => {
      return await context.db.posts.find({ isPublic: true });
    },

    // 🔴 Protected Endpoint: Must be logged in!
    getMyProfile: async (_, args, context) => {
      if (!context.user) {
        // Throwing a GraphQL Error will safely reach the frontend UI
        throw new Error("You must be logged in to view your profile");
      }
      return context.user;
    }
  },
  
  Mutation: {
    // 🔴 Protected Endpoint: Must be logged in AND an Admin!
    deleteUser: async (_, { id }, context) => {
      if (!context.user) throw new Error("Not logged in");
      if (context.user.role !== 'ADMIN') throw new Error("Forbidden: Admins only");
      
      return await context.db.users.findByIdAndDelete(id);
    }
  }
};

Dependency Injection via Context

Notice that we also attached the db connection in the context setup function above. Passing the database, API keys, and caching mechanisms into the Context object is a common architectural pattern known as Dependency Injection. It prevents you from having to use require('../../db') in multiple resolver files, and makes writing unit tests much easier since you can simply inject a fake database during testing.

Best Practices

  • Do not throw strict HTTP 401 Unauthorized errors in the main context initialization function. Doing so will completely crash the request and prevent users from viewing public queries without a token. Let the specific resolver decide if it requires authentication.
  • Use custom GraphQL Error extensions (like GraphQLError('Not Authorised', { extensions: { code: 'UNAUTHENTICATED' } })) rather than generic JS Errors so your frontend Apollo Client can automatically intercept and redirect to the login page cleanly.
