Apollo Server Integration

While you can use raw graphql-js and express-graphql to build an engine, the industry standard for production GraphQL APIs is Apollo Server. It provides extensive tooling, easy Express integration, error formatting, and a beautiful web-based IDE (Apollo Sandbox).

Setting Up Apollo Server with Express

Apollo Server v4 utilizes a modular approach. While it can run standalone, it is almost always integrated directly into an existing Express application as a middleware. This enables you to combine REST endpoints (e.g. for simple file uploads or webhooks) and GraphQL simultaneously.

// 1. Install dependencies
// npm install @apollo/server express graphql cors

const express = require('express');
const { ApolloServer } = require('@apollo/server');
const { expressMiddleware } = require('@apollo/server/express4');
const cors = require('cors');

// Import your schema and resolvers
const { typeDefs } = require('./schema');
const { resolvers } = require('./resolvers');

async function startServer() {
  const app = express();

  // 2. Initialize Apollo Server
  const server = new ApolloServer({
    typeDefs,
    resolvers,
    // Optional configuration
    formatError: (formattedError, error) => {
      // Intercept and format errors before they leave the server
      if (formattedError.message.startsWith("Database Error: ")) {
        return { message: 'Internal server error' };
      }
      return formattedError;
    },
  });

  // 3. You MUST await server.start() before applying middleware in v4!
  await server.start();

  // 4. Apply Apollo Middleware to a generic Express route
  app.use(
    '/graphql',
    cors(),
    express.json(), // Body parsing is required for Apollo to read the query strings
    expressMiddleware(server, {
      // The context function (For JWT Auth, DataLoaders, etc.)
      context: async ({ req }) => {
        return { token: req.headers.authorization };
      },
    })
  );

  // You can still have traditional REST endpoints!
  app.get('/health', (req, res) => res.send("OK"));

  app.listen(4000, () => {
    console.log(`🚀 GraphQL API ready at http://localhost:4000/graphql`);
  });
}

startServer();

Apollo Sandbox & GraphQL Explorer

In older versions, Apollo utilized a tool called GraphQL Playground. In v4, going to http://localhost:4000/graphql in your browser automatically launches the Apollo Sandbox.

Because your server supports Introspection (the ability for the schema to advertise its own structure), the Sandbox automatically generates advanced documentation, auto-completion, and a UI to execute Queries and Mutations graphically without writing a frontend app.

Security Note: You should disable introspection in production so hackers cannot map your entire database schema natively.

const server = new ApolloServer({
  typeDefs,
  resolvers,
  // Disable Introspection in Prod!
  introspection: process.env.NODE_ENV !== 'production'
});

Schema Stitching vs Apollo Federation

As your company grows into a microservices architecture, having exactly one massive GraphQL server becomes messy (a monolithic schema). There are two ways to scale a GraphQL architecture:

  1. Schema Stitching: Programmatically combining multiple smaller GraphQL schemas together. (Older technique)
  2. Apollo Federation: The modern industry standard. A Gateway server sits in front of multiple smaller "Subgraph" APIs. The Gateway intelligently distributes queries to the correct microservice (e.g. User Microservice and Inventory Microservice) and automatically merges the results into a single object for the client.

Best Practices

  • Place the expressMiddleware(server) function specifically on the /graphql path. Doing this allows you to serve standard REST webhooks (like Stripe) on /webhooks without interference.
  • Leverage custom Apollo Error classes (e.g. GraphQLError) for typed error handling (Unauthenticated, BadInput) so the Apollo Client frontend can easily interpret them rather than just returning 500 crashes.
