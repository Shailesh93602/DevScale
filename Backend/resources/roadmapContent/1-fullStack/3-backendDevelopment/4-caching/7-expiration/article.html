Understanding Cache Expiration and TTL

In caching, TTL (Time To Live) is the amount of time a piece of data remains valid in the cache before it's automatically deleted. Setting the correct TTL is a balancing act: a TTL that's too long leads to stale data, while a TTL that's too short increases database load.

What is Time To Live (TTL)?

TTL is typically measured in seconds. In Redis, you can set a TTL whenever you store a key using the SET command with the EX (seconds) or PX (milliseconds) option.

// Cache a user's profile for 1 hour (3600 seconds)
await redis.set('user:123', JSON.stringify(userData), 'EX', 3600);

Choosing the Right TTL

  • Short TTLs (seconds/minutes): Ideal for data that changes frequently (e.g., real-time sports scores, currency exchange rates).
  • Medium TTLs (hours/days): Perfect for data that changes daily or where near-real-time accuracy is acceptable (e.g., top news stories, user profile info).
  • Long TTLs (weeks/months): Good for truly static content or data that practically never changes (e.g., country lists, historical data).

Absolute vs. Relative Expiration

  • Relative Expiration: The TTL starts counting when the key is *last accessed* (sliding window). If a key is popular, it stays in the cache forever.
  • Absolute Expiration: The TTL starts counting from the moment the key was *placed* in the cache, regardless of how often it's accessed. This ensures that even popular keys are eventually refreshed with fresh data.

The "Infinite" Cache Fallacy

While you *can* set keys and never expire them, you should almost never do this in a production application. If your database changes but your cache doesn't expire, you'll eventually run into bugs where users see completely incorrect or outdated information.

Best Practices Summary

  1. Every Key Needs a TTL: Unless you have a very specific reason not to, always set an expiration time for every cache key you create.
  2. Implement TTL Randomization (Jitter): If you set 10,000 keys with the exact same 1-day TTL, they will all expire at the exact same second tomorrow, causing a massive spike in database load. Add a bit of random "jitter" to your TTLs (e.g., 24 hours ± 10 minutes) to spread out the expiration events.
  3. Use Absolute Expiration for Consistency: For data that absolute must be refreshed periodically, use absolute expiration to ensure the cache stays in sync with the primary database.
