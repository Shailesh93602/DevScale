Understanding SQL Data Types

Every column in a SQL table must have a defined **Data Type**. Choosing the correct data type is critical for performance, storage efficiency, and ensuring data integrity. While exact names vary slightly between databases (PostgreSQL vs. MySQL), the general categories remain the same.

1. Numeric Types

  • INT / INTEGER: For whole numbers (e.g., age, count).
  • DECIMAL / NUMERIC: For exact fractional numbers (essential for financial data and currency).
  • FLOAT / REAL: For approximate fractional numbers (used in scientific calculations where small errors are acceptable).
  • BOOLEAN: For true/false values (though some databases like MySQL use TINYINT(1)).

2. Character (String) Types

  • CHAR(n): Fixed-length string. If the string is shorter than n, it's padded with spaces.
  • VARCHAR(n): Variable-length string with a maximum of n. This is the most common string type.
  • TEXT: For long strings of unlimited (or very large) length (e.g., blog post content).

3. Date and Time Types

  • DATE: Stores only the date (YYYY-MM-DD).
  • TIME: Stores only the time (HH:MM:SS).
  • DATETIME / TIMESTAMP: Stores both date and time. Use TIMESTAMPTZ in PostgreSQL for time-zone awareness.

4. Advanced Types (NoSQL in SQL)

  • JSON / JSONB: Allows you to store unstructured or semi-structured JSON data directly in a SQL column. JSONB (Binary JSON) in PostgreSQL is highly optimized and searchable.
  • UUID: A universally unique identifier. Preferred over sequential integers for primary keys in distributed systems.
  • BLOB (Binary Large Object): For storing raw binary data like images or PDFs (though storing these as files in a CDN is generally preferred).

Best Practices Summary

  1. Be Specific: Don't use TEXT for everything. If a name will never exceed 100 characters, use VARCHAR(100). This helps the database engine optimize memory.
  2. Use DECIMAL for Money: Never use FLOAT for currency, as floating-point math can lead to precision errors (e.g., 0.1 + 0.2 != 0.3).
  3. Default to TIMESTAMPTZ: In global applications, always store timestamps with time zone information to avoid confusion when users in different regions interact.
  4. Use JSONB Wisely: Only use JSONB for data that is truly dynamic. If your data has a fixed structure, traditional columns are faster and more efficient.
