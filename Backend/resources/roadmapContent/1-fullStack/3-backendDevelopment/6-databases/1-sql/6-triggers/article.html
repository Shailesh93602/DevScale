Advanced SQL: Database Triggers

A Trigger is a special type of stored procedure that is automatically executed ("fired") in response to a specific event on a particular table. This allows the database to enforce complex business rules and data consistency automatically, without requiring explicit application-level code.

How Triggers Work

Triggers are attached to a table and can be configured to fire before or after an INSERT, UPDATE, or DELETE operation.

  • BEFORE Trigger: Good for validating or modifying data *before* it's actually written to the table (e.g., ensuring a price is never negative).
  • AFTER Trigger: Good for performing related actions *after* the primary change has succeeded (e.g., logging activity or updating an audit trail).

Common Use Cases for Triggers

  • Auditing: Automatically record every change to a sensitive table (like "Bank Accounts") into a separate audit log table.
  • Automatic Updates: When a new product is added, automatically update the "Last Updated" timestamp on the parent Category table.
  • Data Validation: Enforce complex rules that are difficult to express with simple constraints (e.g., a user can only have one active subscription at a time).
  • Synchronization: When a user updates their profile, automatically update a copy of their name in a cached "Search Results" table.

Creating a Simple Trigger

-- Creating an audit trigger to track user email changes
CREATE TRIGGER LogEmailChange
AFTER UPDATE ON users
FOR EACH ROW
BEGIN
    -- Only log if the email was actually changed
    IF OLD.email != NEW.email THEN
        INSERT INTO audit_logs (user_id, old_email, new_email, changed_at)
        VALUES (OLD.id, OLD.email, NEW.email, NOW());
    END IF;
END;

The Dangers of Triggers

Triggers are powerful but can lead to "hidden magic" in your database that makes debugging very difficult. Overusing triggers can also slow down your database performance significantly, as every single write operation now triggers additional hidden operations.

Best Practices Summary

  1. Use Sparingly: Prefer simple Check Constraints or application-level logic for most validations. Save triggers for truly essential data consistency and auditing needs.
  2. Avoid Long-Running Code: Triggers should be incredibly fast. Never perform complex calculations or external API calls inside a trigger.
  3. Don't Create Cycles: Be careful not to create "trigger chains" where a trigger on Table A updates Table B, which in turn triggers an update back on Table A (infinite loop!).
