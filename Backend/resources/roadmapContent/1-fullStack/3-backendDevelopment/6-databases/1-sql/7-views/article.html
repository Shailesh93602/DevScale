Advanced SQL: Database Views

A View is a virtual table in a SQL database whose contents are defined by a query. It doesn't store data physically (unless it's a "Materialized View"); instead, it acts as a permanent, named window into one or more underlying tables.

What is a View?

Think of web development with React: a view is like a "Component" that combines props (data from base tables) into a specific UI (the results of your query). When you query a view, the database engine executes the underlying SELECT statement on the fly.

Benefits of Using Views

  • Simplification: You can hide complex JOINs and calculations behind a simple view name. Users can then query the view as if it were a regular table.
  • Security: You can give users access to a view without giving them access to the underlying tables. This allows you to hide sensitive columns (like password hashes or social security numbers).
  • Consistency: By using views, you ensure that everyone who fetches "Active Users" or "Recent Sales" is using the exact same query logic.
  • Logical Data Independence: If you rename an underlying table or split it into two, you can just update the view's definition. The application code (which queries the view) doesn't need to change.

Creating a Simple View

-- Creating a view to simplify order reporting
CREATE VIEW ActiveOrders AS
SELECT o.id, u.name as customer_name, o.total, o.created_at
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'ACTIVE';

-- Querying the view just like a table
SELECT * FROM ActiveOrders WHERE total > 100;

What is a Materialized View?

Unlike a regular view, a Materialized View actually *stores* the result of the query physically. This provides incredible performance for very heavy queries but means the data becomes "stale" until the view is manually refreshed.

Best Practices Summary

  1. Use for Security: Always use views to present data to third-party reporting tools, ensuring they only see exactly the columns you want to expose.
  2. Simplify your Code: If your Express app is performing the same 5-table JOIN in multiple places, move that JOIN into a database view.
  3. Don't Over-Layer: Try to avoid creating "views of views of views," as this can make performance tracking and debugging extremely difficult.
