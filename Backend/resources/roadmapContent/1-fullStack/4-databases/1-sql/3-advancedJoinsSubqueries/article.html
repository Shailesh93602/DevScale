Advanced Joins & CTEs in SQL

Relational databases excel at combining data from multiple tables. Rather than executing multiple queries and combining the data manually in your Node.js or Python backend, JOINs and Subqueries allow the database engine to perform this heavy lifting natively and securely.

The 4 Primary JOINs

Understanding which JOIN to use determines exactly what data is returned or excluded.

  • INNER JOIN: The most common. Returns records ONLY if there is a matching value in both tables. (If a User has no Posts, that User is excluded from the results).
  • LEFT JOIN: Returns ALL records from the left table, and the matched records from the right table. (Returns all Users. If a User has no Posts, the Post columns will simply be NULL).
  • RIGHT JOIN: The exact opposite of a LEFT JOIN. Returns all records from the right table.
  • FULL OUTER JOIN: Returns all records when there is a match in either the left OR right table. It returns everything, filling in NULLs wherever data is missing.

Syntax Example

-- Get all users, and any posts they have written (LEFT JOIN)
SELECT u.name, u.email, p.title
FROM users u
LEFT JOIN posts p ON u.id = p.author_id;

-- Get ONLY users who have written at least one post (INNER JOIN)
SELECT u.name, u.email, p.title
FROM users u
INNER JOIN posts p ON u.id = p.author_id;

Subqueries

A Subquery is a query nested entirely inside another query. They are often used in the WHERE clause to filter data based on dynamic values, or in the SELECT clause to calculate aggregated columns.

-- Find all users who have an age greater than the average age
SELECT name, age 
FROM users 
WHERE age > (SELECT AVG(age) FROM users);

-- Find the titles of all posts written by users located in "New York"
SELECT title 
FROM posts 
WHERE author_id IN (
  SELECT id FROM users WHERE city = 'New York'
);

Common Table Expressions (CTEs)

While Subqueries are powerful, nesting them 3 or 4 levels deep makes a query completely unreadable. CTEs (Common Table Expressions) solve this by allowing you to define temporary tables at the very top of your query using the WITH clause.

CTEs act like variables in programming. You define a query, give it a name, and then query that name later down in the file.

-- 1. Define the CTE at the top
WITH NewYorkUsers AS (
  SELECT id, name FROM users WHERE city = 'New York'
),
-- 2. You can define multiple CTEs, separated by commas
PopularPosts AS (
  SELECT id, title, author_id FROM posts WHERE views > 1000
)

-- 3. The actual main query using the CTEs defined above!
SELECT ny.name, p.title 
FROM NewYorkUsers ny
INNER JOIN PopularPosts p ON ny.id = p.author_id;

Why use CTEs?

  1. Readability: They read from top to bottom, making complex analytical queries much easier for other developers to understand.
  2. Reusability: You can reference the same CTE multiple times within the main query without writing the logic out twice.
  3. Recursion: CTEs can be marked as WITH RECURSIVE to query hierarchical tree structures (like an Employee & Manager organizational chart, or deeply nested Reddit comments).
