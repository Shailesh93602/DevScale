Indexing & Performance

When an SQL table grows to millions of rows, executing a simple query like SELECT * FROM users WHERE email = 'test@example.com' can suddenly take 5 or 10 seconds. This happens because the database must perform a Full Table Scan, reading every single row one by one until it finds a match. To solve this, you must create Indexes.

What is an Index?

An Index in an SQL database works exactly like the index at the back of a 1,000-page book. Instead of flipping through all 1,000 pages to find the word "Apple," you go to the index, look up "Apple," and it tells you exactly which page it is on (e.g., Page 42). The database index is a separate, highly optimized data structure pointing to the actual rows on disk.

-- CREATE INDEX command
CREATE INDEX idx_users_email ON users(email);

B-Tree Indexes (The Default)

If you don't specify an index type, PostgreSQL uses a B-Tree (Balanced Tree). B-Trees sort data hierarchically. Because the data is perfectly sorted, the database can use a Binary Search algorithm to find a record in milliseconds, even if there are 100 million rows.

B-Trees are excellent for:

  • Exact matches (WHERE age = 25)
  • Range queries (WHERE age > 18 AND age < 30)
  • Sorting (ORDER BY age DESC)

The Cost of Indexing

If indexes make reads so fast, why not put an index on every single column in the table? Because indexes heavily damage Write Performance.

Every time you INSERT, UPDATE, or DELETE a row, the database must not only update the main table but also re-sort and update every single index attached to that table. Over-indexing causes writes to crawl to a halt and wastes massive amounts of hard drive space. Only index columns that are frequently used in WHERE, JOIN, or ORDER BY clauses.

Understanding Query Plans (EXPLAIN ANALYZE)

How do you know if your query is slow because of missing indexes? You prepend the command EXPLAIN ANALYZE to your query. This prevents the database from just returning the rows, and instead returns the Execution Plan detailing exactly how the engine fetched the data and how long each step took.

EXPLAIN ANALYZE SELECT * FROM users WHERE last_name = 'Smith';

-- Output (Simplified):
-- Seq Scan on users (cost=0.00..15462.00 rows=153 width=45) (actual time=0.012..450.000)
-- Filter: ((last_name)::text = 'Smith'::text)
-- Rows Removed by Filter: 999847
-- Execution Time: 450.034 ms

In the above example, seeing Seq Scan (Sequential Scan) means the backend read nearly 1,000,000 rows just to find the 153 people named Smith, taking almost half a second. This is a massive red flag indicating a missing index.

After adding CREATE INDEX idx_last_name ON users(last_name);, the result changes:

-- Output after Indexing:
-- Index Scan using idx_last_name on users (cost=0.42..8.44 rows=153 width=45) (actual time=0.012..0.120)
-- Index Cond: ((last_name)::text = 'Smith'::text)
-- Execution Time: 0.150 ms

The time dropped from 450ms to 0.15ms. The presence of an Index Scan ensures the database directly navigated the B-Tree structure flawlessly.
