ACID Transactions in SQL

When you build an application that handles money (like Stripe or a Bank), things will go wrong. Servers crash, code throws errors, and database connections drop. If a user transfers $100 to their friend, the database must execute two separate commands:

  1. Subtract $100 from the Sender's balance.
  2. Add $100 to the Receiver's balance.

What happens if the server crashes exactly between step 1 and step 2? The $100 disappears entirely! To prevent this, Relational Databases guarantee ACID Properties by utilizing Transactions.

What is a Transaction?

A Transaction groups multiple SQL operations into a single, indivisible block of work. It is wrapped with BEGIN and finalized with either COMMIT (save everything permanently) or ROLLBACK (undo everything instantly).

BEGIN; -- Open the transaction block

UPDATE accounts SET balance = balance - 100 WHERE id = 1; -- Execute Step 1
UPDATE accounts SET balance = balance + 100 WHERE id = 2; -- Execute Step 2

COMMIT; -- If both succeed, lock them in.
-- If Step 2 fails, you run: ROLLBACK; to restore the balance.

The ACID Properties Defined

A database must guarantee these four properties to be considered reliable:

1. Atomicity (All or Nothing)

A transaction must act as a single "Atomic" unit. If a transaction has 5 steps, and step 4 fails, steps 1, 2, and 3 are automatically aborted and rolled back. The database is never left in a partially updated state.

2. Consistency

The database must transition from one valid state to another valid state. If you defined an SQL constraint stating balance >= 0, and a transaction tries to subtract $500 from a $200 account, the database rejects the entire transaction violently to maintain Consistency.

3. Isolation

If 100 users hit the database at the exact same millisecond, their transactions might try to read/write the same row simultaneously. Isolation ensures that transactions executing concurrently produce the exact same final result as if they had executed sequentially (one after the other). The database achieves this by using Row-Level Locks.

Example: If Transaction A reads a row, and Transaction B tries to modify that row before A finishes, the database physically forces Transaction B to pause and wait until A executes a COMMIT.

4. Durability

Once a transaction is successfully completed (COMMIT), the changes are guaranteed to be permanently saved to the hard drive, even if the database server loses power 10 milliseconds later. The database writes changes to an append-only WAL (Write-Ahead Log) physically on the disk before acknowledging the commit to your backend API.

Implementing Transactions in Node.js

If you are using PostgreSQL in Node.js (via the pg library or an ORM like Prisma), you MUST manage the transaction programmatically. If your Node code crashes, you must catch the error and execute the SQL ROLLBACK command.

// Using raw 'pg' library with async/await
const client = await pool.connect();

try {
  // 1. Begin the Transaction!
  await client.query('BEGIN');

  // 2. Execute multiple queries
  await client.query('UPDATE accounts SET balance = balance - 100 WHERE id = 1');
  await client.query('UPDATE accounts SET balance = balance + 100 WHERE id = 2');

  // 3. Commit successfully
  await client.query('COMMIT');

} catch (e) {
  // 4. Critical: Catch the error and aggressively ROLLBACK!
  await client.query('ROLLBACK');
  throw e;
} finally {
  client.release(); // Return connection to pool
}

Isolation Levels

Relational Databases allow you to tune how aggressively they lock rows (Isolation). Locking rows strictly prevents bugs, but drastically slows down the database because transactions have to wait in line. Read Committed is the default in PostgreSQL. Serializable is the safest (but slowest) level, forcing transactions to behave as if they literally ran one by one in a single-file line.
