Stored Procedures & Triggers

Most modern web development relies on the Backend Server (Node.js/Django) executing business logic and then simply sending raw strings of data to the database to be stored. However, relational databases like PostgreSQL are incredibly powerful execution engines in their own right. You can write actual programming logic inside the database itself using Stored Procedures, Functions, and Triggers.

Stored Procedures & Functions (PL/pgSQL)

Instead of executing 10 different SQL queries over the network (which introduces heavy network latency), you can write a "Function" directly inside PostgreSQL using a language like PL/pgSQL (Procedural Language/PostgreSQL). This function sits on the database server. Your Node server hits it via a single CALL my_transfer_function(1, 2, 100) request.

-- Creating a function directly in PostgreSQL
CREATE OR REPLACE FUNCTION transfer_funds(sender_id INT, receiver_id INT, amount INT) 
RETURNS VOID AS $$
BEGIN
  -- We can use IF statements directly inside the database!
  IF (SELECT balance FROM accounts WHERE id = sender_id) < amount THEN
    RAISE EXCEPTION 'Insufficient Funds';
  END IF;

  -- The function natively executes transactions
  UPDATE accounts SET balance = balance - amount WHERE id = sender_id;
  UPDATE accounts SET balance = balance + amount WHERE id = receiver_id;
END;
$$ LANGUAGE plpgsql;

Triggers (Automated Database Events)

A Trigger is a special type of database function that executes automatically whenever a specific event occurs on a table, such as an INSERT, UPDATE, or DELETE.

They are incredibly powerful for maintaining data integrity without relying on the Node.js application to "remember" to do something.

Example: Creating an Audit Log automatically

If you want to track every time a user changes their email, you can rely on the Node.js controller to insert an audit log. BUT what if a developer modifies the database manually via psql? Or a microservice updates it from python? To enforce it globally, use a trigger.

-- 1. Create a function that writes to the audit_logs table
-- It has access to the special variables OLD and NEW representing the row before and after!
CREATE OR REPLACE FUNCTION log_email_change()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.email <> OLD.email THEN
    INSERT INTO audit_logs (user_id, old_email, new_email, changed_at)
    VALUES (OLD.id, OLD.email, NEW.email, NOW());
  END IF;
  
  -- Must return NEW for the original UPDATE to finish
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 2. Bind the function to a Trigger on the 'users' table!
-- Before an update actually commits to the users table, execute our function!
CREATE TRIGGER email_change_trigger
BEFORE UPDATE ON users
FOR EACH ROW
EXECUTE FUNCTION log_email_change();

Materialized Views

A standard VIEW in SQL is just a saved query (a macro) that executes every time you call it, hiding complex JOINs. It does not store data on the disk.

A Materialized View acts like a real table. It executes a massively complex analytical query (e.g., calculating the total sales for every store, taking 5 minutes to run) and physically saves the results to the hard drive holding gigabytes of cached data. To update the cache, you run REFRESH MATERIALIZED VIEW periodically on a cron job, allowing instant, microsecond query speeds for data dashboards.

Best Practices

  • Do not put all your business logic in the database! Debugging JavaScript in VSCode is wonderful. Debugging massive PL/pgSQL procedures crashing a live database is a nightmare. Limit triggers to strict data-integrity enforcement (like updated_at timestamps or audit logs).
  • If you use ORMs like Prisma, managing heavily triggered databases becomes difficult because Prisma expects the database to stay exactly as it defined it. Raw SQL migrations are required when writing heavy database-level logic.
