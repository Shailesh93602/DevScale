Prisma Client & Queries

The core philosophy of Prisma is to eliminate the bugs that happen when your database schema drifts away from your backend code. You define your database in a schema.prisma file. Then, you run npx prisma generate. Prisma reads the schema and writes a customized, heavily typed TypeScript library directly into your node_modules. This autogenerated library is the Prisma Client.

Basic CRUD Operations

Because the Client is autogenerated based on your exact schema, TypeScript knows statically that a User has an email and name. If you misspell them, VSCode throws a red error before you even run the code.

import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();

// 1. Create a user
const newUser = await prisma.user.create({
  data: {
    name: "Alice",
    email: "alice@example.com",
    // role: "ADMIN" <-- VSCode will autocomplete this!
  }
});

// 2. Read users
const users = await prisma.user.findMany({
  where: { 
    name: { startsWith: "A" } 
  },
  orderBy: { createdAt: 'desc' },
  take: 10 // Limit to 10 rows (Pagination)
});

// 3. Update a user
const updatedUser = await prisma.user.update({
  where: { email: "alice@example.com" }, // Must be a unique field!
  data: { role: "ADMIN" }
});

// 4. Delete a user
const deletedUser = await prisma.user.delete({
  where: { id: 5 }
});

Select vs Include

In standard SQL, if you do `SELECT * FROM users`, you pull down massive amounts of data you might not need (like password hashes). Prisma solves this elegantly using select and include.

  • select: Explicitly state exactly which columns you want returned. (Everything else is ignored).
  • include: Tell Prisma to automatically execute a JOIN and pull down a related table's data.

Crucial Rule: You cannot use select and include on the exact same level! If you use select, you must do the include inside the select.

// ❌ Bad: Pulls the 500kb 'bio' column and 'passwordHash'
const user = await prisma.user.findUnique({ where: { id: 1 } });

// ✅ Good: Only pulls ID and Name
const lightweightUser = await prisma.user.findUnique({
  where: { id: 1 },
  select: {
    id: true,
    name: true
  }
});

// ✅ The power of Include: Fetch user AND their posts magically
const userWithPosts = await prisma.user.findUnique({
  where: { id: 1 },
  include: {
    posts: true // Executes a JOIN automatically!
  }
});

// 🚀 Advanced: Select exactly what you want from the JOIN!
const complexQuery = await prisma.user.findUnique({
  where: { id: 1 },
  select: {
    id: true,
    email: true,     // Select from main table
    posts: {         // Execute the Join!
      select: {      // Select specifically from the joined table!
        title: true
      }
    }
  }
});
// TypeScript knows complexQuery.posts[0].title exists!

Nested Writes

In traditional ORMs, creating a User and inserting 3 Posts for them requires 4 separate queries. Prisma supports Nested Writes, allowing you to create the parent and all children in one giant, perfectly typed object.

const user = await prisma.user.create({
  data: {
    email: "bob@bob.com",
    name: "Bob",
    posts: {
      create: [
        { title: "First Post", content: "Hello" },
        { title: "Second Post", content: "World" }
      ]
    }
  }
});
// Prisma automatically inserts the User, gets the ID, and inserts the 2 Posts 
// pointing to that exact new AuthorID!

Aggregations and Grouping

Prisma supports mathematical aggregations natively.

const analytics = await prisma.order.aggregate({
  _sum: { amount: true },
  _avg: { amount: true },
  _count: { id: true },
  where: { status: 'COMPLETED' }
});

console.log(`Total Sales: $${analytics._sum.amount}`);
