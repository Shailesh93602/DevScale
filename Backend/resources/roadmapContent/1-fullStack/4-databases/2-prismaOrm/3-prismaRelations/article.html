Modeling Relations in Prisma

Relational databases thrive on relationships. Defining how your tables connect (One-to-One, One-to-Many, Many-to-Many) is the most critical part of architecting your schema.prisma file. Prisma's schema syntax makes defining these relations highly readable compared to raw SQL.

The Two Sides of a Relation

Every relationship has two sides. If a User has many Posts (1-to-N):

  1. The User side has an array: posts Post[]
  2. The Post side has a singular object: author User and it must explicitly declare the Foreign Key mapping using the @relation attribute!

One-to-Many (1-to-n)

The most common relationship. A single User creates multiple Posts.

model User {
  id    Int    @id @default(autoincrement())
  email String @unique
  
  // A User can have an array of many Posts
  posts Post[] 
}

model Post {
  id       Int    @id @default(autoincrement())
  title    String
  
  // 1. The typed Prisma property. This points to the User model.
  author   User   @relation(fields: [authorId], references: [id])
  
  // 2. The physical database Foreign Key column!
  authorId Int    
}

Notice that author isn't a column in PostgreSQL; it's a Prisma-only mapping that allows you to do include: { author: true }. The actual column physically existing in SQL is exactly authorId.

One-to-One (1-to-1)

Often used when splitting out massive data into a separate table to keep queries fast (e.g., splitting a User Profile from the core User credentials).

The syntax is nearly identical to a One-to-Many, but the parent side uses a singular object (Profile?) instead of an array (Profile[]), and crucially, the Foreign Key on the child is forced to be @unique!

model User {
  id      Int      @id @default(autoincrement())
  
  // Singular! A user can only have ONE profile (or null)
  profile Profile? 
}

model Profile {
  id     Int    @id @default(autoincrement())
  bio    String
  
  user   User   @relation(fields: [userId], references: [id])
  
  // Must be unique! If not unique, it becomes a 1-to-Many relation!
  userId Int    @unique 
}

Many-to-Many (m-to-n)

A classic example: Posts and Tags. A Post can have many Tags ("tech", "news"). A Tag can be attached to many Posts.

In standard SQL, mapping this requires you to manually build a third "Join Table" (e.g., PostTag) containing two foreign keys. Prisma brilliantly simplifies this using Implicit Many-to-Many Relations. You just define an array on both sides, and Prisma automatically generates and manages the join table behind the scenes invisibly!

model Post {
  id    Int    @id @default(autoincrement())
  title String
  
  // Array!
  tags  Tag[] 
}

model Tag {
  id    Int    @id @default(autoincrement())
  name  String
  
  // Array!
  posts Post[] 
}

When you run prisma migrate dev, Prisma will automatically execute SQL to create a hidden table named _PostToTag. You never have to manually query it!

Explicit Many-to-Many

If you need to store extra information on the relationship itself (e.g., storing the exact timestamp someone joined a group, or an employee's specific role within a specific department), you cannot use an Implicit relation. You must explicitly model the Join Table so you can add columns to it.

model User {
  id       Int             @id @default(autoincrement())
  // The User points to the Join Table array
  memberships GroupMembership[] 
}

model Group {
  id       Int             @id @default(autoincrement())
  // The Group points to the Join Table array
  memberships GroupMembership[]
}

// The Explicit Join Table!
model GroupMembership {
  user       User     @relation(fields: [userId], references: [id])
  userId     Int
  
  group      Group    @relation(fields: [groupId], references: [id])
  groupId    Int
  
  // The extra data!
  role       String   @default("MEMBER")
  joinedAt   DateTime @default(now())

  // The Primary Key is a combination of both foreign keys
  @@id([userId, groupId]) 
}
