Prisma Migrations & Deployment

When you build an application, your database schema is never finished. You will constantly add new tables, rename columns, and change data types. If you change your schema.prisma file, but the actual PostgreSQL database on your AWS server doesn't match it, your application will crash instantly. This is called Database Drift. The solution is using Migrations.

What is a Migration?

A migration is a version-controlled SQL script that tracks the exact historical changes made to your database. Instead of manually logging into your database and running ALTER TABLE users ADD COLUMN age INT;, you let Prisma examine your schema file, detect what changed, and automatically generate the SQL script to apply that change.

The Development Workflow (prisma migrate dev)

When you are building your app on your local machine, you follow a strict workflow every time you change your schema.prisma file.

  1. You add a new age Int field to the User model in schema.prisma.
  2. You run: npx prisma migrate dev --name added_user_age
  3. Prisma connects to your local development database, compares its current state to your new schema file, and detects the difference.
  4. Prisma generates a new folder inside prisma/migrations/ containing a raw .sql file (e.g., ALTER TABLE "User" ADD COLUMN "age" INTEGER NOT NULL;).
  5. Prisma executes that SQL script against your local database automatically.
  6. Prisma updates your autogenerated TypeScript Client (@prisma/client) so VSCode instantly recognizes the age property!

You then commit this migrations folder to Git/GitHub, so your teammates can pull your code, run npx prisma migrate dev, and perfectly synchronize their own local databases with yours.

Resolving Database Drift

Database drift occurs when your database schema differs from what Prisma expects it to be (often because a developer manually altered the database using a GUI tool like DBeaver or pgAdmin instead of using a Prisma migration).

If you run migrate dev and Prisma detects drift, it will warn you: "Drift detected: Your database schema is ahead of your migration history."

To forcefully fix drift during local development (WARNING: This can result in data loss!):

npx prisma migrate reset

This command drops the entire database, recreates it from scratch using your migration history, and runs your seed.ts file to refill it with dummy data.

The Production Workflow (prisma migrate deploy)

You NEVER run prisma migrate dev on a production server. The dev command is highly destructive; if it detects an unresolvable conflict, it might prompt you to reset the entire database—which would delete all your real customer data!

When deploying to production (e.g., via Vercel, Render, or a Docker container), you must use the designated production command:

npx prisma migrate deploy

How it works in CI/CD:

  1. Your server pulls the latest code from the main branch, which includes your newly committed migrations/ folder.
  2. Before starting the Node.js server, the deployment pipeline runs npx prisma migrate deploy.
  3. Prisma checks the production database's special _prisma_migrations tracking table to see which migrations have already been applied.
  4. It instantly executes any new `.sql` migrations it hasn't seen before, updating the production database structure safely.
  5. If a migration fails in production, the deploy command crashes and halts the deployment, preventing broken code from running against an older database.

Seeding the Database

When setting up a new environment, you often need default data (like an initial Admin account). Prisma supports a seed.ts script.

Define the seed script in your package.json:

"prisma": {
  "seed": "ts-node prisma/seed.ts"
}

You can execute this manually by running npx prisma db seed.
