NoSQL Design Patterns

Moving from a Relational Database (SQL) to a Document Database (MongoDB) requires a fundamental shift in how you structure data. In SQL, you spread your data across many tables and rely heavily on JOIN operations to combine them. In MongoDB, data that is accessed together should be stored together.

The Golden Rule of NoSQL

Design your database schema to match your application's read operations. If your frontend needs to display a User Profile along with their last 5 addresses, you should design the database so that a single query returns that exact JSON structure natively.

Data Modeling: Referencing vs. Embedding

When modeling relationships (One-to-One, One-to-Many), you must choose whether to Embed the data (store it as an array/object inside the parent document) or Reference it (store the ObjectId, similar to a Foreign Key in SQL).

1. Embedding (Denormalization)

Embedding places related data directly inside the parent document. This is exceptionally fast because retrieving the parent document automatically retrieves all related data in exactly one disk read.

// Embedding 'Addresses' directly inside the User
{
  "_id": "user123",
  "name": "Alice",
  "addresses": [
    { "type": "Home", "city": "New York" },
    { "type": "Work", "city": "Boston" }
  ]
}

When to use Embedding:

  • When the related data is almost entirely unique to the parent document (e.g., a User's distinct shipping addresses).
  • When the array of related documents is strongly bounded and small (e.g., a user will rarely have more than 5 addresses).
  • When the data is frequently read together.

2. Referencing (Normalization)

Referencing stores the related documents in a separate collection entirely, and stores their IDs inside the parent document (or stores the parent document's ID inside the children). You must execute an extra query or use $lookup/populate() to fetch them.

// Users Collection
{ "_id": "user_alice", "name": "Alice" }

// Posts Collection (Referencing the User)
{ "_id": "post1", "title": "Hello World", "authorId": "user_alice" }
{ "_id": "post2", "title": "NoSQL is cool", "authorId": "user_alice" }

When to use Referencing:

  • For Many-to-Many relationships.
  • For Unbounded Data Arrays. If you embedded "Posts" inside the User document, a prolific author with 10,000 posts would cause the User document to exceed MongoDB's strict 16MB document size limit, crashing the database!
  • When the child data is frequently queried on its own.

Embracing Data Duplication

In SQL, data duplication is considered a cardinal sin. If a user changes their name, you only want to update it in one place (the Users table). In NoSQL, data duplication is an active strategy used to optimize for Read-Heavy Workloads.

The Extended Reference Pattern

Imagine reading a list of recent Orders. For the UI, you need the Order Total and the Customer's Name and Shipping City.

If you use strict referencing, you must execute a secondary lookup for every single Order just to get the customer's name.

If you use the Extended Reference Pattern, you deliberately duplicate the frequently accessed data (Name and City) directly into the Order document, while leaving heavily mutating or massive data (like the User's Password Hash or massive bio) in the User collection.

// The Order Document natively duplicates data!
{
  "_id": "order555",
  "total": 99.99,
  "customer": {
    "userId": "user_alice",  // The true reference ID
    "name": "Alice Smith",   // Duplicated data for fast reads!
    "city": "New York"       // Duplicated data!
  }
}

The Trade-off: If Alice changes her name, the application must run a background update query to change her name on all 5,000 of her historical orders. However, because Profile updates occur perhaps once a year, and Order queries occur thousands of times a day, optimizing the massive read volume via duplication is mathematically the correct decision in NoSQL architecture.
