Branching & Conflict Resolution

Git allows multiple developers to work on the exact same codebase simultaneously. This is achieved through Branching. A branch is essentially an independent timeline of your project. You can break things on your branch, and the main production branch remains completely unaffected.

Feature Branching Workflow

In modern software teams, no one commits directly to the main branch. Instead, for every new task, a developer creates a Feature Branch.

# 1. Start from main
git checkout main
git pull

# 2. Create and switch to a new feature branch
git checkout -b feature/login-page

# 3. Make your code changes...
# 4. Stage and commit your changes securely isolated in your branch
git add .
git commit -m "feat: implement OAuth login"

# 5. Push your branch to the central GitHub server
git push origin feature/login-page

Once pushed, you open a Pull Request (PR) on GitHub. Your teammates review your code, and if approved, it is mathematically Merged into main.

Types of Merges

When you merge feature/login-page into main, Git calculates how to combine the two timelines.

1. Fast-Forward Merge

If no one else has touched main since you created your branch, Git simply moves the main pointer forward to point to your new commits. The history remains a perfectly straight, linear line.

2. The Merge Commit (Recursive)

If another developer merged their branch into main while you were still working on yours, the timelines have diverged. Git must mathematically stitch them back together by creating a brand new, special commit called a Merge Commit. This commit has two parents and permanently ties the two divergent branches back into one.

Resolving Merge Conflicts

Most of the time, Git's auto-merging algorithm perfectly combines code. If Developer A edits Line 5, and Developer B edits Line 50, Git seamlessly merges them.

However, a Merge Conflict occurs when two developers modify the exact same line of the exact same file.

# When merging your branch into main...
git merge feature/login-page

# Output:
# Auto-merging src/App.js
# CONFLICT (content): Merge conflict in src/App.js
# Automatic merge failed; fix conflicts and then commit the result.

Reading Conflict Markers

If you open src/App.js, Git has injected special markers indicating exactly where the collision happened:

<<<<<<< HEAD (Current branch: main)
<button>Sign In to Account</button>
=======
<button>Login via Google</button>
>>>>>>> feature/login-page (Incoming branch)

To resolve it, you must manually edit the file to decide which version to keep (or combine them). Once the file looks exactly how you want it, you delete the `<<<`, `===`, and `>>>` symbol lines.

# The resolved code:
<button>Sign In via Google</button>

Finally, you tell Git that the conflict is mathematically resolved by staging the file and committing it:

git add src/App.js
git commit -m "Merge branch feature/login-page and resolve App.js conflict"

Best Practices to Avoid Conflicts

  • Keep your feature branches small and short-lived. Work lasting 3 weeks on a branch will inevitably conflict massively.
  • Run git pull origin main frequently to pull your teammates' changes down into your local environment early.
