Rebasing & History Rewrite

While git merge is the standard way to combine timelines, it often leaves the Git history intensely cluttered. If 5 developers merge 5 branches rapidly, the history graph looks like a tangled spiderweb of crossover lines and noisy "Merge branch main into feature" commits. Advanced teams demand a perfectly clean, linear history. The mathematical solution to this is Rebasing.

What is Git Rebase?

Imagine your commit history is a stack of blocks. You created a branch from the main branch 3 days ago. Since then, your teammates added 5 new blocks to main. Your timelines have diverged.

If you run git rebase main while on your feature branch, Git performs a mathematical magic trick:

  1. It temporarily "unplugs" your feature branch commits and puts them in memory.
  2. It fast-forwards your feature branch to sync perfectly with the absolute tip of the new, updated main branch.
  3. It physically recalculates and re-applies your commits one by one on top of the new timeline.
# 1. Ensure you have the latest code
git checkout main
git pull

# 2. Switch to your feature branch
git checkout feature/new-button

# 3. Mathematically rewrite history to build ON TOP of the latest main
git rebase main

Because you re-based your work on top of the new updates, when you finally merge your feature branch back into `main`, it will be a perfectly clean Fast-Forward merge. No clunky Merge Commits exist in the history.

Interactive Rebase & Squashing (git rebase -i)

As you code locally, you might write horribly messy commits like: "fixed typo", "oops fixed again", "WIP", "finally works". Junior developers push these to GitHub, permanently polluting the project history.

An Interactive Rebase allows you to pause history and mathematically compress 10 messy commits into exactly 1 pristine, beautifully documented commit before pushing.

# Begin an interactive rebase for the last 4 commits
git rebase -i HEAD~4

This opens a text file in your terminal:

pick 2fcfb83 add login form
pick 8a931bb fix css typo
pick 4c58cdq oops fix form state
pick 9b8c7f1 ready to test

# Change 'pick' to 'squash' (or 's') for the commits you want to merge UPWARD into the top commit:
pick 2fcfb83 add login form
s 8a931bb fix css typo
s 4c58cdq oops fix form state
s 9b8c7f1 ready to test

When you save, Git compresses all 4 blocks of code into a single, cohesive timeline block. You provide one clean message: feat: implement login architecture. The history is now flawless.

The Golden Rule of Rebasing (Force Pushing)

Because `rebase` mathematically destroys old commits and generates completely new ones (with new SHA hashes), you must NEVER rebase a branch that is publicly shared with other developers.

If you rebase your branch locally and attempt to push it to your existing branch on GitHub, GitHub will reject it entirely because the timelines no longer match. To bypass this, you must explicitly Force Push:

git push origin feature/new-button --force

The Force Push aggressively overwrites the GitHub server's timeline with your local rewritten timeline. If someone else was working on that branch, you just permanently deleted their local environment synchronization. Only force-push branches that belong exclusively to you.
