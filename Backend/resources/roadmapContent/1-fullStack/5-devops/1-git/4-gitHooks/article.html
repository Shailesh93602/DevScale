Git Hooks & Husky

In a team environment, maintaining code quality is critical. If a developer accidentally commits code with syntax errors, failing tests, or improperly formatted files (like missing semicolons or messy indentation), it completely breaks the CI/CD pipeline or annoys code reviewers.

You cannot simply ask developers to remember to run npm run lint and npm test before every single commit. They will forget. The mathematical solution is automation through Git Hooks.

What are Git Hooks?

A Git Hook is a script that Git executes automatically before or after specific actions occur in the repository. They are stored natively in the hidden .git/hooks/ directory.

The two most common hooks in web development are:

  • pre-commit: Runs the exact millisecond you execute `git commit`. If this script fails (returns a non-zero exit code), Git violently rejects the commit.
  • pre-push: Runs right before you push to GitHub, ensuring no completely broken code is uploaded to the central server.

The Problem with Native Git Hooks

By default, files inside the .git folder are not shared when you push code to GitHub. If you write a brilliant `pre-commit` script on your laptop, your teammates will never get it when they pull the project. To enforce rules globally across the entire team, Node.js developers use a library called Husky.

Setting up Husky

Husky forces the configuration to live inside the project codebase (which IS tracked by Git) and automatically wires up the local Git hooks for anyone who runs `npm install`.

# 1. Install Husky as a dev dependency
npm install husky --save-dev

# 2. Initialize Husky natively (Creates a .husky folder)
npx husky install

# 3. Add a script to package.json so teammates auto-install hooks!
npm pkg set scripts.prepare="husky install"

# 4. Create your first automated Hook! (Create a pre-commit file)
npx husky add .husky/pre-commit "npm test"

Now, if any developer on the team tries to commit code while a test is currently failing, Husky intercepts the action and blocks the commit entirely. "Fix your tests before you commit!"

Optimizing with Lint-Staged

If you have a massive enterprise app, running npm test and npm run lint against 5,000 files during every single commit could take 10 minutes. Developers will hate this workflow.

To solve this, you use lint-staged. It ensures that the formatting scripts ONLY execute against the specific 2 or 3 files that were actually modified and staged in this exact commit.

// 1. Install lint-staged
npm install lint-staged --save-dev

// 2. Add configuration to package.json
"lint-staged": {
  "*.js": "eslint --cache --fix", // Automatically fix JS files
  "*.{js,css,md}": "prettier --write" // Auto-format everything else!
}

Finally, you modify your Husky pre-commit file to execute `npx lint-staged` instead of running everything globally.

#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

# Only process the exact files that are being committed!
npx lint-staged

The result: Every commit is now perfectly formatted to team standards instantly and flawlessly, without adding 5 minutes of wait time to the developer's workflow.
