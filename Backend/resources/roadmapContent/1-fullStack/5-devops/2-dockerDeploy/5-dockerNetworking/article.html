Networking & Security

By default, Docker containers are securely isolated from the outside world. If you run a Node.js server inside a container on port 8080, and try to visit http://localhost:8080 on your Mac, your browser will say "Site cannot be reached."

Port Mapping (Host vs Container)

To access the Node server from the outside physical world, you must specify a Port Mapping when executing the container:

docker run -p 3000:8080 my-node-app

This syntax is defined as [HOST_PORT]:[CONTAINER_PORT]. Docker listens to Port 3000 on your Host Machine (your Mac). It takes any incoming HTTP request and funnels it through the firewall into the container's isolated internal Port 8080.

Security Golden Rule: Never publicly expose your internal Database Port! If you map -p 5432:5432 for PostgreSQL on a production AWS server, hackers will attempt to brute-force your root password immediately. You should only expose your API server to the public internet.

Bridge Networks

If you don't expose your Database to the outside world, how does your Node API server securely connect to it?

When you execute docker compose up locally, Docker constructs a private Virtual Internet called a Bridge Network.

  • Your Database container is placed directly on this private local network.
  • Your API container is also placed directly on this exact same private network.

Because both containers share this isolated Bridge Network, they can magically ping each other using their exact Service Names (e.g., postgresql://admin:pass@database:5432/db). They do not need to use the public internet. This creates a highly secure, closed ecosystem where the database is completely invisible to the outside world, but fully accessible to your API.

Injecting Environment Variables

You must never hardcode database passwords or API keys directly inside your Dockerfile or your source code. If you do, that password becomes permanently baked into the final Image, which means anyone who downloads your image can extract your passwords.

Instead, your application must rely on Environment Variables dynamically injected at runtime.

You define an .env file on your host machine:

STRIPE_SECRET_KEY=sk_live_12345
DATABASE_URL=postgresql://admin:hiddenpass@database:5432/main

And you instruct Docker Compose to pass this file explicitly into the running container:

services:
  api-server:
    image: my-node-app
    env_file:
      - .env

Now, your Node application can freely access process.env.STRIPE_SECRET_KEY, but the actual secret key is never saved inside the immutable Docker Image file.
