Serverless Architecture

If you deploy your backend to a single AWS EC2 server, you face a major problem: traffic spikes. If your server can handle 100 concurrent users, what happens when a viral marketing campaign sends 10,000 users to your site instantly? The server completely crashes. You could spin up 10 more servers, but then you are paying thousands of dollars a month for idle machines when traffic dies down.

The modern solution to this problem is Serverless Architecture.

AWS Lambda Functions

Serverless does not mean there are no servers. It means you don't manage them, and you don't pay for idle time. In AWS, the core of this architecture is Lambda.

Instead of deploying a massive, monolithic Node.js Express server that runs perpetually 24/7, you deploy tiny, individual Javascript functions into the cloud.

// This code only runs when a user hits a specific API endpoint!
exports.handler = async (event) => {
  const userId = event.queryStringParameters.id;
  const data = await database.getUser(userId);

  return {
    statusCode: 200,
    body: JSON.stringify(data),
  };
};

When user #1 hits your API, AWS instantly boots up a micro-container running your function. If 10,000 users hit the API at the exact same millisecond, AWS instantly boots up 10,000 independent micro-containers simultaneously.

You have essentially achieved infinite horizontal scaling without configuring a single load balancer.

The Two Major Trade-offs

1. The Cold Start Problem

If nobody has visited your API in the last hour, AWS powers down your function's containers to save money. When a new user clicks a link, AWS must physically boot up a brand new micro-container, install your code, and start Node. This process can take over 1.5 seconds.

This delay is known as a Cold Start. After the function is warm, subsequent requests execute in less than 50 milliseconds. To mitigate this on Enterprise applications, you can pay for "Provisioned Concurrency" to ensure a pool of functions is kept constantly warm.

2. Connection Exhaustion

If you use a traditional relational database (like PostgreSQL) inside a Serverless architecture, you will instantly crash it.

PostgreSQL is designed to handle roughly 100 persistent, long-lived TCP connections. But Serverless architecture creates a massive spike of thousands of tiny, instantaneous connections because each Lambda container must open its own dedicated line to the database.

To safely use Serverless alongside SQL, you must route all traffic through a massively scalable Connection Pooler (like PgBouncer or Prisma Accelerate), which multiplexes 10,000 HTTP lambda requests down into 50 safe, secure, persistent TCP database connections.

The Serverless Framework

Manually uploading `.zip` files of Javascript code to the AWS console is tedious and error-prone. The industry standard tool for defining infrastructure is the Serverless Framework.

You write a serverless.yml file locally on your laptop:

service: my-backend-api
provider: aws

functions:
  getUserProfile:
    handler: src/handlers/getUser.handler
    events:
      - httpApi:
          path: /users/{id}
          method: get

When you run serverless deploy, it automatically zips your code, provisions an AWS API Gateway router, and deploys every single Lambda function perfectly into the cloud in minutes.
