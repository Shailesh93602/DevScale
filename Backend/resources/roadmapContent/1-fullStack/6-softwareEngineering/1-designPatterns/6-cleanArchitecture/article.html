Clean Architecture & DDD

When you build a small side-project, sticking all your database queries directly inside your Express route handlers works fine. But when building enterprise applications with 100,000+ lines of code, this tightly coupled monolithic design becomes a nightmare to maintain, test, and scale.

The Core Concept: Separation of Concerns

Clean Architecture mandates that your code is strictly divided into concentric boundary rings. The overarching rule is The Dependency Rule: Dependencies can only point inwards. Outer layers (like UI and Databases) can know about inner layers, but inner layers must know absolutely nothing about outer layers.

1. Domain Entities (The Core)

The deepest layer contains the core business rules. These are pure functions and standard classes representing your exact business models. They should contain absolutely zero dependencies on external frameworks (like React, Express, or Prisma).

// Pure Domain Entity. No imports. No database connection.
class User {
  constructor(public id: string, public email: string) {}

  isValidEmail(): boolean {
    return this.email.includes("@");
  }
}

2. Use Cases (Application Logic)

The next layer outwards defines the specific Use Cases of the system (e.g. `RegisterUser`, `ProcessPayment`). These classes orchestrate the flow of data exclusively to and from the Domain Entities.

3. Interfaces & Controllers

This layer adapts the core use cases to whatever external delivery mechanism you are using. If you are building a REST API using Express, this layer grabs the HTTP request, parses the JSON body, and hands clean data deeply inwards into the Use Case layer.

4. Infrastructure (The Edge)

The outermost ring contains all external dependencies. The Database (PostgreSQL/Prisma), the Web Framework (Next.js/Express), and third-party APIs (Stripe/SendGrid). According to Domain-Driven Design, the Database is simply an implementation detail. You should theoretically be able to swap out MongoDB for PostgreSQL without touching your core Domain logic.

Domain-Driven Design (DDD)

Clean Architecture is usually paired with Domain-Driven Design (DDD). The core principle of DDD is establishing a Ubiquitous Language. Every single developer, product manager, and database architect must use the exact same terminology.

If the marketing team calls an account a "Subscriber," the database table shouldn't be called `users`, and the frontend component shouldn't be called `<ClientProfile>`. Everything mathematically aligns globally around the exact Domain Language.
