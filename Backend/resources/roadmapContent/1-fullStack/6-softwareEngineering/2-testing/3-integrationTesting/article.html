Integration Testing

Unit Tests focus on an isolated function or a single component. However, even if your isolated calculateTax() function works perfectly, your application might still crash when multiple systems try to collaborate. Integration Testing bridges this gap by verifying that different pieces of your application (Routing, Controllers, Middleware, and Databases) interact flawlessly.

Testing APIs with Supertest

In the Node.js ecosystem, the industry standard library for testing HTTP endpoints is Supertest. It allows you to fire real HTTP requests against your Express app without actually binding it to a live network port.

import request from "supertest";
import app from "../src/app"; // Your Express app instance

describe("POST /api/users", () => {
  it("should create a new user and return 201 status", async () => {
    const response = await request(app)
      .post("/api/users")
      .send({ email: "test@example.com", password: "Password123" })
      .expect(201); // Supertest asserts the status code natively

    // We can then assert against the returned JSON payload
    expect(response.body).toHaveProperty("id");
    expect(response.body.email).toBe("test@example.com");
  });
});

Notice that this single test verifies the Router correctly matched the URL, the Body Parser middleware correctly extracted the JSON, the Controller processed the logic, the Database created the record, and the Server sent the exact response back.

Managing the Test Database

Integration tests that mutate a database run the massive risk of polluting your local development database. If your test creates a user named "test@example.com", and you run the test twice, the second test might fail due to a "Unique Constraint" error.

To safely run integration tests, you must follow the Setup and Teardown lifecycle flawlessly:

  1. Before All: Connect specifically to an isolated Test Database (e.g., `test_db` instead of `dev_db`).
  2. Before Each: Wipe the database tables completely clean to ensure a pristine state for the next test.
  3. Run Test: Execute the Supertest HTTP request.
  4. After All: Safely disconnect the Prisma Client or Mongoose connection to prevent open-handle warnings.
// Example Jest Teardown Lifecycle
beforeEach(async () => {
  // Wipe the User table clean before every single test
  await prisma.user.deleteMany(); 
});
