Test-Driven Development (TDD)

In traditional software engineering, developers write the logic of an application first, test it manually in their browser, and then write a few Unit Tests afterwards.

Test-Driven Development (TDD) structurally reverses this exact process: You write your test before you write a single line of actual application code.

The Red-Green-Refactor Cycle

The core methodology of TDD is built around a strict three-step lifecycle.

Step 1: The Red Phase

You write exactly one failing test for a tiny slice of functionality. The test defines how you want your future Javascript API to behave.

// mathUtils.test.ts
import { addNumbers } from "./mathUtils";

test("addNumbers should add two positive integers correctly", () => {
  expect(addNumbers(5, 7)).toBe(12);
});

When you execute this, Jest fails immediately because `addNumbers` does not even exist yet. The test is "Red".

Step 2: The Green Phase

You write the absolute bare minimum amount of production code required to make that specific test pass. You do not worry about performance, design patterns, or future edge cases.

// mathUtils.ts
export function addNumbers(a: number, b: number): number {
  return a + b; 
}

The test now outputs a "Green" success message.

Step 3: The Refactor Phase

Now that the test is passing, the code works. But the code might be ugly or inefficient. In the Refactor phase, you alter the code to make it clean, apply design patterns, and remove duplication.

Because you already have a Green test, you can refactor confidently. If you accidentally break the logic while refactoring, the test will instantly turn Red and alert you.

The Architectural Benefits of TDD

TDD forces you to think about the "Interface" of your code before you think about the "Implementation". Since you must write the test first, you naturally design your functions to be modular, decoupled, and easy to test. It prevents you from building massive, untestable spaghetti-code functions.
