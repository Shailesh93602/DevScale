Core Web Vitals & Web Performance

Core Web Vitals are Google's standardized metrics for measuring real-user experience on the web. They directly impact SEO rankings and, more importantly, user satisfaction. A page that loads in 1 second converts 3× better than one that loads in 5 seconds. This article covers how to measure, understand, and optimize all three Core Web Vitals.

LCP — Largest Contentful Paint

LCP measures when the largest visible element (hero image, H1, video thumbnail) finishes rendering. It's the best proxy for "when does the page feel loaded?"

Common LCP issues and fixes:

  • Slow server response (TTFB): Use a CDN (Cloudflare, Vercel Edge), add server-side caching, or upgrade your hosting tier.
  • Render-blocking resources: Move non-critical CSS/JS to load after the initial paint. Defer non-critical scripts with defer or async.
  • Unoptimized images: Use next-gen formats (WebP, AVIF), correct srcset for responsive images, and always set explicit width and height attributes.
<!-- Bad: LCP image blocks render -->
<img src="hero.jpg">

<!-- Good: Preloaded + WebP + explicit dimensions -->
<link rel="preload" as="image" href="hero.webp" fetchpriority="high">
<img
  src="hero.webp"
  width="1200"
  height="600"
  alt="Hero image"
  fetchpriority="high"
  decoding="async"
  srcset="hero-400w.webp 400w, hero-800w.webp 800w, hero-1200w.webp 1200w"
  sizes="(max-width: 768px) 100vw, 1200px"
>

INP — Interaction to Next Paint

INP measures the delay from a user interaction (click, tap, keypress) to the next frame being painted. High INP means the UI feels "janky" or unresponsive — usually caused by long JavaScript tasks blocking the main thread.

// Bad: Long synchronous task blocks the main thread for 300ms
function processLargeArray(arr) {
  return arr.map(item => expensiveTransform(item)); // 300ms = very bad INP
}

// Good: Break work into chunks with scheduler.yield()
async function processLargeArray(arr) {
  const results = [];
  for (let i = 0; i < arr.length; i++) {
    results.push(expensiveTransform(arr[i]));
    // Yield to the browser every 50 items to allow rendering/input handling
    if (i % 50 === 0) await scheduler.yield();
  }
  return results;
}

// Alternative: Use a Web Worker for CPU-intensive work off the main thread
const worker = new Worker('./processor.worker.js');
worker.postMessage({ items: largeArray });
worker.onmessage = (e) => displayResults(e.data);

CLS — Cumulative Layout Shift

CLS measures unexpected layout shifts — when elements jump around the page after load, causing mis-clicks. The most common causes:

  • Images and videos without explicit dimensions
  • Ads, embeds, or iframes with dynamically injected size
  • Web fonts causing text to reflow (FOIT/FOUT)
<!-- Bad: Image with no dimensions shifts layout when loaded -->
<img src="product.jpg" alt="Product">

<!-- Good: Reserve space with aspect-ratio CSS -->
<style>
  .product-img {
    width: 100%;
    aspect-ratio: 16 / 9;  /* Browser reserves exact space before image loads */
    object-fit: cover;
  }
</style>
<img class="product-img" src="product.jpg" width="800" height="450" alt="Product">

<!-- Good: Prevent font swap with font-display: optional -->
<style>
  @font-face {
    font-family: 'Inter';
    src: url('/fonts/inter.woff2') format('woff2');
    font-display: swap;  /* Use system font until Inter loads — minimal CLS */
  }
</style>

Measuring CWV

// Measure CWV in real users' browsers with the web-vitals library
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendToAnalytics({ name, value, rating }) {
  // Send to your analytics endpoint
  fetch('/api/vitals', {
    method: 'POST',
    body: JSON.stringify({ metric: name, value, rating, url: location.href }),
  });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);

For lab testing (during development): use PageSpeed Insights, Lighthouse in Chrome DevTools, or npx lighthouse https://yoursite.com --output html.

Practical Performance Checklist

  • ✅ All images have explicit width and height attributes
  • ✅ Hero/LCP image is preloaded with <link rel="preload" fetchpriority="high">
  • ✅ Images are WebP/AVIF with correct srcset for responsive sizes
  • ✅ Web fonts use font-display: swap
  • ✅ JavaScript is code-split and loaded with defer
  • ✅ Critical CSS is inlined; non-critical CSS is loaded asynchronously
  • ✅ Static assets are served from a CDN with long-lived cache headers
  • ✅ Server response time (TTFB) is under 800ms
