Web Bundling & Build Optimization

Modern web applications import hundreds of JavaScript modules, TypeScript files, CSS, and images. Bundlers combine and optimize all these assets into production-ready files that browsers can load efficiently. Understanding bundling is critical for shipping fast applications because the bundle size and loading strategy directly control how fast users see your application.

Why Bundle?

  • HTTP overhead: Loading 500 separate ES module files creates 500 HTTP requests, each with its own round-trip latency. Bundling reduces this to 1–5 requests.
  • Dead code elimination: Bundlers remove code you import but never call.
  • Ahead-of-time compilation: TypeScript is compiled, JSX is transformed, and CSS is processed at build time — not at runtime.
  • Minification: Variable names are shortened, whitespace is removed, reducing file size by 60–80%.

Vite — The Modern Build Tool

Vite (French for "quick") is the dominant modern build tool. It uses native ES modules during development (no bundling — instant HMR) and Rollup for production builds:

# Create a new Vite + React + TypeScript project
npm create vite@latest my-app -- --template react-ts
cd my-app && npm install && npm run dev

# vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  build: {
    target: 'es2020',         // Browser compatibility target
    minify: 'esbuild',        // Fast minification
    sourcemap: false,         // Disable in production (enable for debugging)
    rollupOptions: {
      output: {
        // Split vendor libraries into separate chunks
        manualChunks: {
          'vendor-react': ['react', 'react-dom'],
          'vendor-router': ['react-router-dom'],
          'vendor-ui': ['@radix-ui/react-dialog', '@radix-ui/react-dropdown-menu'],
        },
      },
    },
  },
});

Tree Shaking

Tree shaking eliminates exported functions that are never imported — only possible with ES Module import/export syntax (not CommonJS require):

// 🚫 Bad: Imports the entire lodash library (~540KB)!
import _ from 'lodash';
const result = _.debounce(fn, 300);

// ✅ Good: Imports only the debounce function (~2KB)
import debounce from 'lodash-es/debounce';
const result = debounce(fn, 300);

// ✅ Even better: Use a modern, tree-shakeable alternative
import { debounce } from 'perfect-debounce';  // 100 bytes

// Verify your bundle contains no dead code:
npx vite-bundle-visualizer   # Opens interactive treemap of bundle contents

Code Splitting

Rather than sending a 2MB JavaScript bundle for the homepage, code splitting lets the browser download only the code needed for the current page — loading other routes on demand:

// React Router v6 with lazy loading
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// These components are in separate chunks — loaded only when navigated to
const Dashboard = lazy(() => import('./pages/Dashboard'));       // ~80KB
const Analytics = lazy(() => import('./pages/Analytics'));      // ~200KB (Chart.js etc.)
const AdminPanel = lazy(() => import('./pages/AdminPanel'));    // ~120KB

function App() {
  return (
    <Suspense fallback={<div className="loading-spinner" />}>
      <Routes>
        <Route path="/" element={<Home />} />                   {/* Always loaded */}
        <Route path="/dashboard" element={<Dashboard />} />   {/* Loaded on demand */}
        <Route path="/analytics" element={<Analytics />} />   {/* Loads only if user goes here */}
        <Route path="/admin" element={<AdminPanel />} />       {/* Loads only for admins */}
      </Routes>
    </Suspense>
  );
}

Asset Optimization

# vite.config.ts — Image and asset optimization
import { defineConfig } from 'vite';

export default defineConfig({
  build: {
    assetsInlineLimit: 4096,  // Inline assets smaller than 4KB as base64 (removes HTTP request)
    rollupOptions: {
      output: {
        assetFileNames: 'assets/[name]-[hash][extname]',  // Content hash in filename
        chunkFileNames: 'chunks/[name]-[hash].js',        // Hash enables long-lived caching
        entryFileNames: '[name]-[hash].js',
      }
    }
  },
  // CSS is automatically code-split per page
  css: { devSourcemap: true }
});

Caching Strategy for Bundles

Content-hashed filenames enable infinitely long cache TTLs — the file content determines the hash, so browsers always fetch the correct version:

# Set these headers on your static file server (Nginx, Vercel, AWS S3+CloudFront):
# For hashed assets (main-abc123.js, vendor-def456.js):
Cache-Control: public, max-age=31536000, immutable  # 1 year

# For index.html (un-hashed — must always be fresh):
Cache-Control: no-cache

# This way: users download new JS only when it actually changes.
# If you only change Dashboard.tsx, only the dashboard chunk re-downloads.
# Vendor libraries (React, etc.) stay cached unless their version changes.
