Scalability & High Availability Design

Scalability is the ability of a system to handle growing load without requiring complete redesign. High availability (HA) ensures the system remains operational even when individual components fail. These two properties are the foundation of production-grade distributed systems ŌĆö and achieving them requires deliberate architectural decisions at every layer.

Load Balancing Algorithms

Load balancers distribute incoming traffic across a pool of servers. The choice of algorithm depends on your workload:

  • Round Robin: Requests distributed sequentially 1ŌåÆ2ŌåÆ3ŌåÆ1ŌåÆ2ŌåÆ3. Simple, works when all servers have equal capacity and requests take similar time.
  • Least Connections: Routes each new request to the server with fewest active connections. Best for workloads with highly variable request durations (e.g., video transcoding).
  • IP Hash: Same client IP always routes to the same server. Enables server-side session affinity without shared session storage.
  • Weighted Round Robin: Assign weights (e.g., 3├Ś more traffic to servers with higher CPU). Used during gradual rollouts (canary deployments).

The CAP Theorem

A distributed system can guarantee at most two of these three properties simultaneously:

  • C ŌĆö Consistency: Every read receives the most recent write
  • A ŌĆö Availability: Every request receives a response (not necessarily latest data)
  • P ŌĆö Partition Tolerance: System works despite dropped network messages between nodes

In practice, network partitions (P) are inevitable, so your choice is between CP and AP:

  • CP systems (e.g., PostgreSQL, MongoDB with majority writes): Prefer consistency. Will reject writes during a partition to avoid divergence.
  • AP systems (e.g., DynamoDB, Cassandra, DNS): Prefer availability. Will serve possibly stale reads during a partition and reconcile later.

Database Scaling Strategies

Read Replicas

The database leader handles all writes; read replicas handle all reads. This offloads 70ŌĆō90% of traffic in read-heavy workloads:

// Direct writes to primary (leader), reads to replica pool
async function getUser(id: string) {
  return readDb.query('SELECT * FROM users WHERE id = $1', [id]);
  // readDb connection pool points to replica(s)
}

async function createUser(data: UserInput) {
  return writeDb.query('INSERT INTO users ...', [...]);
  // writeDb connection pool points to primary only
}

Database Sharding

When a single database server cannot hold all the data, you shard by splitting the data across multiple independent databases. A common strategy is hash-based sharding:

// Route each user to a shard based on their user_id
function getShardId(userId: string, totalShards: number): number {
  const hash = crc32(userId);  // Deterministic hash function
  return hash % totalShards;
}

function getShardConnection(userId: string) {
  const shardId = getShardId(userId, 4);  // 4 shard databases
  return shardConnections[shardId];
}

Stateless Architecture

Horizontal scaling requires stateless application servers. If server #1 stores user session data in memory, routing the next request to server #2 will require a new login. Move all state out of the application process:

  • Sessions: Store in Redis (shared across all servers) or use JWT tokens (self-contained, no server storage needed)
  • File uploads: Store in object storage (S3, Cloudflare R2) ŌĆö never on the local filesystem
  • Scheduled jobs: Use an external queue (BullMQ, SQS) ŌĆö not in-process setInterval

High Availability Patterns

Circuit Breaker

Prevents a slow downstream service from cascading failures through your entire system:

import CircuitBreaker from 'opossum';

const breaker = new CircuitBreaker(callPaymentService, {
  timeout: 3000,           // If call takes > 3s, count as failure
  errorThresholdPercentage: 50,  // Open circuit if >50% of requests fail
  resetTimeout: 30000,     // Try half-open after 30s
});

breaker.fallback(() => ({ status: 'queued', message: 'Payment will be retried' }));
breaker.on('open', () => alertingService.notify('Payment circuit OPEN ŌĆö downstream failures!'));

const result = await breaker.fire({ amount: 999, userId: 'usr_123' });
// If payment service is down: immediately returns fallback, no timeout waiting

Health Checks & Auto-Healing

// Express health check endpoint (checked by load balancer every 30s)
app.get('/health', async (req, res) => {
  const checks = {
    database: await checkDbConnection(),
    redis: await checkRedisConnection(),
    externalApi: await checkExternalApi(),
  };
  const healthy = Object.values(checks).every(Boolean);
  res.status(healthy ? 200 : 503).json({ status: healthy ? 'ok' : 'degraded', checks });
});
// If this returns 503, the load balancer removes the server from rotation
// Kubernetes/ECS will restart the container automatically

Best Practices

  • Design for failure ŌĆö assume any server will crash at any moment. Build redundancy into every layer: multiple API servers, multi-AZ databases, CDN with origin failover.
  • Implement graceful shutdown ŌĆö drain in-flight requests before stopping (SIGTERM handler with 30s drain window). This eliminates request failures during deployments.
  • Use exponential backoff with jitter when retrying failed requests ŌĆö flat retry intervals cause thundering herd problems that worsen outages.
