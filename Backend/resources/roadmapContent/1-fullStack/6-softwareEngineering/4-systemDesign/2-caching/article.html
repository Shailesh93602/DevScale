Caching Strategies with Redis

Caching is one of the highest-leverage optimizations in software engineering. A properly cached database query that returns in 0.2ms instead of 150ms is a 750× improvement. But caching introduces complexity: you now have two sources of truth that can diverge. This article covers the complete caching landscape — from browser caches to distributed Redis clusters.

Redis — The Standard Application Cache

Redis is an in-memory data store with sub-millisecond latency, supporting strings, hashes, lists, sets, sorted sets, and more. It's the industry standard for application-level caching:

npm install ioredis

// lib/redis.ts
import Redis from 'ioredis';

const redis = new Redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: 6379,
  password: process.env.REDIS_PASSWORD,
  maxRetriesPerRequest: 3,
  lazyConnect: true,
});

export default redis;

Cache-Aside Pattern (Most Common)

The application manages the cache explicitly: look in cache → if miss, query DB → store result in cache → return to client:

import redis from './lib/redis';
import { db } from './lib/db';

async function getUserById(userId: string) {
  const cacheKey = `user:${userId}`;
  const TTL = 300;  // 5 minutes

  // 1. Try cache first
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);  // Cache HIT — return immediately

  // 2. Cache MISS — fetch from database
  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  if (!user) return null;

  // 3. Store in cache with TTL
  await redis.setex(cacheKey, TTL, JSON.stringify(user));

  return user;
}

// Invalidate when user is updated
async function updateUser(userId: string, data: Partial) {
  const user = await db.query('UPDATE users SET ... WHERE id = $1 RETURNING *', [userId, ...]);
  await redis.del(`user:${userId}`);  // Bust the cache
  return user;
}

Cache Invalidation Strategies

// Strategy 1: TTL-only — let entries expire automatically
await redis.setex('popular_products', 3600, JSON.stringify(products));  // Expires in 1 hour

// Strategy 2: Write-through — update cache immediately on every write
async function updateProductPrice(productId: string, newPrice: number) {
  await db.query('UPDATE products SET price = $1 WHERE id = $2', [newPrice, productId]);
  const updated = await db.query('SELECT * FROM products WHERE id = $1', [productId]);
  await redis.setex(`product:${productId}`, 3600, JSON.stringify(updated));  // Update cache too
}

// Strategy 3: Cache tags — invalidate groups of related entries
// When a user updates their profile, bust all pages that mention them
async function invalidateUserRelatedCaches(userId: string) {
  const keys = await redis.keys(`*user:${userId}*`);  // ⚠️ Don't use keys() in prod! Use SCAN
  if (keys.length > 0) await redis.del(...keys);
}

// Production-safe key pattern scanning:
async function invalidateByPattern(pattern: string) {
  const pipeline = redis.pipeline();
  let cursor = '0';
  do {
    const [newCursor, keys] = await redis.scan(cursor, 'MATCH', pattern, 'COUNT', 100);
    cursor = newCursor;
    keys.forEach(key => pipeline.del(key));
  } while (cursor !== '0');
  await pipeline.exec();
}

HTTP Caching — Browser & CDN

// Express.js response with caching headers
app.get('/api/products', async (req, res) => {
  const products = await getProducts();

  res.set({
    'Cache-Control': 'public, max-age=3600, s-maxage=86400, stale-while-revalidate=600',
    //     ↑ browser 1hr   ↑ CDN caches 24hrs        ↑ serve stale for 10min while refreshing
    'ETag': generateEtag(products),    // Browser sends If-None-Match on next request
    'Last-Modified': new Date().toUTCString(),
  });

  // 304 Not Modified — if nothing changed, send no body (saves bandwidth)
  const clientEtag = req.headers['if-none-match'];
  if (clientEtag === res.getHeader('ETag')) {
    return res.status(304).send();
  }

  res.json(products);
});

// Private data (user-specific) — never cache on CDN!
app.get('/api/me', (req, res) => {
  res.set('Cache-Control', 'private, max-age=0, no-store');
  res.json(req.user);
});

What NOT to Cache

  • Rapidly changing data with strong consistency requirements (bank balances, inventory counts)
  • Personally identifiable information (PII) in shared caches — cache-key collisions can leak user data
  • Unique per-user data in public/CDN caches — use 'Cache-Control: private'
  • Very cheap queries — if a DB query takes 1ms, caching it in Redis (0.5ms overhead) provides minimal benefit and adds cache invalidation complexity

Best Practices

  • Monitor your cache hit rate (target >90% for frequently accessed data). A 60% hit rate means 40% of cache overhead for no benefit — revisit your TTL and key strategy.
  • Use SCAN instead of KEYS in production — KEYS * blocks Redis for the entire duration of the scan, causing latency spikes on large instances.
  • Never store full database rows in cache "just in case" — only cache what is actually read-heavy. Stale cached data is a serious bug.
  • Implement stampede protection (probabilistic early expiration or locking) for high-traffic cache keys where many concurrent requests hitting a cache miss simultaneously overload the database.
