Database Scaling: Replication & Sharding

Scaling a stateless Node.js server is easy: you simply spin up 10 identical servers behind a load balancer. However, scaling a stateful database like PostgreSQL is the hardest problem in System Design. You cannot simply spin up 10 independent databases, or else user data will instantly become fragmented.

Vertical Scaling vs Horizontal Scaling

Vertical Scaling (Scaling Up) means buying a strictly bigger, more expensive server with more CPU cores and RAM. This is easiest, but eventually, you hit a physical hardware limit where no larger server exists.

Horizontal Scaling (Scaling Out) means adding more commodity servers to a distributed network. This is theoretically infinite but requires massive architectural complexity.

Database Replication (Master-Slave)

Most applications have a 10-to-1 read-to-write ratio. You read a user profile 10 times for every 1 time you update it. Master-Slave Replication exploits this metric.

You define one primary Master Database. All `INSERT`, `UPDATE`, and `DELETE` queries must go to the Master. You then provision multiple Read-Replica (Slave) Databases. The Master continuously copies its new data to the Replicas asynchronously. All `SELECT` read queries are routed exclusively to the Replicas.

The Trade-off: Replication Lag. Because the data copy takes a few milliseconds, a user might update their profile on the Master, instantly refresh the page, and hit a Replica that hasn't received the update yet, showing old data.

Database Sharding

When your dataset becomes so massive (e.g., 50 Terabytes of User Data) that it cannot physically fit on a single Master hard drive, you must use Sharding.

Sharding partitions your database horizontally across multiple separate machines. You choose a specific "Shard Key". For example, if you shard by User ID:

  • Users with IDs 1 to 1,000,000 are stored completely on Database Server A.
  • Users with IDs 1,000,001 to 2,000,000 are stored completely on Database Server B.

The Trade-off: SQL `JOIN` operations across different shards become virtually impossible. You have to write highly complex application-level logic to stitch the data back together.

The CAP Theorem

When designing distributed systems, you must understand the CAP Theorem. It states that a distributed data store can only simultaneously provide two out of the following three guarantees:

  1. Consistency: Every read receives the most recent write or an error.
  2. Availability: Every request receives a non-error response.
  3. Partition Tolerance: The system continues to operate despite network failures dropping messages between nodes.

Since network failures (Partitions) are practically inevitable on the internet, you are forced to choose between designing an AP system (Highly Available, but might return stale data) or a CP system (Highly Consistent, but will shut down and refuse connections during a network error).
