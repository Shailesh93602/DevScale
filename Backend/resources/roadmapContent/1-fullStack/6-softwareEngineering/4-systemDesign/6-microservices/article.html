Microservices Architecture

When you start a new project, you build a Monolithic Architecture. A Monolith is a single, massive codebase where the User Service, Payment Service, and Notification Service are all compiled and deployed together as one unified Node.js application.

The Monolith vs Microservices

Monoliths are easy to develop, test, and deploy. However, when the engineering team grows to 500 developers, the Monolith becomes a bottleneck.

  • If the Payment Team introduces a memory leak, the entire Monolith crashes, taking the core User Service offline with it.
  • If the Notification Service needs to scale to 100 servers during a spike, you are forced to scale the entire Monolith to 100 servers, wasting massive amounts of money.

Microservices Architecture solves this problem by breaking the Monolith apart into small, autonomous, independently deployable services.

The API Gateway

In a Microservices architecture, the UI (like a React app) should never communicate directly with 50 separate backend microservices.

Instead, all client requests go through an API Gateway. The Gateway is a single entry point that handles authentication, rate limiting, and request routing.

  1. The React App requests `/api/payments/123`.
  2. The API Gateway intercepts the request and validates the JWT token.
  3. The Gateway forwards the secure request to the internal Payment Microservice.

The Saga Pattern (Distributed Transactions)

The hardest problem in Microservices is data consistency.

In a Monolith, if a user books a hotel and a flight, you can wrap both actions in a single SQL Transaction. If the flight booking fails, the database rolls back the hotel booking automatically.

In Microservices, the Hotel DB and the Flight DB are physically separate. You cannot use standard SQL transactions across two different databases. You must use the Saga Pattern.

A Saga is a sequence of local transactions. Each local transaction updates the database and publishes a message to trigger the next local transaction in the saga. If a local transaction fails (e.g., the Flight Service rejects the booking), the Saga automatically executes series of "Compensating Transactions" that undo the changes made by the preceding local transactions (e.g., it sends a message to the Hotel Service to cancel the room).

Event-Driven Architecture

Microservices should avoid communicating with each other synchronously via HTTP REST calls.

If the User Service calls the Email Service via HTTP, and the Email Service is down, the User Service crashes. This defeats the purpose of isolation.

Instead, Microservices use Event-Driven Communication via Apache Kafka or RabbitMQ.

  • The User Service creates a new user, saves it to its database, and publishes a UserCreatedEvent to Kafka.
  • The User Service immediately returns a 200 OK to the client.
  • Three seconds later, the independent Email Service consumes the event from Kafka and sends a welcome email.
