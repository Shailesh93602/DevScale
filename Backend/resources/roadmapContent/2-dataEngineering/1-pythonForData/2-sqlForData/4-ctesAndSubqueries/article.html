CTEs & Subqueries

When you first learn SQL, queries are relatively simple: SELECT * FROM users WHERE age > 18. However, in Data Engineering, you will quickly encounter analytical questions that require deeply complex, multi-step logic. If you try to write this logic as a single massive query, it becomes unreadable and impossible to maintain.

The Problem with Nested Subqueries

A Subquery is simply a query nested inside another query. While useful for simple tasks, deeply nesting subqueries creates what developers call "Spaghetti SQL."

-- A confusing nested subquery
SELECT u.name, u.email
FROM users u
WHERE u.id IN (
    SELECT o.user_id
    FROM orders o
    WHERE o.amount > (
        SELECT AVG(amount) FROM orders
    )
);

To understand this code, a developer must read it from the inside-out. First, find the average amount, then find the user IDs, and finally find the names. When queries become 5 or 6 levels deep, this inside-out reading becomes incredibly difficult.

Common Table Expressions (CTEs)

A CTE (Common Table Expression) utilizing the WITH clause is the modern solution to Spaghetti SQL. It allows you to define a temporary, named result set that exists only for the duration of the execution.

Instead of reading inside-out, CTEs allow you to read SQL structurally from top-to-bottom, exactly like variables in Python or Javascript.

-- The exact same logic, rewritten with CTEs for readability
WITH AverageOrder AS (
    SELECT AVG(amount) as avg_amount 
    FROM orders
),
HighSpenders AS (
    SELECT o.user_id 
    FROM orders o
    JOIN AverageOrder a ON o.amount > a.avg_amount
)

-- The final readable query
SELECT u.name, u.email
FROM users u
JOIN HighSpenders h ON u.id = h.user_id;

The DRY Principle in SQL

CTEs uniquely enforce the DRY (Don't Repeat Yourself) principle natively in SQL. If you need to use the `AverageOrder` calculation in three different places within your final `SELECT` statement, a standard Subquery forces you to literally copy and paste the logic three times.

With a CTE, you declare `AverageOrder` exactly once at the top of the file, and simply reference its name three times below. This drastically reduces code duplication and prevents massive maintenance headaches.
