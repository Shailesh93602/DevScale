XComs and Task Communication

By design, Airflow tasks are completely isolated. Task A and Task B might run on entirely different servers. Because they are isolated, you cannot simply declare a Python variable `my_data = []` in Task A and expect Task B to read it from memory. To solve this, Airflow provides a mechanism called XCom (Cross-Communication).

Pushing and Pulling XComs

When a task needs to pass a small piece of metadata to a downstream task (for example, the ID of a newly created database job, or an API Access Token), the first task "pushes" an XCom. The second task "pulls" that XCom.

from airflow.models import XCom

def push_function(**kwargs):
    # Pushing an XCom manually
    kwargs['ti'].xcom_push(key='api_token', value='12345-ABCDE')
    
    # Or simply return a value (this automatically pushes an XCom named 'return_value')
    return '12345-ABCDE'

def pull_function(**kwargs):
    # Pulling the XCom created by the upstream task
    token = kwargs['ti'].xcom_pull(task_ids='push_task', key='api_token')
    print(f"Received Token: {token}")

When `xcom_push` is called, Airflow serializes the value (often into JSON) and saves it permanently inside the Airflow Metadata Database (usually PostgreSQL or MySQL). When `xcom_pull` is called later, Airflow queries the database and retrieves the value.

The XCom Anti-Pattern: Massive DataFrames

The most common and devastating mistake junior Data Engineers make is attempting to pass actual massive datasets through XComs. Because XComs are stored directly inside the Airflow Metadata Database, pushing a 500-Megabyte Pandas DataFrame into an XCom will immediately crash the Airflow system.

# NEVER DO THIS
def extract_data_from_api(**kwargs):
    df = my_api.get_millions_of_rows()
    # Pushing 2 Gigabytes of data into the metadata database will destroy Airflow
    kwargs['ti'].xcom_push(key='massive_data', value=df)

Airflow is an Orchestrator, not an Execution Engine. It is designed to tell other systems what to do, not to move the heavy data itself.

The Correct Architecture: External Storage

If Task A downloads a massive dataset and Task B needs to transform it, you must use external storage. Task A should save the massive dataset to a cloud object store (like AWS S3 or Google Cloud Storage). Task A should then push the S3 File Path URI (a tiny string) into the XCom.

Task B pulls that tiny URI string from the XCom, and then Task B downloads the massive file directly from S3 to its own local memory for processing. This architectural pattern keeps the Airflow Metadata Database incredibly small and fast.
