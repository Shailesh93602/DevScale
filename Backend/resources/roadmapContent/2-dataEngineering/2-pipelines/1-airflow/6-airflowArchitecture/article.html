Airflow Architecture & Scaling

Understanding how Airflow physically operates under the hood is critical for deploying it in a production environment. A production Airflow cluster consists of four core components working together harmoniously.

The Four Core Components

  • The Webserver: A Flask application that provides the familiar Airflow UI. It allows data engineers to visually inspect DAGs, trigger manual runs, and review task logs.
  • The Metadata Database: The central brain of Airflow, usually a PostgreSQL or MySQL instance. It stores immediately everything: DAG schedules, task states (success, running, failed), XComs, and connection credentials.
  • The Scheduler: The heart of Airflow. A deeply complex daemon process that constantly polls the Metadata Database. When it sees that a DAG is scheduled to run at 5:00 PM, it creates "Task Instances" in the database and queues them up to be executed.
  • The Executor (and Workers): The muscle of Airflow. The Executor explicitly reads the queue created by the Scheduler and assigns those tasks to actual computing Workers to perform the heavy Python logic.

The Sequential and Local Executors

When you install Airflow locally on your laptop using SQLite, it strictly utilizes the Sequential Executor. It can only run one single task at a time. If Task A takes 5 hours, Task B cannot start, even if they are in different DAGs.

If you upgrade your laptop to PostgreSQL, you can unlock the Local Executor. This allows Airflow to spawn multiple Python processes locally, allowing parallelism on a single machine.

Scaling with the Celery Executor

If you have 1,000 DAGs containing 50,000 daily tasks, a single server running a Local Executor will burst into flames. You must scale horizontally utilizing the Celery Executor.

Celery is an asynchronous task queue architecture.

  1. The Scheduler identifies 100 tasks that need to run immediately.
  2. Instead of executing them locally, the Celery Executor pushes 100 "messages" into a Message Broker queue (usually Redis or RabbitMQ).
  3. You provision 50 separate Worker Nodes (virtual machines). These workers constantly poll the Redis queue.
  4. Whenever a worker finishes a task, it pulls the next message off the queue and begins executing it.

This distributed architecture ensures that if 1 worker crashes due to a memory leak, the other 49 workers continue processing the queue flawlessly, drastically increasing the fault tolerance of your data platform.

The Kubernetes Executor

Modern enterprises increasingly prefer the Kubernetes Executor. Instead of maintaining 50 permanent worker machines (which costs thousands of dollars even when they sit idle at night), the Kubernetes Executor dynamically spins up a brand new, isolated Docker container for every single individual task.

When the task finishes successfully, the Kubernetes container is instantly destroyed and the resources are aggressively reclaimed by the cloud provider, optimizing infrastructure costs massively.
