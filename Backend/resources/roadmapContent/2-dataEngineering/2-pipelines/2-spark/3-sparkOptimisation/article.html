Spark Optimisation

While Apache Spark is incredibly fast by default, it requires careful tuning and optimisation when processing petabytes of data to avoid Out Of Memory (OOM) errors and long execution times.

Data Partitioning

Spark splits data into chunks called partitions. How these are divided affects parallelism and network I/O.

# Check current partitions
df.rdd.getNumPartitions()

# Repartition (Increases or decreases partitions evenly, causing a full shuffle)
df = df.repartition(200)

# Coalesce (Decreases partitions without a full shuffle, much faster)
df = df.coalesce(50)

Broadcast Joins

When joining a large table with a small table, Spark defaults to a Sort Merge Join (which requires heavy shuffling). Use a Broadcast Hash Join instead to send the small table to all worker nodes.

from pyspark.sql.functions import broadcast

small_df = spark.table("dimensions.country")
large_df = spark.table("facts.sales")

# Force Broadcast Join
result = large_df.join(broadcast(small_df), "country_id")

Spark implicitly broadcasts tables under 10MB (configurable via spark.sql.autoBroadcastJoinThreshold).

Caching and Persistence

If you use the same DataFrame multiple times, cache it so Spark doesn't recompute it.

# df.cache() uses default storage level (Memory and Disk)
df.cache()

# Use persist() to specify storage levels
from pyspark import StorageLevel
df.persist(StorageLevel.MEMORY_AND_DISK_SER)

# Release memory when done
df.unpersist()

Avoiding Data Skew

Data skew happens when one partition is significantly larger than others (e.g., when a "null" value dominates grouping keys). This leaves most executors idle while one grinds.

  • Solution 1: Filter out nulls before joining or grouping.
  • Solution 2: Salting — append a random number to the skewed key before the join, then aggregate again.
  • Solution 3 (Spark 3.x+): Adaptive Query Execution (AQE) can handle skew automatically! Set spark.sql.adaptive.skewJoin.enabled=true.

Adaptive Query Execution (AQE)

Introduced in Spark 3, AQE dynamically optimises execution plans at runtime based on real statistics.

spark.conf.set("spark.sql.adaptive.enabled", "true")
# Dynamically coalesces shuffle partitions
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")

Monitoring and Debugging

Use the Spark UI (usually on port 4040 locally) to check performance:

  • Jobs / Stages Tab: Find slow stages. If a stage hangs at 99%, you have data skew.
  • SQL Tab: View the visual execution plan. Look for heavy "Exchange" blocks (shuffles).
  • Executors Tab: Monitor memory usage and garbage collection (GC) time. GC time > 10% indicates memory pressure.

Best Practices

  • Filter data early (Predicate Pushdown). Read only columns you need.
  • Avoid UDFs (User Defined Functions) in Python unless necessary; they move data out of the JVM to a Python process, breaking Catalyst optimisations. Use built-in SQL functions.
  • Use Parquet or ORC formats over JSON or CSV.
