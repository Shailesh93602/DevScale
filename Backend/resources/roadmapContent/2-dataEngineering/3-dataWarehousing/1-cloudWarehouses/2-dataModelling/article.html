Data Modelling

A robust data model enables analysts to query massive volumes of data efficiently and avoids duplication of business logic. Data Modelling converts raw event data into structured tables prepared for analytical queries.

Dimensional Modelling: Star Schema

Invented by Ralph Kimball, the Star Schema remains the de facto standard for structuring Data Warehouses.

1. Fact Tables

Fact tables record measurements or events (e.g., a sale, a click). They contain mostly numbers and foreign keys.

  • Transaction Fact: One row per event (e.g., fact_orders)
  • Periodic Snapshot Fact: One row per summary period (e.g., fact_monthly_account_balances)
  • Accumulating Snapshot Fact: One row representing an entity's lifecycle (e.g., fact_order_fulfillment updated as the order ships, delivers)

2. Dimension Tables

Dimension tables provide the "who, what, where, when, why, and how". They contain descriptive attributes.

  • dim_customer: Customer name, email, segment
  • dim_product: Product SKU, category, brand
  • dim_date: Year, quarter, month, is_holiday

Queries form a "Star" shape by joining the central Fact table out to the Dimensions.

Snowflake Schema

A variation of the Star Schema where dimension tables are normalized. For example, dim_product relates to a separate dim_category table.

  • Pros: Saves storage space.
  • Cons: Requires more complex JOINs, reducing performance in modern columnar databases. Cloud warehouses (BigQuery, Snowflake) vastly prefer the simpler Star Schema.

Slowly Changing Dimensions (SCDs)

How do we handle a customer moving to a new state?

  • SCD Type 1 (Overwrite): Always overwrite the old state. We lose history. (Good for fixing spelling errors).
  • SCD Type 2 (Add a row): Add a new row with the new state. We use valid_from and valid_to columns to track when they lived there. This creates perfect historical reporting.
  • SCD Type 3 (Add a column): Add an old_state column. Keeps minimal history without adding rows.

Data Vault 2.0 (Advanced)

Designed for massive, highly agile Enterprise Data Warehouses (EDW) that ingest from many source systems. It separates structural information from descriptive data.

  • Hubs: Core business keys (e.g., a Customer ID).
  • Links: Relationships between Hubs (e.g., Customer bought Product).
  • Satellites: Descriptive attributes and history for Hubs/Links over time.

Best Practices

  • Use Surrogate Keys (auto-incrementing integers or UUID hashes) instead of Business Keys (like a changing email) for reliable JOINs.
  • Denormalize dimension tables in modern columnar data warehouses. Storage is cheap, JOINs are expensive.
  • Never update Fact tables if possible; they should be immutable event logs. Update Dimensions instead.
