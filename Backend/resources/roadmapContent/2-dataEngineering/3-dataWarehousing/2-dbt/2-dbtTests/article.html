dbt Testing (Data Quality)

Data build tool (dbt) treats data like code. Just as application code requires unit testing, analytical data pipelines require robust data validations. dbt allows you to write assertions about your tables directly alongside your logic.

Generic Tests

Generic tests validate common scenarios across columns. You define them right inside your model's schema.yml.

version: 2
models:
  - name: dim_users
    columns:
      - name: user_id
        tests:
          - unique
          - not_null
      - name: email
        tests:
          - not_null
      - name: status
        tests:
          - accepted_values:
              values: ['active', 'inactive', 'pending']
      - name: account_id
        tests:
          - relationships:
              to: ref('dim_accounts')
              field: id

dbt ships with those 4 built-in generic tests: unique, not_null, accepted_values, and relationships.

Running Tests

After defining tests in schema.yml, run them from the CLI:

# Run all models, then run all tests
dbt build

# Run ONLY the tests (models must already be built)
dbt test

# Run tests on a specific model
dbt test --select dim_users

Singular Tests (Custom SQL)

For complex logic spanning multiple tables, you can write Singular Tests. A singular test is a SQL file saved in the tests/ directory that returns failing rows.

-- tests/assert_total_amount_is_positive.sql
-- If any row returns from this query, the test fails.

SELECT
    order_id,
    SUM(amount) as total_amount
FROM {{ ref('fct_order_items') }}
GROUP BY 1
HAVING SUM(amount) < 0

Freshness Checks

Ensuring your source data isn't stale is as important as its internal consistency.

version: 2
sources:
  - name: application_db
    database: raw_zone
    tables:
      - name: users
        loaded_at_field: updated_at
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}

Run dbt source freshness to ensure the source data hasn't stopped loading before building downstream models.

dbt Expectations (External Package)

While dbt's built-in tests handle the basics, the dbt-expectations package provides dozens of powerful generic tests (like Great Expectations).

packages:
  - package: calogica/dbt_expectations
    version: 0.8.0
tests:
  - dbt_expectations.expect_column_values_to_match_regex:
      regex: "^[0-9a-zA-Z_.-]+@[0-9a-zA-Z_.-]+\\.[a-zA-Z]+$"

Best Practices

  • Test primary keys with both unique and not_null on every model.
  • Don't ignore the warning severity: config(severity = 'warn') on noisy tests to avoid breaking the pipeline.
  • Run tests automatically in your CI/CD pipeline on every Pull Request to catch bad data changes before they hit production.
