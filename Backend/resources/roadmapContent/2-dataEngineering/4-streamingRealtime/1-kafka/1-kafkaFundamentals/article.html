Apache Kafka Fundamentals

Apache Kafka is a distributed event streaming platform. Unlike traditional message brokers (like RabbitMQ) which delete messages once read, Kafka stores streams of records continuously as an append-only log, allowing multiple consumers to read the same data at their own pace. It is the backbone of real-time data architectures at companies like LinkedIn, Netflix, and Uber.

Core Architecture Concepts

  • Producer: An application that writes (publishes) data to Kafka. Example: A web server sending user click events.
  • Consumer: An application that reads (subscribes to) data from Kafka. Example: A real-time analytics dashboard updating view counts.
  • Broker: A single Kafka server. A Kafka cluster consists of multiple brokers to guarantee high availability and scale out capacity.
  • Topic: A category or feed name to which records are published. Think of it like a table in a database or a folder on a filesystem. Topics are multi-producer and multi-subscriber.
  • Partition: The core of Kafka's scalability. Topics are divided into partitions distributed across brokers. A topic might have 10 partitions spread across 3 brokers. Messages within a single partition are strictly ordered.
  • Offset: A unique, sequential ID assigned to each message as it arrives in a partition. Consumers track their progress by saving the offset they last read.

The Log and Data Retention

Kafka treats topics as persistent, commit logs. When a producer sends a message, it is appended to the end of the partition log. By default, Kafka retains data for 7 days (or based on size limits, e.g., 50GB per partition). After the retention policy triggers, old segments are deleted from disk to free up space. Kafka doesn't know or care if a consumer has read the data; data retention is purely durable.

Because the data persists, a new consumer can attach to an existing topic and read all historical events from the very beginning (Offset 0), making Kafka ideal for rebuilding databases after a crash.

Consumer Groups & Parallelism

A Consumer Group is a cluster of consumer instances working together to read from a topic. Kafka uses consumer groups to parallelize processing:

  • If a topic has 10 partitions and your consumer group has 10 instances, Kafka assigns exactly one partition to each instance. This allows massive simultaneous throughput.
  • If you have more instances than partitions (e.g., 12 instances, 10 partitions), the 2 extra instances sit idle. They act as hot-standbys.
  • If an active instance crashes, Kafka automatically triggers a rebalance, reassigning its partition to the idle instances (or redistributing them among the survivors) within milliseconds.

Producing Messages and Ordering

Messages in Kafka consist of a Key, Value, Timestamp, and optional Headers. The Key determines which partition the message goes to. For example, if you use a user_id as the key, all events generated by that user are hashed to the same exact partition. Because partitions guarantee strict ordering, you guarantee that you process a user's "Checkout" event strictly after their "Add to Cart" event.

from kafka import KafkaProducer
import json

# Setup Producer
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda m: json.dumps(m).encode('utf-8'),
    key_serializer=lambda k: str(k).encode('utf-8')
)

# Send event (Asynchronous)
future = producer.send(
    'website-clicks', 
    key='user_123', 
    value={'action': 'checkout', 'amount': 99.99}
)

# Block until the message is confirmed by the broker
result = future.get(timeout=10)
print(f"Sent to partition {result.partition} at offset {result.offset}")
from kafka import KafkaConsumer
import json

# Setup Consumer in a specific group
consumer = KafkaConsumer(
    'website-clicks',
    bootstrap_servers=['localhost:9092'],
    group_id='analytics-engine',
    auto_offset_reset='earliest',  # Start from beginning if no offset exists
    enable_auto_commit=True,       # Kafka tracks offsets automatically
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

# Continuous polling loop
for message in consumer:
    # { 'action': 'checkout', 'amount': 99.99 }
    event = message.value
    print(f"User {message.key.decode()} performed {event['action']}")

Why use Kafka instead of a Database?

  • Decoupling Microservices: Service A publishes a "UserCreated" event to Kafka. Service B (Emails), Service C (Analytics), and Service D (Recommendations) all read from it at their own pace. Service A doesn't need to know who is listening and doesn't crash if Service B goes offline.
  • High Throughput: Kafka handles millions of messages per second on modest hardware by leveraging sequential disk I/O and zero-copy data transfer to the network socket, avoiding RAM bottlenecks.
  • Fault Tolerance: Partitions are replicated across multiple brokers (Replication Factor = 3 is standard). If the leader broker for a partition crashes, a follower instantly takes over, guaranteeing zero data loss.
