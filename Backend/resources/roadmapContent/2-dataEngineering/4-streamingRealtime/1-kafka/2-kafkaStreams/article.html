Kafka Streams

Kafka Streams is a client library for building applications and microservices where the input and output data are stored in Kafka clusters. It allows you to do stateful or stateless stream processing directly in your Java/Scala application without needing a separate processing cluster (like Spark or Flink).

Why Kafka Streams?

  • No extra cluster needed: It’s just a library you bundle into your microservice. You run standard containers/pods.
  • Exactly-once processing: Built-in support to guarantee records are processed exactly once, even during failures.
  • Stateful operations: It uses RocksDB locally to store intermediate state (like running totals or windowed aggregates), backed up automatically to internal Kafka topics.

Core Abstractions

  • KStream: Represents a stream of records. Every data record represents a self-contained insert. (e.g., A stream of financial transactions).
  • KTable: Represents a changelog stream (like a database table). Every record represents an update or insert based on the key. (e.g., A stream of user profile updates where the key is user_id).
  • GlobalKTable: Like a KTable, but fully replicated to every Kafka Streams application instance. Perfect for joining small lookup tables (e.g., country codes) to massive streams without shuffling data over the network.

Basic Topology Example (Java)

StreamsBuilder builder = new StreamsBuilder();

// 1. Read from the input topic
KStream<String, String> textLines = builder.stream("TextLinesTopic");

// 2. Process: count words using stateful operations
KTable<String, Long> wordCounts = textLines
    .flatMapValues(textLine -> Arrays.asList(textLine.toLowerCase().split("\\W+")))
    .groupBy((key, word) -> word)   // re-keys the stream by 'word'
    .count(Materialized.as("Counts")); // stores state locally in RocksDB

// 3. Write back to output topic
wordCounts.toStream().to("WordsWithCountsTopic", Produced.with(Serdes.String(), Serdes.Long()));

Joins in Kafka Streams

Kafka Streams supports joining KStreams to KTables, KStreams to KStreams, and KTables to KTables. To join them, they must be co-partitioned (meaning both topics must have the exact same number of partitions and use the same partitioning strategy on the join key).

Time & Windowing

You can aggregate events inside time brackets using windows:

  • Tumbling Windows: Fixed size, non-overlapping. (e.g., hourly aggregates: 1:00-2:00, 2:00-3:00).
  • Hopping (Sliding) Windows: Fixed size, overlapping. (e.g., a 1-hour window that advances every 5 minutes).
  • Session Windows: Based on periods of activity separated by gaps of inactivity. (e.g., user session tracking).
