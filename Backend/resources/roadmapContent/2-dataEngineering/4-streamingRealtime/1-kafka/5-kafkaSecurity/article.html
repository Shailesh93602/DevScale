Kafka Security & Access Control Lists (ACLs)

In a development environment, running Kafka without authentication is perfectly fine. But in a production environment, a Kafka cluster might be the central nervous system for an entire enterprise, receiving millions of payment transactions, healthcare records, and user behavioral events per second. Without proper security configurations, any internal server that can reach the Kafka Broker could read every single sensitive message.

Encryption In-Transit (SSL/TLS)

Without SSL, every message flying over the network between your Producer application and the Kafka Broker is transmitted as raw, unencrypted plain text. Any sophisticated network sniffer could intercept and read every single sensitive message.

You configure SSL/TLS encryption by generating a Certificate Authority (CA), creating Keystores for each Broker, and configuring the Producer and Consumer clients to use TLS handshakes.

# On the Kafka Broker server-side configuration
listeners=SSL://kafka-broker:9093
ssl.keystore.location=/var/ssl/private/kafka.server.keystore.jks
ssl.keystore.password=secure_keystore_password
ssl.truststore.location=/var/ssl/private/kafka.server.truststore.jks

Authentication (SASL)

SASL (Simple Authentication and Security Layer) answers the question: "Who are you?" SSL encrypts the message payload, but it doesn't inherently prove the Producer's identity.

The most popular SASL mechanisms used in enterprise Kafka clusters are:

  • SASL/PLAIN: The simplest form. The Producer sends a username and password in the connection header. This is vulnerable unless wrapped tightly with SSL/TLS.
  • SASL/SCRAM: Significantly more robust. Passwords are cryptographically hashed using a salt challenge. Even if an attacker intercepts the authentication exchange, they cannot recover the raw password.
  • SASL/GSSAPI (Kerberos): The most common protocol in large enterprise environments. The identity is managed by a massive central Kerberos server, enabling single sign-on across thousands of services.

Authorization (Access Control Lists - ACLs)

Once you know who is connecting, ACLs control what they are allowed to do. With ACLs, you can explicitly grant or deny fine-grained permissions per user, per topic, per operation.

# Grant user 'payments-service' permission to WRITE to the 'transactions' topic only
bin/kafka-acls.sh --authorizer-properties \
  --add --allow-principal User:payments-service \
  --operation Write --topic transactions

# Grant 'analytics-team' permission only to READ from 'transactions'
bin/kafka-acls.sh --authorizer-properties \
  --add --allow-principal User:analytics-team \
  --operation Read --topic transactions --group analytics-consumer-group

This ensures that the Payments microservice can write new transaction events but can never read from the topic. The Analytics team can read existing events but can never produce new ones. Any unauthorized access attempt is immediately rejected and logged.
