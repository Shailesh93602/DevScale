Kafka Architecture & ZooKeeper

To truly understand how Apache Kafka achieves its legendary reliability and fault tolerance, you need to understand the internal components that make up a production Kafka cluster.

Brokers

A Kafka Broker is simply a server. A production Kafka cluster typically consists of 3, 5, or more Broker servers running together as a cluster. Each Broker is responsible for persistently writing incoming messages to its local hard drive (called a "log segment") and serving those messages to consumers upon request.

One Broker in the cluster is elected as the Controller Broker. The Controller is responsible for managing the cluster's metadata: which partitions exist, which broker is the leader for each partition, and reassigning partitions when a broker crashes.

Leaders and Replicas

For fault tolerance, each Partition has one Leader Broker and multiple Replica Brokers. All Producers write to the Leader, and the Leader synchronously replicates the data to the Replica Brokers.

If the Leader Broker crashes, Kafka automatically elects one of the in-sync Replicas as the new Leader. Because the Replica already has an exact copy of all data, there is zero data loss, and the switchover happens within seconds without any manual intervention.

# Creating a topic with 3 partitions and a replication factor of 3
# This means each partition will have 1 leader + 2 replicas across the cluster
bin/kafka-topics.sh --create \
  --topic payment-events \
  --partitions 3 \
  --replication-factor 3 \
  --bootstrap-server localhost:9092

ZooKeeper's Role (Legacy)

In older versions of Kafka (pre 2.8), Apache ZooKeeper was an external, mandatory dependency. ZooKeeper acted as the central brain of the cluster, storing all critical metadata: which Brokers are alive, who the elected Controller is, and what the current partition leader assignments are.

This created significant operational complexity. Data Engineering teams had to maintain and scale two complete distributed systems (Kafka AND ZooKeeper) simultaneously.

KRaft: The ZooKeeper Replacement

Starting with Kafka 3.3+, Kafka introduced KRaft (Kafka Raft), a complete internal replacement for ZooKeeper. With KRaft, Kafka manages its own metadata internally using the Raft consensus algorithm, eliminating the operational burden of an external ZooKeeper cluster entirely. All modern Kafka deployments should use KRaft mode.
