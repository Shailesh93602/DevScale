Apache Flink Fundamentals

Apache Flink is an open-source framework for stateful computations over unbounded and bounded data streams. While Kafka is the "storage pipeline" that holds the streams of data, Flink is the "processing engine" that actually computes answers (e.g., aggregations, joins, machine learning inference) in real-time as the data flows through.

Batch vs. Stream Processing

Historically, Big Data was processed in batches (using Hadoop MapReduce or Apache Spark). It involved waiting for a day’s worth of data to accumulate in an S3 bucket, running a gigantic query, and updating a dashboard once a night.

In contrast, Stream Processing analyzes data the millisecond it arrives. Think of a credit card fraud detection system: if you wait until midnight to run a batch job, the fraudster has already made off with the money. Flink enables sub-second latency responses to every single event on a Kafka topic.

  • Unbounded Streams: Data has a start but no end. Continuous processing (Stream).
  • Bounded Streams: Data has a start and an end. Processed completely and then stops (Batch).

Flink uniquely treats batch processing as simply a special edge-case of stream processing, allowing you to use the exact same APIs for both paradigms.

Core Architecture

A Flink cluster consists of multiple components acting together:

  • JobManager: The brain of the operation. It receives the code (JobGraph) from the client, requests resources (slots) from the cluster manager (like Kubernetes or YARN), coordinates checkpoints, and handles recovery if a worker dies.
  • TaskManager: The muscle. These are the worker nodes that execute the actual user code (operators like filter, map, join, or window). Large clusters might run hundreds of TaskManagers.
  • Task Slots: Each TaskManager is divided into a fixed number of slots (typically one per CPU core). An operation is assigned to a slot to guarantee isolated memory and execution time.

API Abstraction Levels

Flink offers developers different ways to express their computation logic, ranging from low-level control to high-level declarative syntax:

  1. ProcessFunction (Low-Level): Direct control over events, detailed state manipulation (e.g., storing a custom variable per user), and manual timer management. Use this when writing complex, custom event-driven applications.
  2. DataStream API (Core): The standard Java/Scala/Python API for building pipelines using functional transformations like .map(), .filter(), and .keyBy().
  3. Table API & SQL (High-Level): Write standard ANSI-SQL queries against a continuous stream. Flink translates the SQL into a distributed stream processing job automatically.
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.datastream.DataStream;

public class BasicFlinkApp {
    public static void main(String[] args) throws Exception {
        // 1. Set up the execution environment
        final StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // 2. Read from a source (Kafka, File, Socket)
        DataStream<String> text = env.socketTextStream("localhost", 9999);

        // 3. Define the transformations (Filter out short lines, convert to upper case)
        DataStream<String> processed = text
            .filter(line -> line.length() > 5)
            .map(String::toUpperCase);

        // 4. Output to a sink (Kafka, Database, Print)
        processed.print();

        // 5. Execute the streaming pipeline
        env.execute("Basic Stream Filtering Job");
    }
}

State and Why It Matters

A simple filter() operation is stateless: looking at one event is enough to decide if it passes the filter. However, aggregating a sum (e.g., "Total sales per hour") is stateful. Flink must remember the running total between the arrival of the first event and the thousandth event.

Managing this state correctly is why Flink is difficult but powerful. State is typically partitioned by a key (e.g., userId), ensuring that all events for User A are routed to the same TaskManager, which holds the current state for User A in its local memory (often backed by RocksDB to prevent memory overflow).

Checkpointing & Exactly-Once Semantics

Because state lives in the RAM/local disk of a TaskManager, if that node crashes, the running total is lost. Flink solves this with Distributed Snapshots (Checkpoints).

Every few seconds, the JobManager commands all TaskManagers to pause processing and flush their current state incrementally to durable storage (like AWS S3). If a node dies, a new one spins up, reads the latest Checkpoint from S3, rewinds the Kafka consumer offset to the exact moment of that checkpoint, and resumes. This provides mathematically guaranteed Exactly-Once processing — even if servers catch fire, your output results are never over-counted or under-counted.
