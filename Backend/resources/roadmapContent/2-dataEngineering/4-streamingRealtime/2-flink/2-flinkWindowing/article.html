Event Time & Windowing in Flink

When computing aggregations over continuous data streams (like "total user signups"), you cannot simply sum forever. You must slice the infinite stream into finite buckets of computation. This is the concept of Windowing. But before you can slice the stream, you must solve a fundamental problem of distributed systems: defining what "time" actually means.

The Concept of Time in Streaming

In batch processing, all data is present before the job starts. In streaming therapy, data arrives unpredictably, sometimes out of order, and sometimes severely delayed. Flink handles this by explicitly distinguishing between Processing Time and Event Time.

  • Processing Time: The system clock of the machine executing the Flink program. If your server is running slowly, real-time calculations stretch out. This is easy to implement but highly inaccurate if the network drops historical events.
  • Event Time (The Standard): The time embedded in the actual event payload (e.g., the timestamp generated by a mobile app when a user clicked "buy"). This is accurate, immutable, and guarantees deterministic processing regardless of network delays or system crashes.

The Problem of Late Data & Watermarks

If you perform an Event Time calculation (e.g., "Revenue between 1:00 PM and 2:00 PM") based on when users clicked "buy", how does Flink know when it is safe to calculate the total and close the 2:00 PM window?

Because a user might have been inside a subway tunnel without service at 1:59 PM, and their device only transmits the click event to Kafka at 2:15 PM, the stream is technically out of order. Flink solves this using Watermarks.

A Watermark is a special internal event Flink injects into the stream declaring, "I guarantee no events older than timestamp T will arrive anymore."

  • Heuristic Watermarks: You configure a maximum out-of-orderness tolerance (e.g., "Wait up to 5 seconds"). If an event arrives 6 seconds late, it is dropped or sent to a separate "side output" for manual review.

Window Functions: Slicing the Stream

Once Flink understands Event Time and Watermarks, you apply Windows (groupings over time constraints) to the stream. There are four primary window assignments:

1. Tumbling Windows (Fixed)

Non-overlapping, fixed-size windows. Every event belongs to exactly one window.

Example: "Calculate total ad views per hour." The windows are 1:00-2:00, 2:00-3:00, 3:00-4:00.

// Java API Example: Sum value per hour based on Event Time
DataStream<SensorReading> stream = ...;

stream
    .keyBy(r -> r.sensorId)
    .window(TumblingEventTimeWindows.of(Time.hours(1)))
    .sum("temperature");

2. Sliding Windows

Overlapping windows of a fixed size that advance by a specified interval. An event can belong to multiple windows.

Example: "Calculate average website traffic over the last 15 minutes, updated every 1 minute." If you click at 1:05, you are included in the 12:51-1:06 window, the 12:52-1:07 window, and so on.

stream
    .keyBy(r -> r.sensorId)
    .window(SlidingEventTimeWindows.of(Time.minutes(15), Time.minutes(1)))
    .sum("temperature");

3. Session Windows

Dynamic windows based on activity, not fixed time. A session window groups events that burst closely together and closes when a period of inactivity (gap) passes.

Example: "Group a user's clicks into a 'browsing session', ending the session if they are inactive for 30 minutes." If they click at 1:00 and 1:15, the session stays open. If the next click is 2:00, the first session closed at 1:45.

stream
    .keyBy(r -> r.userId)
    .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
    .process(new AnalyzeBrowsingBehavior());

4. Global Windows

Assigns all events with the same key to a single, never-ending window. This only works if you specify extremely complex custom Triggers (rules for when to emit a calculation, such as "emit the sum every time the total jumps above $1,000" or "emit every 10 events").

Allowed Lateness & Side Outputs

Handling late data perfectly is arguably Flink's greatest strength. Even if a watermark triggered the closing of a 2:00 PM window and emitted the aggregate result to a dashboard, Flink can be configured with .allowedLateness(Time.hours(24)). If an impossibly late event (23 hours late) from the 2:00 PM timeframe suddenly arrives, Flink re-opens the 2:00 PM window, updates the previously computed aggregation, and emits a correction event to the dashboard.

Events that exceed the allowed lateness tolerance are sent to a "Dead Letter Queue" (Side Output) for manual processing rather than crashing the primary stream.
