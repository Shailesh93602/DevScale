Flink Deployments & Connectors

Writing a Flink job locally is one thing; running it reliably in a production environment 24/7 is another challenge entirely. Flink supports several deployment modes and provides a rich ecosystem of Connectors for integrating with other systems in your data stack.

Flink Deployment Modes

Flink can be deployed across several strategies, each with different trade-offs:

  • Session Mode: A single long-running Flink cluster is pre-provisioned (like a shared server). You submit multiple jobs to it, and all jobs share the available cluster resources. This is efficient for development and test environments but can cause resource contention — one memory-hungry job can steal resources from all other jobs running on the same cluster.
  • Application Mode: The recommended mode for serious production deployments. Each Flink job gets its completely own dedicated, isolated cluster. The cluster is automatically created when the job is submitted and destroyed when the job completes. This provides complete resource isolation and is the cleanest approach on Kubernetes or YARN.
  • Per-Job Mode: Similar to Application Mode, but the cluster lifecycle is managed by an external resource manager like YARN or Mesos rather than Flink itself.

Flink Connectors (Sources and Sinks)

A Flink streaming application has a Source (where data comes in) and a Sink (where results go out). Flink provides official, high-performance connectors for all major systems in the data engineering ecosystem:

// Source: Read a real-time stream from a Kafka Topic
KafkaSource<String> kafkaSource = KafkaSource.<String>builder()
    .setBootstrapServers("kafka:9092")
    .setTopics("payment-events")
    .setGroupId("flink-processor")
    .setStartingOffsets(OffsetsInitializer.latest())
    .setValueOnlyDeserializer(new SimpleStringSchema())
    .build();

// Sink: Write the processed results to a JDBC-compatible database
JdbcSink.sink(
    "INSERT INTO aggregated_payments (user_id, total) VALUES (?, ?)",
    (statement, row) -> { /* set parameters */ },
    JdbcExecutionOptions.defaults(),
    new JdbcConnectionOptions.Builder().withUrl("jdbc:postgresql://...").build()
);

The Kafka-Flink-Iceberg Pipeline

The most powerful modern Data Engineering pattern is the Kafka → Flink → Apache Iceberg → Data Warehouse pipeline. Real-time events flow from Kafka into Flink for stateful enrichment and windowed aggregations. The results are then written to an Apache Iceberg table on S3, which is immediately queryable by Snowflake or BigQuery without any loading process. This architecture provides both real-time stream processing and batch analytics on the same data store, eliminating the need for separate Lambda architecture pipelines entirely.
