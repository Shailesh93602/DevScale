Data Lineage & Observability

Imagine a critical KPI on the CEO's dashboard showing "Total Revenue This Quarter" suddenly drops by 40% overnight. Is it a real business problem, or did a Data Engineer accidentally break a transformation pipeline? Without data observability tools, tracking down the root cause is a painstaking, hours-long process of manually inspecting dozens of interdependent SQL models and ETL scripts.

Data Lineage

Data Lineage is the ability to trace the complete journey of a specific data point, from its original raw source all the way to its final representation in a dashboard or machine learning model.

Tools like Apache Atlas, OpenLineage, and the built-in lineage graph in dbt provide a visual map answering: "If I change this raw `users` table in the source database, which 23 downstream models, dashboards, and ML features will be impacted?"

When dbt runs, it automatically generates a rich manifest.json that encodes the exact dependency graph of every model in your project, which integrates directly with lineage visualization platforms.

The Silent Data Quality Crisis

The most dangerous data quality issues are not crashes — crashes are immediately obvious. The most dangerous failures are silent data quality issues:

  • The source system team changed a column name from user_id to userId without telling anyone. Your pipeline silently starts writing NULL for every row.
  • A new software release introduced a bug that sends 0 as the price for every checkout event. Your revenue model still runs successfully and writes clean-looking aggregations — but the numbers are wrong.

Data Observability Platforms

Modern Data Observability platforms (like Monte Carlo, Soda Core, and Great Expectations) work by continuously monitoring your Data Warehouse tables for anomalies using a combination of statistical tests and user-defined rules. They check metrics like:

  • Volume: "The `fact_transactions` table usually grows by ~50,000 rows per hour. If it's Monday at 9 AM and it grew by only 3 rows, trigger an alert."
  • Freshness: "The `stg_payments` model should be rebuilt by Airflow every hour. If it's been 3 hours since the last run, send a Slack alert to the Data Engineering team."
  • Distribution: "The `country_code` column should contain ~200 distinct values. If it suddenly has 1 distinct value, something is catastrophically wrong upstream."

The OpenLineage Standard

OpenLineage is an open-source, vendor-neutral standard for capturing lineage metadata from tools like Apache Airflow, dbt, Apache Spark, and Flink. Instead of each tool inventing its own proprietary lineage format, they all emit events in the OpenLineage schema, enabling a unified lineage graph across your entire data stack.
