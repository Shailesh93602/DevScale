Model Serving

Once a machine learning model is trained and validated, it needs to be made available for inference (generating predictions). This process is known as Model Serving. In the MLOps lifecycle, Data Engineers and ML Engineers work closely here to ensure standardisation, scalability, and monitoring.

Batch Prediction vs. Real-Time Inference

Models are deployed differently depending on the business need:

  • Batch Prediction: The model is loaded onto a distributed computing cluster (like Apache Spark) in an Airflow DAG. It processes millions of records overnight, storing the pre-computed predictions in a database (like PostgreSQL or Cassandra). When a user opens the app, the backend simply runs a fast SELECT query. (High throughput, high latency).
  • Real-Time Inference: The model is exposed as a REST or gRPC API endpoint. When a user taking an action on the app triggers a request, the model generates a prediction on-the-fly and returns it dynamically within milliseconds. (Low throughput, low latency).

The Complexity of Real-Time Serving

Serving an ML model in a generic Flask or FastAPI app is risky for production due to framework-specific dependencies and the hardware demands of Neural Networks (e.g., managing GPU memory). If your Python API is blocked by a massive tensor matrix multiplication, all subsequent incoming HTTP requests will timeout.

Model Serving Frameworks

To handle these complexities safely, the industry uses dedicated Model Serving platforms:

BentoML

BentoML simplifies generating a production-ready API wrapper around models natively tracked from MLflow or created in Scikit/PyTorch. It automatically containerises the model into a standard Docker image (a "Bento").

import bentoml
from bentoml.io import JSON

# Define the runner (handles hardware logic inside BentoML)
runner = bentoml.xgboost.get("my_rf_model:latest").to_runner()
svc = bentoml.Service("fraud_detector", runners=[runner])

@svc.api(input=JSON(), output=JSON())
async def predict(input_data):
    # Asynchronously serves requests without blocking
    result = await runner.predict.async_run(input_data)
    return {"fraud_probability": result}

KServe & Seldon Core

These are Kubernetes-native platforms tailored for massive enterprise MLOps. Once you have a Dockerised model, these tools orchestrate it on Kubernetes, providing:

  • Serverless Autoscaling: Scale from zero replicas to hundreds simultaneously based strictly on API request volume.
  • A/B Testing: Route exactly 10% of production traffic to "Model V2" and 90% to "Model V1" to confidently assess if the new model performs better in the real world.

Monitoring Model Drift

Unlike standard APIs, ML APIs can silently fail due to Data Drift. If user behavior changes over time, the model's accuracy deteriorates, even if the API responds with a successful HTTP 200 OK. A robust serving infrastructure continuously logs the incoming live data and compares its distribution against the data the model was originally trained on, triggering an alert for a pipeline retrain if the drift becomes significant.
