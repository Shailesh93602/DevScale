Apache Iceberg: ACID on Data Lakes

Before open table formats like Apache Iceberg existed, updating data in a Data Lake was a nightmare. If you wanted to delete all records for a specific user from a collection of Parquet files on S3 (for GDPR compliance), you had to:

  1. Read the entire massive Parquet file into Spark.
  2. Filter out the user's records in memory.
  3. Rewrite the entire file back to S3.
  4. Hope that no other job was reading the file while you were overwriting it (there was no locking mechanism).

Apache Iceberg solved all of this by introducing a metadata layer on top of standard Parquet files that brings full ACID transaction guarantees to the Data Lake.

The Iceberg Metadata Layer

An Iceberg Table is not just a folder of Parquet files. It's a structured hierarchy of metadata files:

  • Metadata Files: JSON files describing the current table schema, partition specifications, and pointers to Snapshot files.
  • Snapshot files (Manifest Lists): Each snapshot represents the table's state at a specific point in time. When you INSERT or DELETE rows, Iceberg creates a new snapshot rather than modifying existing files.
  • Manifest Files: Track which specific Parquet data files belong to a given snapshot, along with column-level statistics for each file.

ACID Transactions & Schema Evolution

Because Iceberg manages snapshots immutably, it can provide ACID guarantees natively:

  • Atomicity: An INSERT of 1 million rows either fully succeeds (creating a new snapshot) or fully fails. There is no partial write visible to concurrent readers.
  • Isolation: A Spark job reading the table will read a consistent snapshot even while another Spark job is simultaneously writing new data.

Iceberg also handles Safe Schema Evolution. You can add a new column, rename an existing column, or drop a column without rewriting any Parquet files. Iceberg's metadata layer handles the mapping transparently.

Time Travel Queries

-- Query the table exactly as it appeared at a specific timestamp
SELECT * FROM analytics.user_events
  TIMESTAMP AS OF '2024-01-15 00:00:00';

-- Query a specific historical snapshot by snapshot ID
SELECT * FROM analytics.user_events
  FOR VERSION AS OF 8483296678498373637;

Because old snapshots are preserved, Time Travel allows you to instantly recover from accidental deletes, run reproducible historical analyses, and debug data quality issues by comparing the table at different points in time.

Hidden Partitioning

In traditional Hive-style partitioning, users must manually specify partition columns in every query: WHERE dt='2024-01-15'. Iceberg's Hidden Partitioning automatically applies partition filters based on query predicates, meaning users just write WHERE event_time BETWEEN '...' AND '...' and Iceberg silently prunes irrelevant partitions without the user needing to know the underlying partition scheme.
