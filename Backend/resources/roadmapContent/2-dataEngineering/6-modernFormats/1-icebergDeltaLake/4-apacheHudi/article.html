Apache Hudi: Incremental Processing

Apache Hudi (Hadoop Upserts Deletes and Incrementals) was originally developed by Uber's Data Engineering team to solve a very specific, urgent problem: how do you efficiently ingest millions of real-time ride and payment record updates directly into their massive S3-based Data Lake?

Before Hudi, Uber's pipeline had to periodically rewrite entire massive Parquet partitions to incorporate updates from their OLTP database. Hudi introduced native, record-level upsert and delete semantics directly on top of cloud object storage, enabling true incremental data lake ingestion.

Copy-on-Write (CoW) Tables

In Copy-on-Write mode, when you upsert a batch of records, Hudi identifies exactly which Parquet files contain the affected records and rewrites only those files — incorporating all new updates and deletions in a fresh version of those files.

Trade-offs:

  • ✅ Excellent Read Performance: All data is always in clean, optimized Parquet files. No merge overhead at read time.
  • ❌ Higher Write Latency: Rewriting Parquet files for each batch is expensive. Not suitable for very high-frequency micro-batch ingestion.

Best use case: Nightly batch ETL jobs that ingest millions of daily updates. Read-heavy workloads served to many BI Analysts.

Merge-on-Read (MoR) Tables

In Merge-on-Read mode, new updates are written directly into small delta log files (in Avro format) alongside the existing base Parquet files. The actual merge of delta logs into the base Parquet column store happens lazily during background compaction jobs (typically triggered hourly or daily).

Trade-offs:

  • ✅ Near-Real-Time Ingestion: Writing to small Avro delta files is extremely fast. Can ingest high-frequency streaming updates with very low latency.
  • ❌ Higher Read Overhead: Every read-time query must merge the base Parquet files with the delta log files, adding computational overhead to analytical queries.

Hudi's Incremental Query Mode

Hudi's most distinctive feature is its Incremental Query API, which allows downstream consumers to efficiently query only the records that changed since a specific commit timestamp:

# Only read records committed after a specific timestamp -- extremely efficient!
spark.read.format("hudi") \
    .option("hoodie.datasource.query.type", "incremental") \
    .option("hoodie.datasource.read.begin.instanttime", "20240115120000") \
    .load("s3://lake/transactions/")

This is architecturally superior to full table scans for downstream pipelines that only need to process new changes, making Hudi especially powerful for building reliable, cost-efficient Change Data Capture pipelines on top of Data Lakes.
