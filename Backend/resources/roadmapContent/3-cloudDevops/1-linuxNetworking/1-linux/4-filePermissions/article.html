File Permissions & Ownership

Linux is a multi-user operating system, and every single file and directory on the system has a clear ownership model and permission set controlling exactly who can read it, write to it, or execute it. Understanding this model is non-negotiable for any DevOps engineer or system administrator working in production Linux environments.

The Permission Model

When you run ls -la in a terminal, you'll see output like:

-rwxr-xr-- 1 deploy devops 4096 Mar 10 08:30 deploy.sh
drwxr-x--- 2 root   root   4096 Mar 10 08:00 secrets/

The first 10 characters break down as follows:

  • Character 1: File type. - = regular file, d = directory, l = symlink
  • Characters 2-4: Owner permissions (User)
  • Characters 5-7: Group permissions
  • Characters 8-10: Others (everyone else) permissions

Each set of 3 characters represents r (read), w (write), x (execute). A - means that permission is denied.

The chmod Command

You change permissions with chmod. There are two modes: symbolic and octal.

# Symbolic mode: give owner execute, remove others write
chmod u+x,o-w deploy.sh

# Octal mode: the most common form in production scripts
# r=4, w=2, x=1 — add them up per group
chmod 755 deploy.sh   # Owner: rwx (7), Group: r-x (5), Others: r-x (5)
chmod 600 .ssh/id_rsa  # Owner: rw- (6), Group: --- (0), Others: --- (0)
chmod 644 index.html   # Owner: rw- (6), Group: r-- (4), Others: r-- (4)

The chown Command

Ownership is managed with chown. The root user can change any file's owner; regular users can only change files they own.

# Change owner to 'nginx', group to 'www-data'
sudo chown nginx:www-data /var/www/html/index.html

# Recursively change ownership of an entire directory tree
sudo chown -R deploy:devops /opt/application/

Special Permission Bits: setuid, setgid, and Sticky Bit

  • setuid (4000): When set on an executable, the program runs with the permissions of the file owner, not the user who launched it. This is how /usr/bin/passwd allows regular users to modify /etc/shadow (which is owned by root).
  • setgid (2000): When set on a directory, new files created inside inherit the directory's group, not the creating user's primary group. Critical for shared collaborative directories.
  • Sticky Bit (1000): When set on a directory, only the file's owner can delete or rename that file, even if others have write access. Used on /tmp so users cannot delete each other's temporary files.
chmod 1777 /tmp          # Sticky bit + full permissions
chmod 2755 /shared-dir   # setgid on shared directory
ls -la /tmp  # Shows as drwxrwxrwt — the 't' indicates sticky bit

Access Control Lists (ACLs)

Standard Unix permissions only allow one owner and one group. ACLs extend this, allowing you to grant specific permissions to multiple individual users and groups:

# Grant alice read/write access to a specific file
setfacl -m u:alice:rw /opt/project/config.json

# Grant the 'contractors' group read-only access
setfacl -m g:contractors:r /opt/project/config.json

# View the current ACL entries
getfacl /opt/project/config.json
