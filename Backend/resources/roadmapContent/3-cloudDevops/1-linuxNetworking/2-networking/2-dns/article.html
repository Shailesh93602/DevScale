DNS (Domain Name System)

The Domain Name System (DNS) is often called the "phonebook of the Internet." Humans interact online through easily readable domain names like google.com or nytimes.com. Web browsers, however, communicate via Internet Protocol (IP) addresses (e.g., 142.251.46.110 or 2607:f8b0:4005:800::200e). DNS translates these human-readable domain names to machine-readable IP addresses.

The Recursive Resolution Process (How "example.com" loads)

When you type "example.com" into your browser, a complex hierarchy of servers works together to find the IP address. This happens in milliseconds:

  1. Browser / OS Cache: The browser first checks its own cache, then the Operating System cache, to see if it recently resolved example.com. If not, it asks a DNS Resolver.
  2. Recursive Resolver (ISP / Google 8.8.8.8): Usually provided by your Internet Service Provider, this server acts as the middleman. If it doesn't have the answer cached, it starts querying the hierarchy.
  3. Root Name Server: The Resolver asks the Root Server (there are 13 logical root servers globally). The Root doesn't know the IP, but it knows which server handles the .com Top-Level Domain (TLD).
  4. TLD Name Server: The Resolver asks the .com TLD Server. The TLD Server knows which specific Authoritative Name Server is responsible for example.com.
  5. Authoritative Name Server: This is the final server, owned by whoever manages the domain (e.g., AWS Route 53, Cloudflare, GoDaddy). It holds the actual DNS records. It returns the exact IP address mapped to the A Record back down the chain to the client.

Core DNS Record Types

An Authoritative Server holds Zone Files containing different types of DNS records:

  • A Record (Address): The most basic record. Maps a domain to an IPv4 address (e.g., example.com -> 104.18.2.1).
  • AAAA Record: Maps a domain to a modern IPv6 address.
  • CNAME (Canonical Name): Maps an alias name to a true (canonical) domain name. You cannot point a CNAME to an IP address. Typically used to point subdomains (e.g., www.example.com) to the root domain (example.com), or to external services like a Heroku app format (xyz.herokuapp.com). Note: A CNAME cannot be placed at the root/apex of a domain.
  • MX Record (Mail Exchange): Dictates which mail servers accept email sent on behalf of your domain (e.g., pointing to Google Workspace servers). Includes priorities for failover.
  • TXT Record: Arbitrary text. Originally for human notes, it is now exclusively used for security. Specifically for proving domain ownership (e.g., to generate an SSL cert or verify Google Search Console) or Email authentication (SPF, DKIM, and DMARC records to prevent spoofing).
  • NS Record (Name Server): Indicates which servers are authoritative for the zone. When you buy a domain on Namecheap but want to manage it in AWS, you change the domain's NS records to point to AWS Route 53.

DNS TTL (Time To Live)

Every DNS record has a TTL value (in seconds). It dictates how long Resolving Name Servers and browsers are allowed to cache the record before they must ask the Authoritative Server for an update.

For example, a standard A Record might have a TTL of 86400 (24 hours). If you change the IP address of your server, users around the world might continue hitting the old IP for up to 24 hours until their local ISP cache expires.

DevOps Best Practice: If you are planning an IP migration or blue-green deployment, temporarily lower your TTL to 60 seconds (1 minute) a day beforehand. When you make the switch, caching servers update almost instantly. You can raise the TTL back to 24 hours after a successful migration to reduce query costs.

Advanced DNS Routing (AWS Route 53 / Cloudflare)

Modern Authoritative Name Servers offer intelligent routing policies that go far beyond simple IP mappings:

  • Weighted Routing: Route 80% of traffic to Server A, and 20% to Server B. Used for Canary Deployments or A/B testing.
  • Latency-Based Routing: Route users to the AWS Region that provides the lowest network latency. A user in Tokyo gets the Tokyo IP; a user in Berlin gets the Frankfurt IP.
  • Failover / Active-Passive Routing: Route traffic to the Primary datacenter. If the DNS server detects the Primary is down via an automated Health Check, it automatically updates the A Record to point to the Disaster Recovery (Secondary) datacenter.

Troubleshooting with dig

When a website won't load, the first thing a DevOps engineer checks is DNS resolution from the terminal using dig or nslookup.

# Query the A record for google.com
dig google.com +short
# Output: 142.251.46.110

# Check where the CNAME points
dig www.facebook.com CNAME

# Query a specific nameserver directly (bypassing local ISP caching)
# This is vital for checking if an update has "propagated" fully
dig @8.8.8.8 mynewdomain.com

# Fetch all records (ANY) including MX and TXT
dig example.com ANY
