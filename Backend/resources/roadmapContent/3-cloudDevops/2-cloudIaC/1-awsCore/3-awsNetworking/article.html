AWS Networking: VPC, Subnets & Routing

AWS Networking is the foundation of all cloud security and infrastructure. Every EC2 instance, Lambda function (usually), and RDS database you provision lives inside an isolated, private network boundary called an Amazon Virtual Private Cloud (VPC). A VPC gives you complete control over your virtual networking environment, including IP address ranges, subnets, route tables, and network gateways.

The VPC & IPv4 CIDR Blocks

When you create a VPC, you assign it an IPv4 CIDR block (e.g., 10.0.0.0/16). This defines the total pool of private IP addresses available in the network. A /16 VPC provides 65,536 distinct IP addresses. By default, resources inside a VPC can communicate with each other freely across all Availability Zones, but the VPC is entirely cut off from the public Internet.

Subnets and Availability Zones

You cannot launch an EC2 instance directly into a VPC; you must launch it into a Subnet. Subnets divide your VPC's large CIDR block into smaller, manageable chunks (e.g., 10.0.1.0/24 provides 256 IPs).

Crucially, a VPC spans an entire AWS Region (like us-east-1: N. Virginia), but a Subnet is strictly tied to a single Availability Zone (AZ) (like us-east-1a). To achieve High Availability—surviving an entire data center going offline—you must architect your application across multiple subnets in different AZs.

Public vs. Private Subnets and the Internet Gateway

The distinction between a "Public" and "Private" subnet is solely determined by Routing rules.

  • Internet Gateway (IGW): A horizontally scaled, redundant VPC component that allows communication between your VPC and the Internet.
  • Public Subnet: A subnet whose Route Table contains a default route (0.0.0.0/0) pointing directly to the Internet Gateway. Instances here (like Load Balancers or jump hosts) must have an Elastic IP or Auto-assigned Public IP to be reachable from the outside.
  • Private Subnet: A subnet whose Route Table does not point to the IGW. Instances in a private subnet (like backend Node.js servers or RDS databases) cannot be accessed directly from the Internet, making them inherently secure from external hacking attempts.

NAT Gateways: Giving Internet Access to Private Resources

If your backend server in a Private Subnet needs to download a security patch using apt-get or hit a third-party API (like Stripe), it needs a way to reach the Internet. It cannot use the IGW because it has no Public IP address.

The solution is a NAT Gateway (Network Address Translation). You provision the NAT Gateway in your Public Subnet (where it gets an Elastic IP). You then update the Private Subnet's Route Table: "Send all Internet traffic to the NAT Gateway."

The NAT Gateway proxies the traffic to the Internet, receives the response, and forwards it back to the private instance. It only allows outbound connections. Hackers on the outside cannot initiate a connection inward through a NAT.

# Standard 3-Tier Web Architecture Route Tables

# Public Subnet Route Table (For Load Balancers / Bastions / NAT Gateway)
Destination    Target
10.0.0.0/16    local        # Internal VPC communication is default and implied
0.0.0.0/0      igw-idxxx    # All internet traffic goes to Internet Gateway

# Private Subnet Route Table (For App Servers / Databases)
Destination    Target
10.0.0.0/16    local        # Internal communication
0.0.0.0/0      nat-idxxx    # Outbound internet traffic goes to NAT in Public Subnet

Network ACLs vs. Security Groups

AWS provides two layers of firewalls to secure your VPC:

1. Security Groups (Stateful, Instance-level)

Act as a virtual firewall tied directly to an EC2 instance (or RDS database). They operate on an "allow-list" basis—all traffic is denied by default. Stateful means if an inbound request is allowed (e.g., port 443), the return traffic is automatically allowed, regardless of outbound rules. Standard architectural practice places the Load Balancer in an "HTTP/S Allowed" Security Group, and the Backend Servers in a Security Group that only accepts inbound traffic from the Load Balancer's Security Group ID.

2. Network ACLs (Stateless, Subnet-level)

Act as a firewall at the Subnet boundary, affecting all instances inside. They process numbered rules in order (e.g., Allow port 80 at rule 100, Deny Blocklist IP at rule 200). Stateless means you must explicitly allow both inbound and outbound return traffic. Generally left open by default and only used to forcefully block specific malicious IP ranges.

VPC Peering & Transit Gateway

By default, two VPCs cannot communicate. If you have a "Production VPC" and a "Shared Services/Jenkins VPC", you connect them using VPC Peering. This uses the private AWS backbone (not the public internet) to link the networks, provided their CIDR blocks do not overlap. For complex architectures with 50+ VPCs, you use an AWS Transit Gateway to manage routing centrally like a hub-and-spoke model.
