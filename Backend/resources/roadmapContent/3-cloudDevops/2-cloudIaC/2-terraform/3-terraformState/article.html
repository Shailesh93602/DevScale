Terraform State Management

Terraform keeps track of the infrastructure it creates using a massive JSON file called the State File (terraform.tfstate). When you run terraform apply, it compares your local code against the State File to calculate the diff. Without the state, Terraform has no idea what it has built.

The Danger of Local State

By default, Terraform saves the state file locally on your laptop. This is extremely dangerous for teams.

  1. Collaboration: If you and a coworker run terraform apply from your local laptops, your local state files diverge.
  2. Secrets Exposure: The state file stores database passwords and AWS keys in plain text. Committing terraform.tfstate to Git is a massive security breach.

Remote State Backends

To fix this, teams configure a Backend. A backend tells Terraform where to store the state file remotely. The most common enterprise backend is AWS S3 + DynamoDB.

# Adding this block instructs Terraform to never store state locally
terraform {
  backend "s3" {
    bucket         = "my-company-terraform-states"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}

State Locking

What happens if your CI/CD pipeline runs `terraform apply`, and at the exact same second, your colleague runs `terraform apply`? Terraform could corrupt the state file, destroying your cloud.

That's what the DynamoDB table is for. When Terraform starts executing, it writes a short-lived "lock" item to DynamoDB. If a second run attempts to start, it sees the lock, fails immediately, and waits until the first run finishes.

Configuration Drift

If someone logs into the AWS Web Console manually and changes an EC2 instance size from t2.micro to t3.large, the cloud no longer matches the Terraform Code. This is called Drift.

On the next terraform plan, Terraform will detect this (by pinging the real AWS API and comparing it against the state). During terraform apply, it will forcibly revert the instance back to t2.micro to enforce the Infrastructure-as-Code state.

Workspaces

If you want to use the exact same code to deploy to Dev and Prod without copying folders, you use Workspaces. Workspaces allow a single codebase to maintain multiple separate remote state files.

# Create a new workspace
terraform workspace new production

# Switch to development
terraform workspace select dev

# In your code, you can reference the workspace name!
resource "aws_instance" "web" {
  tags = {
    Name = "Web-Server-${terraform.workspace}"
  }
}
