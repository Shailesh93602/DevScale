Kubernetes Networking (Services)

Pods are mortal. They die, they are scaled up, they are recreated across different nodes. Because of this, Pod IP addresses are entirely ephemeral (temporary). If a Frontend application hardcodes the IP address of a Backend Pod, the entire app breaks the moment that Backend Pod is restarted.

To solve this, Kubernetes uses Services.

What is a Service?

A Service provides a stable, permanent IP address and DNS name that routes traffic to a set of underlying Pods. It acts as an internal Load Balancer.

Types of Services

  1. ClusterIP (Default): Provides an internal IP accessible only from inside the Kubernetes cluster. Use this for backend systems (like a database or an internal API) that should never be directly accessible from the internet.
  2. NodePort: Opens a specific port (between 30000-32767) on every single physical Server Node in the cluster, forwarding traffic to the underlying pods. It is primarily used for primitive external access or local development (minikube).
  3. LoadBalancer: Used in cloud environments (AWS, GCP, Azure). When you create this service, Kubernetes tells AWS to automatically provision a real AWS Application Load Balancer, assigns it a public IP, and links it to your cluster. This is the standard way to expose an app to the public internet.

How Services find Pods

Services use Label Selectors to know which Pods to route traffic to.

apiVersion: v1
kind: Service
metadata:
  name: my-backend-service
spec:
  type: ClusterIP
  selector:
    app: backend-api  # Routes traffic to ANY pod with this exact label
  ports:
    - port: 80        # The port the Service listens on
      targetPort: 8080 # The exact port the container inside the Pod is running on

Kubernetes automatically updates the internal DNS. Now, your frontend container can simply make an HTTP request to http://my-backend-service:80, and Kubernetes will handle dynamically routing it to an alive backend pod.

Ingress Controllers

If you have 50 microservices and use a LoadBalancer service for each, AWS will bill you for 50 expensive Load Balancers. An Ingress solves this.

An Ingress is a single entry-point Load Balancer that routes traffic based on URL paths. Example rule:
- /api/* routes to the api-service (ClusterIP)
- /auth/* routes to the auth-service (ClusterIP)
- /* routes to the frontend-service (ClusterIP)
