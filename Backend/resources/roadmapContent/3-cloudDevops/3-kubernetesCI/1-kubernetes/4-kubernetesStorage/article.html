Kubernetes Storage: PVs, PVCs & StorageClasses

Kubernetes pods are ephemeral by design — when a pod is deleted or crashes, all data written to its container filesystem is permanently lost. For stateful applications like databases (PostgreSQL, MySQL, Redis), or any application that needs to persist data beyond a pod's lifecycle, Kubernetes provides a decoupled storage model: Persistent Volumes (PV) and Persistent Volume Claims (PVC).

The PV / PVC Abstraction

  • Persistent Volume (PV): A piece of physical storage in the cluster — could be an AWS EBS volume, an Azure Disk, an NFS share, or a local SSD. Provisioned by a cluster administrator (or automatically by a StorageClass). PVs have a lifecycle independent of any pod.
  • Persistent Volume Claim (PVC): A request for storage by a user or application. The developer says "I need 20Gi of ReadWriteOnce storage" — they don't need to know where or how the storage is physically provisioned.
# pvc.yaml — The application requests storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce      # Can be mounted by exactly ONE pod at a time
  storageClassName: gp3  # References a StorageClass for dynamic provisioning
  resources:
    requests:
      storage: 50Gi
---
# deployment.yaml — Mount the PVC into the PostgreSQL pod
spec:
  containers:
  - name: postgres
    image: postgres:16
    volumeMounts:
    - mountPath: /var/lib/postgresql/data
      name: postgres-storage
  volumes:
  - name: postgres-storage
    persistentVolumeClaim:
      claimName: postgres-data

StorageClasses: Dynamic Provisioning

In the past, a cluster admin had to manually pre-create EBS volumes and register them as PVs. StorageClasses enable dynamic provisioning — when a PVC is created, Kubernetes automatically provisions the underlying storage (e.g., creates an EBS volume in AWS) and binds it to the PVC:

# StorageClass for AWS EKS — auto-creates EBS gp3 volumes on demand
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer  # Don't create until pod is scheduled
parameters:
  type: gp3
  encrypted: "true"

Access Modes

ReadWriteOnce (RWO): Mounted read-write by ONE node only — AWS EBS, Azure Disk
ReadOnlyMany  (ROX): Mounted read-only by MANY nodes
ReadWriteMany (RWX): Mounted read-write by MANY nodes — AWS EFS (NFS), Ceph

StatefulSets for Ordered, Stable Deployments

Regular Deployments give pods random names (e.g., postgres-65d4b8c9f-x2kqp). For databases and clustered apps that need stable, predictable hostnames and ordered deployment, use a StatefulSet. StatefulSets give pods stable names like postgres-0, postgres-1, postgres-2, with each pod having its own dedicated PVC that persists across pod restarts.
