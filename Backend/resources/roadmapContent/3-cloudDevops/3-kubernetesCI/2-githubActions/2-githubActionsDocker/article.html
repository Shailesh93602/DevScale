GitHub Actions & Docker

Modern applications are packaged into Docker Containers before deployment. The process of building, tagging, testing, and pushing the Docker image to a registry should not be done manually; it should be 100% automated via CI/CD. GitHub Actions is a powerful tool for this.

The Goal of the CI Pipeline

When a developer opens a Pull Request to the main branch, GitHub Actions spins up a temporary runner server in the cloud, clones your code, and runs your tests. If tests fail, the PR is blocked.

When the PR is merged into main, another Action builds your Docker image, tags it with a unique identifier (like the Git commit hash), and pushes it to a secure repository (like Docker Hub, Amazon ECR, or GitHub Container Registry).

Building and Pushing a Docker Image

Here is a basic GitHub action workflow file (saved in `.github/workflows/docker-publish.yml`):

name: Build and Push Docker Image

on:
  push:
    branches: [ "main" ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - name: Checkout Code
      uses: actions/checkout@v3

    # Step 1: Login to DockerHub securely
    - name: Login to DockerHub
      uses: docker/login-action@v2
      with:
        username: ${{ secrets.DOCKER_USERNAME }}
        password: ${{ secrets.DOCKER_PASSWORD }}

    # Step 2: Build the image and push it to the registry
    - name: Build and Push
      uses: docker/build-push-action@v4
      with:
        context: .
        push: true
        tags: |
          myworkspace/my-api:${{ github.sha }}
          myworkspace/my-api:latest

Image Tagging Best Practices

Notice the tags: array. Never rely solely on the latest tag in production deployments!

  • If your Kubernetes Deployment is instructed to pull my-api:latest, and a pod crashes and restarts, it will aggressively pull whatever image happens to currently hold the latest tag at that exact moment. If an intern pushed buggy code 5 minutes prior, your production cluster will instantly download the bug and crash.
  • By tagging the image with the specific Git Commit Hash (${{ github.sha }}), you guarantee that the image version is permanently immutable. Your deployment file should reference `my-api:abcdef123` specifically.

Image Caching in Actions

Downloading all dependencies (like running npm install inside the Dockerfile) is slow. If you don't configure caching, your 2-minute build could take 10 minutes every single time you push a typo fix. GitHub Actions provides caching actions (actions/cache or Docker layer caching) to reuse layers from previous builds, slashing CI pipeline times.
