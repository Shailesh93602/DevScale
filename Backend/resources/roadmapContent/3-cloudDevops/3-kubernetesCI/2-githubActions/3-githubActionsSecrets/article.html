GitHub Actions: Secrets, Environments & OIDC Security

CI/CD pipelines require incredibly privileged access to your infrastructure. To deploy a Docker container, your pipeline needs the AWS Access Keys for ECR. To connect to the database for migrations, it needs the production database password. Managing these credentials securely is the most critical aspect of DevSecOps. Hardcoding a password in your repository or exposing an API key in the build logs is an immediate, catastrophic security breach.

GitHub Secrets Management

GitHub provides an encrypted key-value store called Secrets. When you create a secret (e.g., DB_PASSWORD) in the repository settings, GitHub encrypts it immediately. It can never be viewed again, even by repository administrators — it can only be updated or deleted, and it is only decrypted securely during the exact moment a GitHub Actions runner needs it.

To use a secret in your workflow, you reference it using the ${{ secrets.SECRET_NAME }} context:

name: Deploy to Production
on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Run Database Migration
        env:
          # Inject secrets as environment variables for the script
          DB_HOST: ${{ secrets.PROD_DB_HOST }}
          DB_PASS: ${{ secrets.PROD_DB_PASSWORD }}
        run: npm run db:migrate

Log Redaction: If your script accidentally echo $DB_PASS, GitHub automatically detects the secret value in the output and masks it with asterisks (***) in the public build logs.

GitHub Environments & Approval Gates

If you have a PROD_DB_PASSWORD secret at the repository level, any developer who can trigger a workflow (even from a feature branch) can theoretically write a script to extract that string or execute malicious commands against production. This violates the principle of least privilege.

GitHub Environments solve this by scoping secrets to specific deployment environments (e.g., staging and production). Instead of adding secrets to the repository, you add them to the production environment.

When a workflow attempts to use an Environment, you can configure Protection Rules:

  • Required Reviewers: A senior engineer must manually click "Approve" before the deployment job executes.
  • Wait Timers: Delay the job for 30 minutes to allow time to cancel if an issue is discovered.
  • Branch Restrictions: Only allow the main branch or release/* tags to deploy to the production environment. An attacker pushing to a feature/hack branch physically cannot access the production secrets.
jobs:
  deploy_prod:
    runs-on: ubuntu-latest
    environment: 
      name: production           # This triggers the approval gate
      url: https://myapp.com     # Adds a handy link to the GitHub UI
    steps:
      - run: deploy.sh
        env:
          # Safely pulls the password scoped only to the 'production' environment
          DB_PASS: ${{ secrets.DB_PASSWORD }}

The Modern Standard: OIDC (OpenID Connect)

While GitHub Secrets are secure, storing long-lived AWS Access Keys or GCP JSON Service Accounts in GitHub is fundamentally dangerous. If an employee with admin access leaves the company but takes the AWS Access Key stored in GitHub, they retain access to your cloud indefinitely until someone remembers to rotate the key.

OpenID Connect (OIDC) is a game-changing security feature that eliminates long-lived credentials entirely. Instead of giving GitHub an AWS Access Key, you establish a direct trust relationship between your AWS Account and your GitHub Repository.

How OIDC works during a CI/CD run:

  1. The GitHub Action runner generates a temporary JWT (JSON Web Token) containing claims like: "repository": "my-org/my-app", "ref": "refs/heads/main".
  2. The runner sends this token to AWS IAM's AssumeRoleWithWebIdentity endpoint.
  3. AWS verifies the token's cryptographic signature via OIDC and inspects the claims. It checks: "Is this token really from GitHub? Is it really from the 'my-org/my-app' repo? Is the branch 'main'?"
  4. If the claims match the IAM Trust Policy, AWS grants the runner a temporary, short-lived session token (valid for exactly 1 hour, or just the duration of the build).
# The most secure way to authenticate with AWS in CI/CD
permissions:
  id-token: write   # Required to request the OIDC JWT token
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS Credentials using OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          # Zero secrets involved! Just providing the IAM Role ARN
          role-to-assume: arn:aws:iam::1234567890:role/github-actions-deploy-role
          aws-region: us-east-1

By using OIDC, there is no key to rotate, no string to leak, and no long-lived access. It is the absolute highest standard for DevSecOps pipelines.
