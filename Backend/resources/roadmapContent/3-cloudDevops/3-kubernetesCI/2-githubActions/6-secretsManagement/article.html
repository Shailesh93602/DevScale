Secrets Management in CI/CD

Secrets — database passwords, API keys, TLS certificates, SSH private keys — are the most sensitive assets in any software system. Yet they are routinely mishandled: committed to Git, stored in environment variables visible in CI logs, hardcoded in Dockerfiles, or emailed between engineers. A single leaked secret can result in data breaches costing millions. Proper secrets management is non-negotiable in production engineering.

The Secret Anti-Patterns to Avoid

  • ❌ Committing .env files with real credentials to Git — even "private" repos get breached
  • ❌ Storing secrets as plain text GitHub Actions environment variables
  • ❌ Printing secrets in CI logs (echo $DB_PASSWORD is logged forever)
  • ❌ Baking secrets into Docker images with ENV DB_PASSWORD=...
  • ❌ Long-lived static credentials that never rotate

GitHub Actions: Encrypted Secrets

jobs:
  deploy:
    steps:
    # Secrets are injected as environment variables — never appear in logs
    - name: Deploy to Production
      env:
        DB_PASSWORD: ${{ secrets.PRODUCTION_DB_PASSWORD }}
        API_KEY: ${{ secrets.STRIPE_API_KEY }}
      run: |
        ./scripts/deploy.sh
        # GitHub automatically masks secrets values in log output

Kubernetes Secrets & External Secrets Operator

Kubernetes has a built-in Secret resource, but base64 encoding is not encryption — it's trivially reversible. The recommended pattern is to store actual secret values in AWS Secrets Manager or HashiCorp Vault and sync them into Kubernetes Secrets automatically using the External Secrets Operator (ESO):

# ExternalSecret — syncs AWS Secrets Manager → Kubernetes Secret
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: production-db-credentials
  namespace: production
spec:
  refreshInterval: 1h      # Re-sync every hour (picks up rotated secrets)
  secretStoreRef:
    name: aws-secrets-store
    kind: SecretStore
  target:
    name: db-credentials   # Creates this Kubernetes Secret
  data:
  - secretKey: password    # Key in the resulting K8s Secret
    remoteRef:
      key: production/database    # Path in AWS Secrets Manager
      property: password          # Field within the JSON secret

HashiCorp Vault: Dynamic Secrets

The most advanced secrets management pattern is dynamic secrets — credentials that Vault generates on-demand, with a short TTL, for a specific application identity:

import hvac

# Vault generates a unique, short-lived PostgreSQL credential for this app
client = hvac.Client(url='https://vault.internal')
client.auth.kubernetes.login(role='my-api', jwt=service_account_token)

# Dynamic credential — valid for 1 hour, unique to this pod instance
creds = client.secrets.database.generate_credentials(name='production-postgres')
db_password = creds['data']['password']
# → 'a3f2k9x1p8...' (unique, expires in 1 hour, auto-rotated)

When the pod dies, the credential automatically expires. If the credential is leaked, it becomes useless within an hour — no manual rotation needed, no stale credentials persisting indefinitely.
