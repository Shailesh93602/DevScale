AWS CloudWatch & Log Management

Amazon CloudWatch is the unified monitoring, logging, and performance management service native to AWS. Instead of spinning up third-party clusters (like ELK/Elasticsearch or Datadog), CloudWatch works out-of-the-box for practically every managed service in AWS without requiring custom agents or heavy configuration overhead. It is the central nervous system for cloud infrastructure.

CloudWatch Logs Architecture

Logs in CloudWatch are organized hierarchically:

  • Log Group: The top-level container, representing a logical application or service (e.g., /aws/lambda/UserAuthFunction or /var/log/nginx/access). IAM permissions, data retention policies (e.g., delete logs older than 30 days), and encryption KMS keys are defined at the Log Group level.
  • Log Stream: Inside a Log Group, a Stream represents a specific instance of the service over time. If your Lambda function spins up 5 concurrent containers to handle a traffic spike, CloudWatch creates 5 separate Log Streams inside the Lambda's Log Group to isolate their output.
  • Log Event: The individual time-stamped message, such as an error stack trace or a JSON log line.

Ingesting Logs via the CloudWatch Agent

Managed services like Lambda, ECS (Fargate), and API Gateway send logs to CloudWatch automatically. However, standard EC2 instances running custom web servers (like Apache or Node.js) do not.

To collect OS-level files (like /var/log/syslog or your application's custom file output), you must install and configure the CloudWatch Agent on the EC2 instance via SSH or User Data scripts.

// Example CloudWatch Agent JSON config (/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json)
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/nginx/access.log",
            "log_group_name": "production-nginx-access",
            "log_stream_name": "{instance_id}",
            "timezone": "UTC"
          }
        ]
      }
    }
  }
}

Note: Ensure the EC2 instance has an IAM Role attached with the CloudWatchAgentServerPolicy permission so it can securely push logs to the AWS API.

Metric Filters & Alarms

CloudWatch allows you to continuously extract actionable metrics directly from unstructured text logs using Metric Filters. If an incoming log line matches a specific pattern, CloudWatch increments a custom metric counter.

For example, you can create a filter matching the string "ERROR" or "Timeout" inside the production-nginx-error Log Group. Once the filter generates the custom metric, you attach a CloudWatch Alarm: "If the ErrorCount metric exceeds 5 in a 1-minute period, trigger an SNS Topic to page the on-call engineer."

CloudWatch Logs Insights

Scrolling endlessly through thousands of Log Streams looking for a specific UUID is impossible. CloudWatch Logs Insights provides an interactive SQL-like query language designed specifically to analyze massive volumes of structured JSON logs across multiple Log Groups simultaneously.

# Scenario: Find the Top 10 IP addresses causing HTTP 500 errors today across all Nginx servers
fields @timestamp, @message, status_code, client_ip
| filter status_code >= 500
| stats count(*) as ErrorCount by client_ip
| sort ErrorCount desc
| limit 10

# Scenario: Calculate the 95th Percentile API Response Latency (Requires JSON structured logging)
fields @timestamp, request_duration_ms, endpoint
| filter endpoint = '/api/checkout/process'
| stats pct(request_duration_ms, 95) as p95_latency by bin(1h)

Cost Optimization & Exporting to S3

While CloudWatch is convenient, it is extremely expensive to store Terabytes of data long-term compared to Amazon S3. For compliance purposes (like holding PCI data logs for 7 years), storing the data strictly in CloudWatch Logs is cost-prohibitive.

The standard architectural pattern is Log Archival via Subscription Filters:

  1. Configure the Log Group's retention policy strictly (e.g., delete automatically after 14 days) to save money on hot SSD storage.
  2. Create a Subscription Filter that continuously streams the real-time logs directly to an Amazon Kinesis Data Firehose.
  3. The Firehose automatically batches, severely compresses, and dumps the logs into cheaper Amazon S3 Standard-Infrequent Access storage for multi-year retention.
  4. If an auditor requests 3-year-old logs, you simply query the cheap S3 bucket using Amazon Athena instead of paying CloudWatch storage rates.
