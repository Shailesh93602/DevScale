SAST, DAST & Penetration Testing

Traditional security testing happened after development was complete — a penetration tester would attack the finished application before release. Modern DevSecOps "shifts security left," embedding automated security tests at every stage of the SDLC so vulnerabilities are caught at the exact moment they are introduced, when they are cheapest to fix.

SAST: Static Application Security Testing

SAST analyzes source code (without executing it) to find security vulnerabilities using pattern matching and data flow analysis. It runs in the CI pipeline on every pull request:

# Semgrep: fast, rule-based SAST for many languages
semgrep --config=p/owasp-top-ten ./src/

# Common SAST findings:
# - SQL Injection: raw string interpolation in DB queries
#   BAD:  f"SELECT * FROM users WHERE id = {user_id}"
#   GOOD: db.execute("SELECT * FROM users WHERE id = ?", [user_id])
#
# - Hardcoded secrets: API_KEY = "sk-abc123..." in source code
# - Insecure deserialization: pickle.loads(user_input)
# - Path traversal: open(f"/uploads/{filename}")  (filename could be "../../etc/passwd")
# - Missing authentication on endpoint handlers

# In GitHub Actions CI — block PRs with high-severity SAST findings
- name: Run Semgrep
  uses: semgrep/semgrep-action@v1
  with:
    config: p/owasp-top-ten
    publishToken: ${{ secrets.SEMGREP_APP_TOKEN }}

DAST: Dynamic Application Security Testing

DAST tests a live, running application from the outside — exactly like an attacker would. It fires crafted HTTP requests to discover vulnerabilities that only manifest at runtime:

# OWASP ZAP — industry standard open source DAST tool
# Run a baseline scan against the staging environment
docker run --rm owasp/zap2docker-stable zap-baseline.py \
  -t https://staging.myapp.com \
  -r zap-report.html

# DAST tests for:
# - XSS: injects  into every form field and URL parameter
# - SQL Injection: appends ' OR 1=1-- to all inputs
# - CSRF: checks for missing CSRF tokens on state-changing forms
# - Broken authentication: tests for session fixation, cookie security flags
# - Security headers: checks for missing HSTS, CSP, X-Frame-Options
# - Exposed sensitive paths: /admin, /.env, /backup.sql, /phpinfo.php

Penetration Testing

While SAST and DAST are automated, penetration testing (pen testing) involves skilled human security engineers actively trying to breach your systems using the same techniques real attackers use. A pen test goes far beyond automated scans:

  • Reconnaissance: Discovering subdomains, open ports, technology fingerprinting, employee social engineering vectors
  • Exploitation: Chaining multiple low-severity issues into high-impact attack paths (e.g., SSRF → IAM metadata service → AWS credential theft)
  • Privilege Escalation: Starting from a low-privilege position and escalating to admin/root using misconfigurations
  • Reporting: Detailed reproduction steps and prioritized remediation recommendations

Best practice: Run automated SAST/DAST continuously in CI/CD, and commission a professional pen test annually (or before a major product launch or compliance certification).
