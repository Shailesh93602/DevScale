CIS Benchmarks: Hardening Infrastructure at Scale

When you provision a fresh EC2 instance, create a new AWS account, or bootstrap a Kubernetes cluster, the default configuration is optimized for convenience, not security. Root SSH is enabled, all traffic is allowed, audit logging is disabled, and passwords have no complexity requirements. The Center for Internet Security (CIS) is a non-profit that publishes consensus-based, peer-reviewed checklists defining exactly how to harden these systems to the standard used by banks, militaries, and major enterprises.

The Two Hardening Levels

  • Level 1 (Practical Security): Recommendations that deliver meaningful security improvements with minimal disruption to normal application functionality. Every production system should implement Level 1.
  • Level 2 (Defense in Depth): Extremely aggressive configurations intended for high-security environments (defense, government, healthcare). Level 2 implementations will break standard applications — test thoroughly before applying.

CIS AWS Foundations Benchmark — Key Controls

The AWS benchmark defines controls across IAM, logging, networking, and monitoring. Here's how to automate compliance with Terraform:

## terraform/aws_cis_foundations.tf

## CIS 1.2 — Ensure MFA is enabled for the root account (must be done manually)
## CIS 1.5 — IAM password policy requirements
resource "aws_iam_account_password_policy" "cis_compliant" {
  minimum_password_length        = 14
  require_uppercase_characters   = true
  require_lowercase_characters   = true
  require_numbers                = true
  require_symbols                = true
  allow_users_to_change_password = true
  max_password_age               = 90   # Force rotation every 90 days
  password_reuse_prevention      = 24   # Cannot reuse last 24 passwords
}

## CIS 2.1 — CloudTrail enabled in all regions with encryption
resource "aws_cloudtrail" "cis_trail" {
  name                          = "cis-management-trail"
  s3_bucket_name                = aws_s3_bucket.cloudtrail_logs.id
  include_global_service_events = true
  is_multi_region_trail         = true  # Required: must cover all regions
  enable_log_file_validation    = true  # Detects tampered log files
  kms_key_id                    = aws_kms_key.cloudtrail.arn  # Encryption at rest
}

## CIS 4.1 — No Security Group allows SSH from 0.0.0.0/0
resource "aws_security_group_rule" "deny_all_inbound" {
  type        = "ingress"
  from_port   = 22
  to_port     = 22
  protocol    = "tcp"
  cidr_blocks = ["10.0.0.0/8"]  # Only VPN/corporate CIDR, never 0.0.0.0/0
  security_group_id = aws_security_group.bastion.id
}

AWS Security Hub — Continuous CIS Monitoring

## Enable AWS Security Hub with CIS standard via Terraform
resource "aws_securityhub_account" "main" {}

resource "aws_securityhub_standards_subscription" "cis" {
  depends_on    = [aws_securityhub_account.main]
  standards_arn = "arn:aws:securityhub:::ruleset/cis-aws-foundations-benchmark/v/1.2.0"
}

CIS Ubuntu Linux Benchmark — Key Controls

#!/bin/bash
# CIS Ubuntu 22.04 Level 1 hardening script

## CIS 1.6.1 — Restrict core dumps (prevent memory dumping of sensitive data)
echo "hard core 0" >> /etc/security/limits.conf
echo "fs.suid_dumpable = 0" >> /etc/sysctl.d/60-cis.conf

## CIS 3.1.1 — Disable IPv6 if not used
echo "net.ipv6.conf.all.disable_ipv6 = 1" >> /etc/sysctl.d/60-cis.conf

## CIS 3.3.2 — Disable source routing (prevents IP spoofing attacks)
echo "net.ipv4.conf.all.accept_source_route = 0" >> /etc/sysctl.d/60-cis.conf

## CIS 5.2 — SSH Hardening
sed -i 's/#PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
echo "Protocol 2" >> /etc/ssh/sshd_config
echo "MaxAuthTries 3" >> /etc/ssh/sshd_config

sysctl -p /etc/sysctl.d/60-cis.conf
systemctl restart ssh

Best Practices

  • Use Terraform to provision CIS-compliant infrastructure from day zero — retrofitting security controls onto existing infrastructure is error-prone and time-consuming.
  • Run kube-bench and AWS Security Hub reports in CI — treat compliance failures like test failures that block deployments.
  • Start with Level 1 benchmarks; only apply Level 2 for systems processing regulated data (PCI-DSS, HIPAA) after thorough testing in a staging environment.
  • Subscribe to CVE databases and CIS Benchmark update notifications — benchmarks are revised as new threats emerge.
