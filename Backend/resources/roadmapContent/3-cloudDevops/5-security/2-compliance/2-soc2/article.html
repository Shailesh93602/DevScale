SOC 2 Compliance for Cloud Infrastructure

If you build a SaaS product that stores sensitive customer data, enterprise customers — banks, hospitals, law firms — will demand a SOC 2 Type II Report before signing a contract. SOC 2 (System and Organization Controls 2) is an auditing standard developed by the AICPA. A licensed third-party auditor evaluates your systems over a period of months (typically 6–12) and certifies that your security controls are not just documented on paper, but actively enforced and working. Without SOC 2, many enterprise sales cycles simply cannot close.

The Five Trust Services Criteria

CriteriaRequired?What It Verifies
SecurityMandatorySystem protected against unauthorized access
AvailabilityOptionalSystem available per committed SLAs
Processing IntegrityOptionalProcessing is complete, valid, and timely
ConfidentialityOptionalConfidential data is protected as agreed
PrivacyOptionalPII handled per the privacy notice

Type I vs Type II

  • SOC 2 Type I: A point-in-time snapshot. The auditor verifies that your controls are correctly designed at one specific date. Faster to obtain (1–3 months), but less trusted by enterprise buyers.
  • SOC 2 Type II: Evaluates control effectiveness over a period of time (typically 6–12 months). Enterprises require Type II because it proves controls are consistently enforced, not just set up for the audit date.

Key DevOps Controls the Auditor Evaluates

1. Access Control (Least Privilege)

The auditor will ask: "Who has access to production databases, and how is that access granted and revoked?" Evidence required: IAM role policies, access logs, user provisioning/deprovisioning records.

## Terraform — Enforce least privilege IAM for a backend service
## This Lambda function can only read from the specific DynamoDB table it needs
resource "aws_iam_role_policy" "order_processor_policy" {
  name = "order-processor-prod-policy"
  role = aws_iam_role.order_processor.id
  
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["dynamodb:GetItem", "dynamodb:Query"]
        # Scope to specific table and specific partition key prefix
        Resource = "${aws_dynamodb_table.orders.arn}/index/*"
        Condition = {
          StringLike = {
            "dynamodb:LeadingKeys" = ["ORDER#*"]
          }
        }
      }
    ]
  })
}

2. Comprehensive Audit Logging

The auditor will ask: "If an unauthorized access occurred at 3 AM last Tuesday, can you show me exactly who logged in, from what IP, and what they accessed?" Evidence: CloudTrail logs, VPC Flow Logs, application access logs.

## Terraform — CloudWatch alarms that prove active monitoring
resource "aws_cloudwatch_metric_alarm" "unauthorized_api_calls" {
  alarm_name          = "CIS-UnauthorizedAPICalls"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = "1"
  metric_name         = "UnauthorizedAttemptCount"
  namespace           = "CloudTrailMetrics"
  period              = "300"
  threshold           = "1"
  alarm_actions       = [aws_sns_topic.security_alerts.arn]
  
  # This alarm fires and pages the security team immediately if
  # anyone makes unauthorized API calls — proves active monitoring
}

3. Change Management (Branch Protection)

The auditor will ask: "Can a single engineer push unreviewed code to production at 2 AM without anyone knowing?" Required evidence: GitHub branch protection rules, required PR reviews, CI/CD pipeline logs showing tests passed before merge.

## .github/branch-protection.json (enforced via Terraform GitHub provider)
resource "github_branch_protection" "main" {
  repository_id = github_repository.app.node_id
  pattern       = "main"

  require_conversation_resolution = true
  required_pull_request_reviews {
    required_approving_review_count = 2     # Two reviewers required
    dismiss_stale_reviews           = true  # Auto-dismiss on new commits
    require_code_owner_reviews      = true  # CODEOWNERS must approve
  }
  required_status_checks {
    strict   = true                          # Branch must be up-to-date
    contexts = ["ci/tests", "ci/security"]   # ALL these checks must pass
  }
  enforce_admins = true  # CRITICAL: Admins cannot bypass protection rules
}

4. Vendor Risk Management

SOC 2 also covers your supply chain. The auditor will ask about third-party services that process your customer data (Stripe, SendGrid, AWS itself). Each must have their own SOC 2 report or equivalent, which you collect and review annually. This is called "sub-service organization management" in the audit report.

Compliance Automation Tools

Companies like Vanta, Drata, and Sprinto automate SOC 2 evidence collection by integrating with your AWS, GitHub, Jira, and HR systems. They continuously monitor your controls and auto-generate the evidence packages auditors need — dramatically reducing the engineering time required for a SOC 2 audit from months to weeks.

Best Practices

  • Start your SOC 2 Type I the moment your first enterprise customer asks for it — it takes 3+ months minimum from kickoff to report, and you'll lose deals while waiting.
  • Use a compliance automation tool (Vanta, Drata) from the start — manual evidence collection across 100+ controls is prohibitively time-consuming without tooling.
  • Enforce branch protection and least-privilege IAM from day one — retrofitting these controls onto an existing team's workflow triggers significant organizational resistance.
  • Conduct internal penetration testing at least annually and remediate all critical/high findings before the auditor's review period ends.
