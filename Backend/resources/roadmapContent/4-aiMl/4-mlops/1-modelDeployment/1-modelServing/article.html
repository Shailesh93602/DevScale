Model Serving Strategies

Training a machine learning model is only half the job. The other half is making it available so that real applications can use it to get predictions. Model serving is the infrastructure and process of exposing a trained model through an API or service endpoint, allowing upstream applications to send input data and receive predictions.

The Two Core Serving Modes

The fundamental design choice in model serving determines your latency profile, infrastructure cost, and user experience.

1. Real-Time (Online) Inference

A client sends a single request and waits for an immediate response. This is the standard REST/gRPC API model. Use it when a human or interactive process is waiting for the result.

# FastAPI real-time inference endpoint
from fastapi import FastAPI
from pydantic import BaseModel
import joblib
import numpy as np

app = FastAPI()
model = joblib.load("model/fraud_detection.pkl")
scaler = joblib.load("model/scaler.pkl")

class Transaction(BaseModel):
    amount: float
    hour_of_day: int
    merchant_category: int
    distance_from_home_km: float

@app.post("/predict/fraud")
def predict(tx: Transaction):
    features = np.array([[
        tx.amount, tx.hour_of_day,
        tx.merchant_category, tx.distance_from_home_km
    ]])
    scaled = scaler.transform(features)
    probability = model.predict_proba(scaled)[0][1]
    return {
        "fraud_probability": round(float(probability), 4),
        "is_fraud": probability > 0.5,
        "model_version": "v2.1.0"
    }

2. Batch (Offline) Inference

You run predictions on a large dataset all at once and store the results. The key insight is that no human is waiting — the results will be read later. This is common for nightly recommendation refreshes, credit risk scoring on a loan portfolio, or personalized email content generation.

# Batch inference with PySpark
from pyspark.sql import SparkSession
from pyspark.ml import PipelineModel

spark = SparkSession.builder.appName("BatchScoring").getOrCreate()
model = PipelineModel.load("s3://models/churn/v3")

# Load all customers updated in the last 7 days
customers = spark.read.parquet("s3://data/customers/daily/")

# Run predictions on the entire dataset in parallel
predictions = model.transform(customers)

# Write results back to the data warehouse
predictions.select("customer_id", "churn_probability", "risk_segment") \
    .write.mode("overwrite") \
    .parquet("s3://data/churn_scores/latest/")

Dedicated Serving Frameworks

Raw FastAPI works, but production teams use specialized serving frameworks that add model versioning, A/B traffic splitting, hardware acceleration, and automatic batching.

  • TensorFlow Serving (TFServing): Google's battle-tested gRPC server for TensorFlow SavedModel format. Handles multiple model versions simultaneously and can hot-swap a new version without downtime.
  • TorchServe: The equivalent from PyTorch/AWS for serving PyTorch models with custom handlers.
  • Triton Inference Server (NVIDIA): The most performant option. Can serve TensorFlow, PyTorch, ONNX, and TensorRT models on GPU, with automatic dynamic batching to maximize GPU utilization.
  • BentoML: A Python-native framework that lets you define the entire serving logic (pre-processing, model call, post-processing) as code and packages it as a Docker container automatically.

Model Formats for Efficient Serving

The format you save your model in directly impacts inference speed and cross-platform compatibility:

  • ONNX (Open Neural Network Exchange): An open standard format that converts models from any framework (PyTorch, TensorFlow, scikit-learn) into a unified graph. Runtimes like ONNX Runtime or TensorRT can then execute it 2–5x faster than the original framework.
  • TorchScript: Serializes PyTorch models into a standalone format that can be loaded in C++ without any Python dependency — critical for edge deployment and mobile apps.
  • TensorRT: NVIDIA's compiler and runtime that quantizes and fuses operations in a neural network specifically for a target GPU, often achieving 3–10x speedup over the original model.

Scaling Serving Infrastructure

When a single model server can't handle the load, you scale out:

  • Horizontal Scaling: Run multiple identical instances of the model server behind a load balancer (e.g., NGINX or an AWS ALB). Kubernetes makes this trivial with a Deployment and HorizontalPodAutoscaler.
  • GPU Batching: GPUs are most efficient when processing multiple inputs simultaneously. Servers like Triton will wait up to a configurable window (e.g., 5ms) to accumulate multiple incoming requests and batch them through the GPU in one shot, dramatically increasing throughput.
  • Caching: If the same input arrives repeatedly (e.g., same product ID requested for recommendations), cache the result in Redis to avoid redundant inference.

Monitoring a Serving Endpoint

A production model server must track both infrastructure and ML-specific metrics:

  • Infrastructure: Latency (p50, p95, p99), throughput (requests/sec), error rate, GPU/CPU utilization.
  • ML-specific: Prediction distribution drift (are the model's outputs changing over time?), input feature drift (is the incoming data different from training data?), and business KPI impact (did this recommendation model actually increase clicks?).

Best Practices

  • Always version your models and endpoints (e.g., /v1/predict, /v2/predict) so clients can migrate gradually.
  • Use shadow mode (deploy new model alongside old one, mirror 5% of traffic) to validate a new model before cutting over.
  • Set a request timeout on all inference endpoints. A slow model prediction should fail fast, not cascade into a timeout for the entire user request chain.
  • Pre-load models into memory at startup, not on the first request — the first request should never pay the model loading penalty.
