Weights & Biases (W&B): ML Experiment Tracking at Scale

While MLflow is excellent for self-hosted tracking, Weights & Biases (W&B) is the cloud-hosted experiment platform used by teams at OpenAI, Hugging Face, and Samsung. It offers a richer, more collaborative experience — especially for deep learning workloads — with real-time metric streaming, interactive dashboards, and built-in hyperparameter optimization.

Basic Integration

import wandb
import torch
import torch.nn as nn

# Initialize a W&B run — this creates a run on your W&B dashboard
run = wandb.init(
    project="image_classification",
    name="resnet50-lr0.001-batch64",
    config={
        "learning_rate": 0.001,
        "batch_size": 64,
        "epochs": 30,
        "optimizer": "AdamW",
        "architecture": "ResNet50"
    }
)

# Access config safely from wandb.config
lr = wandb.config.learning_rate

model = resnet50().to(device)
optimizer = torch.optim.AdamW(model.parameters(), lr=lr)

for epoch in range(wandb.config.epochs):
    train_loss = train_one_epoch(model, train_loader, optimizer)
    val_loss, val_acc = evaluate(model, val_loader)

    # Log metrics — they stream to the dashboard in real-time
    wandb.log({
        "epoch": epoch,
        "train_loss": train_loss,
        "val_loss": val_loss,
        "val_accuracy": val_acc,
        "learning_rate": optimizer.param_groups[0]["lr"]
    })

# Save the final model as a W&B Artifact
artifact = wandb.Artifact("resnet50_classifier", type="model")
torch.save(model.state_dict(), "model.pt")
artifact.add_file("model.pt")
run.log_artifact(artifact)
run.finish()

W&B Sweeps — Automated Hyperparameter Optimization

A Sweep automates the search over your hyperparameter space. You define the search space and strategy, and W&B manages launching and tracking all the runs.

# sweep.yaml — define the search configuration
sweep_config = {
    "method": "bayes",  # Bayesian optimization (smarter than grid search)
    "metric": {"name": "val_accuracy", "goal": "maximize"},
    "parameters": {
        "learning_rate": {"distribution": "log_uniform_values", "min": 1e-5, "max": 1e-1},
        "batch_size": {"values": [32, 64, 128]},
        "dropout": {"distribution": "uniform", "min": 0.1, "max": 0.5}
    }
}

W&B Artifacts — Dataset and Model Versioning

# Versioning a dataset as a W&B Artifact
with wandb.init(project="data_pipeline") as run:
    dataset_artifact = wandb.Artifact(
        "customer_churn_dataset",
        type="dataset",
        description="Processed churn dataset v3 — includes new loyalty features",
        metadata={"num_rows": 120000, "features": 45, "split": "80/10/10"}
    )
    dataset_artifact.add_dir("data/processed/")
    run.log_artifact(dataset_artifact)

# In the training run, declare what input artifact you used
with wandb.init(project="image_classification") as run:
    # This creates a DAG: training_run → uses → customer_churn_dataset:v3
    artifact = run.use_artifact("customer_churn_dataset:v3")
    data_dir = artifact.download()

W&B Reports for Team Communication

W&B Reports are shareable, editable documents that embed live charts from your experiments. Teams use them to document findings, share results with stakeholders, and keep a lab notebook of model development progress — all without needing a separate presentation tool.

Best Practices

  • Set your W&B API key as the environment variable WANDB_API_KEY rather than calling wandb.login() in code — keeps credentials out of your notebooks.
  • Use wandb.config to define all hyperparameters at run init, not as variables in code. This ensures W&B captures them even if the run crashes mid-training.
  • Use Artifacts for datasets — this creates a full lineage graph showing exactly which dataset version produced which model version.
  • Tag runs (run.tags = ["baseline", "resnet50"]) to make filtering in the dashboard easier.
