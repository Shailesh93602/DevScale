Global State Management with Zustand

In mobile applications, data frequently needs to be shared across completely independent navigation screens — a user profile photo on a Drawer menu needs to update immediately when a user changes it in their Settings screen. React's built-in Context API causes unnecessary re-renders when managing complex global state. For modern React Native apps, Zustand provides an incredibly fast, minimalist global store without the dense boilerplate of Redux.

Creating a Zustand Store

Zustand stores are just custom React Hooks. You define a state object and the functions (actions) that mutate it in a single file:

npx expo install zustand
// store/useAuthStore.ts
import { create } from 'zustand';

// 1. Define the store's typescript shape
interface AuthState {
  user: { id: string; name: string; email: string } | null;
  isAuthenticated: boolean;
  token: string | null;
  
  // Actions
  login: (userData: AuthState['user'], token: string) => void;
  logout: () => void;
  updateProfileName: (newName: string) => void;
}

// 2. Create the store implementation
export const useAuthStore = create<AuthState>()((set) => ({
  user: null,
  isAuthenticated: false,
  token: null,

  // Action: Replace state completely
  login: (user, token) => set({ user, token, isAuthenticated: true }),

  // Action: Clear state
  logout: () => set({ user: null, token: null, isAuthenticated: false }),

  // Action: Merge nested state — zustand handles partial updates
  updateProfileName: (newName) => 
    set((state) => ({
      user: state.user ? { ...state.user, name: newName } : null
    }))
}));

Using the Store in Components

Unlike Context, there is no Provider to wrap your app in. Any component can extract just the state fragment it needs. Zustand ensures the component only re-renders if that specific extracted piece of data changes:

// screens/ProfileScreen.tsx
import { View, Text, Button } from 'react-native';
import { useAuthStore } from '../store/useAuthStore';

export default function ProfileScreen() {
  // Extract just the user object — component ignores token changes
  const user = useAuthStore((state) => state.user);
  
  // Extract actions separately (or together)
  const logout = useAuthStore((state) => state.logout);

  if (!user) return You are completely logged out.;

  return (
    <View style={{ padding: 20 }}>
      <Text style={{ fontSize: 24, fontWeight: 'bold' }}>Hello, {user.name}</Text>
      <Text style={{ color: 'gray' }}>{user.email}</Text>

      <Button title="Log Out" color="red" onPress={logout} />
    </View>
  );
}

Handling Async Actions

Zustand has no concept of "middleware" or "thunks" for asynchronous logic — you just use standard ES6 async/await functions inside your store configuration:

// store/useRepoStore.ts
interface RepoState {
  repos: string[];
  isLoading: boolean;
  error: string | null;
  fetchRepos: (username: string) => Promise<void>;
}

export const useRepoStore = create<RepoState>()((set) => ({
  repos: [],
  isLoading: false,
  error: null,

  fetchRepos: async (username: string) => {
    // 1. Set loading state before network request
    set({ isLoading: true, error: null });

    try {
      const resp = await fetch(`https://api.github.com/users/${username}/repos`);
      if (!resp.ok) throw new Error('Failed to fetch repositories');
      
      const data = await resp.json();
      const names = data.map((r: any) => r.name);

      // 2. Set successful data
      set({ repos: names, isLoading: false });
    } catch (err: any) {
      // 3. Set error state
      set({ error: err.message, isLoading: false });
    }
  }
}));

Persisting State with AsyncStorage

Normally, when a user closes their app, all memory state is lost. You want preferences like Dark Mode or Authentication state to persist to disk. Zustand offers a native Persist middleware that integrates instantly with React Native's AsyncStorage:

npx expo install @react-native-async-storage/async-storage
// store/useSettingsStore.ts
import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';
import AsyncStorage from '@react-native-async-storage/async-storage';

interface SettingsState {
  theme: 'light' | 'dark' | 'system';
  notificationsEnabled: boolean;
  setTheme: (t: 'light' | 'dark' | 'system') => void;
  toggleNotifications: () => void;
}

export const useSettingsStore = create<SettingsState>()(
  // Wrap your setup function in the "persist" middleware
  persist(
    (set, get) => ({
      theme: 'system',
      notificationsEnabled: true,
      
      setTheme: (theme) => set({ theme }),
      toggleNotifications: () => 
        set({ notificationsEnabled: !get().notificationsEnabled })
    }),
    {
      name: 'app-settings-storage', // The unique key used in AsyncStorage
      storage: createJSONStorage(() => AsyncStorage), // Define mobile disk engine
      
      // Optional: Filter out things you don't want saved to disk
      // e.g. "Don't save theme to disk, only save notifications"
      // partialize: (state) => ({ notificationsEnabled: state.notificationsEnabled }),
    }
  )
);

The next time the app opens, Zustand automatically reads app-settings-storage off the device disk and hydrates its memory state. Your components will immediately render with the correct persisted values.

Best Practices

  • Do not put standard navigation parameters in global state (e.g. currentProductID). These belong in React Navigation route params.
  • Zustand actions can be called completely outside of React components. Need to log someone out when an Axios interceptor detects a 401 error? Just call useAuthStore.getState().logout() from your API service.
  • Avoid huge monolithic stores. Zustand allows you to split concerns easily: have a useAuthStore, a useCartStore, and a useSettingsStore.
