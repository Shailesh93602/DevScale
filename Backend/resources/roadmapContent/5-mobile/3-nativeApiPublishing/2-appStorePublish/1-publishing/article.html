App Store Publishing with Expo EAS

Building a React Native app is only half the battle. To reach your users, you must publish it to the Apple App Store and Google Play Store. This involves compiling native binaries, managing cryptographic certificates, and navigating each store's submission process. Expo Application Services (EAS) makes this manageable from the command line.

Step 1: EAS Build — Compiling Native Binaries

The stores don't accept JavaScript. You need:

  • .ipa (iOS App Package) for the Apple App Store
  • .aab (Android App Bundle) for Google Play

Building an .ipa normally requires a Mac with Xcode installed. EAS Build removes this constraint — it runs the compilation on Expo's cloud servers and e-mails you the finished binary.

# Install the EAS CLI globally
npm install -g eas-cli

# Log in (create a free account at expo.dev first)
eas login

# Initialize EAS in your project
eas build:configure

# Build for both stores and run the native simulator
# profile: "preview" generates a debug .apk for Android sideloading
eas build --platform all --profile production

# Build for a single platform
eas build --platform ios --profile production
eas build --platform android --profile production

The build profiles are defined in eas.json:

{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "distribution": "internal",
      "android": { "buildType": "apk" }
    },
    "production": {
      "autoIncrement": true
    }
  }
}

Step 2: iOS Code Signing — Certificates & Provisioning Profiles

Apple requires cryptographic proof of your identity before allowing any app onto their platform. This involves two documents:

  1. Distribution Certificate — a key pair that proves you are a registered Apple Developer. Generated once per developer account, valid for 1 year.
  2. App Store Provisioning Profile — a document that ties your App ID (e.g., com.yourcompany.appname) to your Distribution Certificate, authorizing it to be distributed via the App Store.

EAS handles this automatically via eas credentials. Run this once and it creates and stores the certificates in your Expo account, ready for every future build:

eas credentials

Step 3: App Store Connect Setup (iOS)

Before submitting, create your app record in App Store Connect:

  1. Click My Apps → + → New App
  2. Set the Bundle ID (must match ios.bundleIdentifier in your app.json)
  3. Fill in App Name, Primary Language, SKU (a unique internal identifier)
  4. On the App Information page, upload at minimum: 6.5" iPhone screenshots, App Icon (1024×1024 px), Privacy Policy URL, and an Age Rating questionnaire

Step 4: Google Play Store Setup (Android)

  1. Pay the one-time $25 developer registration fee at play.google.com/console
  2. Create a new app → Declarations (accept policies, set app category)
  3. Upload a Feature Graphic (1024×500 px) and at least 2 phone screenshots
  4. Set the Content Rating questionnaire (required by all stores)

Step 5: EAS Submit — Automated Upload

Once your build is complete, EAS Submit uploads it to the stores automatically:

# Submit the latest build to both stores
eas submit --platform all

# Submit a specific build ID
eas submit --platform ios --id <build-id>

# For iOS, you need an App Store Connect API key:
# Apple Developer Portal → Users & Access → Keys → Generate API Key
# Set in eas.json or provide as environment variable

For Android, you need a Google Play JSON service account key (created in Google Cloud Console, then linked in Google Play Console under "Release → Setup → API access").

Step 6: Over-the-Air (OTA) Updates

For JavaScript-only changes (bug fixes, text copy, business logic), you don't need a full store resubmission. EAS Update pushes the new JS bundle directly to installed apps on next launch:

# Push an OTA update to the "production" branch
eas update --branch production --message "Fix checkout bug"

OTA updates cannot change native code (new native packages, permissions, app icon). Those require a full EAS Build and store resubmission.

Best Practices

  • Always test on a real device before submitting — simulators don't surface crashes caused by native permissions or hardware differences.
  • Apple reviews take 1–3 days for the first submission; subsequent updates typically 24 hours. Submit well before your launch deadline.
  • Store all sensitive credentials (API keys, signing certificates) in EAS Secrets, never in your Git repository.
  • Use expo-updates channel branching: production for live users, staging for QA, to avoid accidental OTA updates to production users.
