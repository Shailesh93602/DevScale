Push Notifications: Expo, APNs & FCM

Push notifications are the most powerful re-engagement tool in mobile development — they let your server wake up a user's device even when the app is closed. Implementing them correctly requires understanding the three-party architecture involving your backend, Apple/Google's notification services, and the user's device.

The Core Architecture

You cannot push a notification directly from your server to a user's phone — all push traffic must route through the official platform services:

  • APNs (Apple Push Notification service) — Apple's servers that deliver notifications to iOS devices
  • FCM (Firebase Cloud Messaging) — Google's service that delivers notifications to Android devices

The flow for every notification:

  1. User installs your app → grants notification permission
  2. Device requests a unique push token from APNs/FCM
  3. Your app sends that token to your backend and stores it against the user's account
  4. When an event occurs (new chat message, order shipped), your backend sends the notification payload + token to APNs/FCM
  5. APNs/FCM delivers the notification to the physical device

Using Expo Notifications (Recommended Approach)

Instead of managing APNs certificates and FCM keys separately, Expo Notifications abstracts both behind a single unified API. Your backend sends all notifications to Expo's Push API, which forwards them to the correct platform service automatically.

Step 2: Register for Push Notifications (Client)

// hooks/usePushNotifications.ts
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';
import { Platform } from 'react-native';

// How notifications behave when the app is in the foreground
Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
    shouldSetBadge: false,
  }),
});

export async function registerForPushNotificationsAsync(): Promise<string | null> {
  // Push notifications only work on physical devices
  if (!Device.isDevice) {
    alert('Push notifications require a physical device');
    return null;
  }

  // Check existing permissions
  const { status: existingStatus } = await Notifications.getPermissionsAsync();
  let finalStatus = existingStatus;

  // Request permission if not already granted
  if (existingStatus !== 'granted') {
    const { status } = await Notifications.requestPermissionsAsync();
    finalStatus = status;
  }

  if (finalStatus !== 'granted') {
    return null; // User denied — handle gracefully in UI
  }

  // Android requires a notification channel
  if (Platform.OS === 'android') {
    await Notifications.setNotificationChannelAsync('default', {
      name: 'Default',
      importance: Notifications.AndroidImportance.MAX,
      vibrationPattern: [0, 250, 250, 250],
      lightColor: '#6366f1',
    });
  }

  // Get the Expo Push Token (this is what you store in your database)
  const token = await Notifications.getExpoPushTokenAsync({
    projectId: 'your-expo-project-id', // From app.json or expo.dev
  });

  return token.data; // e.g., "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]"
}

Step 3: Save Token & Handle Notifications (Client)

// App.tsx or root component
import { useEffect, useRef } from 'react';
import * as Notifications from 'expo-notifications';
import { registerForPushNotificationsAsync } from './hooks/usePushNotifications';

export default function App() {
  const notificationListener = useRef<Notifications.Subscription>();
  const responseListener = useRef<Notifications.Subscription>();

  useEffect(() => {
    registerForPushNotificationsAsync().then(async (token) => {
      if (token) {
        // Save to your backend — associate with the logged-in user
        await fetch('https://api.yourapp.com/user/push-token', {
          method: 'POST',
          headers: { 'Authorization': `Bearer ${userJwt}`, 'Content-Type': 'application/json' },
          body: JSON.stringify({ token, platform: Platform.OS }),
        });
      }
    });

    // Fires when a notification arrives while the app is open
    notificationListener.current = Notifications.addNotificationReceivedListener(notification => {
      console.log('Notification received:', notification.request.content);
    });

    // Fires when the user taps a notification (foreground OR background)
    responseListener.current = Notifications.addNotificationResponseReceivedListener(response => {
      const data = response.notification.request.content.data;
      // Navigate to the relevant screen based on notification data
      if (data.type === 'order_shipped') navigation.navigate('OrderDetails', { orderId: data.orderId });
    });

    return () => {
      Notifications.removeNotificationSubscription(notificationListener.current!);
      Notifications.removeNotificationSubscription(responseListener.current!);
    };
  }, []);
}

Step 4: Sending Notifications (Backend — Node.js)

// Using the official Expo Server SDK
// npm install expo-server-sdk

const { Expo } = require('expo-server-sdk');
const expo = new Expo();

async function sendPushNotification(expoPushToken: string, title: string, body: string, data = {}) {
  // Validate the token before trying to send
  if (!Expo.isExpoPushToken(expoPushToken)) {
    console.error(`Invalid Expo push token: ${expoPushToken}`);
    return;
  }

  const messages = [{
    to: expoPushToken,
    sound: 'default',
    title,
    body,
    data,                    // Custom payload — accessible in addNotificationResponseReceivedListener
    badge: 1,                // iOS badge count
    priority: 'high',        // Ensures delivery even if device is in low-power mode
    channelId: 'default',    // Android notification channel
  }];

  // Expo batches up to 100 messages per API call
  const chunks = expo.chunkPushNotifications(messages);
  for (const chunk of chunks) {
    try {
      const tickets = await expo.sendPushNotificationsAsync(chunk);
      // Check tickets for DeviceNotRegistered errors → remove invalid tokens from DB
      tickets.forEach(ticket => {
        if (ticket.status === 'error' && ticket.details?.error === 'DeviceNotRegistered') {
          // Remove this token from your database
        }
      });
    } catch (error) {
      console.error('Error sending push notification:', error);
    }
  }
}

// Usage — triggered by a database event (e.g., new order)
await sendPushNotification(
  user.expoPushToken,
  'Order Shipped! 🚚',
  'Your order #1234 has been dispatched and will arrive in 2-3 days.',
  { type: 'order_shipped', orderId: '1234' }
);

Local Notifications (No Server Required)

For reminders and scheduled alerts that don't need a server trigger:

// Schedule a notification 5 seconds from now
await Notifications.scheduleNotificationAsync({
  content: {
    title: "Don't forget! ⏰",
    body: 'Your daily workout reminder',
    data: { screen: 'Workout' },
  },
  trigger: {
    seconds: 5,    // Or use a Date object for exact time scheduling
  },
});

// Cancel all pending scheduled notifications
await Notifications.cancelAllScheduledNotificationsAsync();

Best Practices

  • Always handle the case where the user denies notification permission — never block the app, just hide notification-related features gracefully.
  • Store multiple tokens per user (one per device/installation) — users often have multiple phones or reinstall the app.
  • Purge DeviceNotRegistered tokens from your database immediately — expired tokens waste API quota and slow down sends.
  • Use data.type in your payload to categorize notifications (order updates, chat messages, promotions) and deep-link users to the correct screen.
  • For high-volume sends (100,000+ users), use a queue (Bull, BullMQ) with workers to avoid blocking your API server during mass notification blasts.
