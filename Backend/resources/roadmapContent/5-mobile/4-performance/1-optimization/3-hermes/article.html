Hermes JavaScript Engine in React Native

React Native apps historically ran JavaScript using JavaScriptCore (JSC) — the same engine that powers Safari. In 2019, Meta engineered and open-sourced Hermes, a JavaScript engine purpose-built for running React Native on mobile devices. Hermes is now the default engine for all new React Native and Expo projects because it delivers measurably better performance on real devices — especially on lower-end Android hardware that represents most of the global market.

How Hermes Improves Performance

Traditional JS engines (JSC, V8) compile JavaScript to bytecode at runtime using a Just-In-Time (JIT) compiler. Hermes takes a different approach: it uses Ahead-of-Time (AOT) compilation.

  • Build time: Hermes compiles your JavaScript bundle into Hermes Bytecode (HBC) at build time — the CPU-intensive compilation work happens on the build server, not on the user's phone.
  • App startup: When the user opens the app, Hermes loads pre-compiled bytecode directly, skipping compilation entirely. This drastically reduces Time to Interactive (TTI).
  • Memory efficiency: Hermes is designed with a smaller memory footprint than JSC or V8, which is critical on 2-4GB RAM Android devices.

Verifying Hermes Is Active

import React from 'react';
import { View, Text, StyleSheet } from 'react-native';

const HermesCheck = () => {
  // global.HermesInternal exists only when Hermes is the active engine
  const isHermes = () => !!global.HermesInternal;

  return (
    <View style={styles.container}>
      <Text style={styles.status}>
        JS Engine: {isHermes() ? '✅ Hermes' : '⚠️ JavaScriptCore'}
      </Text>
      {isHermes() && (
        <Text style={styles.info}>
          Hermes Version: {global.HermesInternal?.getRuntimeProperties?.()?.['OSS Release Version']}
        </Text>
      )}
    </View>
  );
};

// You can also check in Metro output during `npx react-native run-android`
// Look for: "JS engine: hermes" in the build output

Enabling Hermes in Expo

// app.json / app.config.js — Hermes is enabled by default in Expo SDK 48+
{
  "expo": {
    "jsEngine": "hermes"
  }
}

// To verify in Expo Go / custom dev client
import { Platform } from 'react-native';
console.log('Hermes enabled:', Platform.constants?.reactNativeVersion);

Enabling/Managing Hermes in Bare React Native

// android/gradle.properties
hermesEnabled=true    // This is the default in RN 0.70+

// ios/Podfile — also enabled by default in recent RN versions
:hermes_enabled => true

Hermes and Proguard (Android)

For production Android builds, Hermes works with Proguard (code shrinking/obfuscation). In release builds, your JS is compiled to Hermes bytecode, then the native Hermes binary is stripped down by Proguard — further reducing APK size. You don't usually need to configure this manually, but knowing it exists helps when debugging "release works different from debug" issues.

Debugging with Hermes

Hermes ships with its own Chrome-compatible DevTools integration. When you open the in-app dev menu and tap "Open Debugger", it connects directly to Hermes via a Chrome DevTools Protocol (CDP) server — you get the full Chrome debugger experience (breakpoints, call stacks, memory snapshots) rather than the older Remote JS Debugging which sent all code across the network:

// Check Hermes runtime properties programmatically
const properties = global.HermesInternal?.getRuntimeProperties();
console.log(properties);
// → {
//     "OSS Release Version": "for RN 0.73",
//     "Build": "Release",
//     "GC": "hades"        ← Concurrent GC (low-pause garbage collector)
//   }

Hermes Feature Support

A common concern is JavaScript feature parity. Hermes fully supports:

  • ES2022 features (optional chaining, nullish coalescing, async/await, generators)
  • CommonJS and ESM module systems
  • Full React Hooks and Context API
  • TypeScript (after transpilation by Babel/Metro)

Exceptions: Hermes does NOT support eval() or new Function() — both are security risks and performance anti-patterns. If a third-party library uses these, it will break with Hermes.

Best Practices

  • Keep Hermes enabled at all times — there is no reason to disable it for a standard React Native app.
  • Profile startup time with and without Hermes using the Flipper Performance plugin to quantify the improvement in your specific app.
  • If a library breaks with Hermes (due to eval() usage), file a bug with the library maintainer — eval() is an anti-pattern in any environment.
  • For release builds, always test on a low-end Android device (2GB RAM) — Hermes's benefits are most visible there.
