Content Security Policy (CSP): Hardening Your Website Against Attacks

A Content Security Policy (CSP) is a powerful, extra layer of security that helps detect and mitigate certain types of attacks, such as Cross-Site Scripting (XSS) and data injection. It is essentially an "allowlist" that tells the browser which sources of content (scripts, styles, images) are trusted and should be executed.

How CSP Is Delivered

The policy can be sent as an HTTP response header or as a <meta> tag in the HTML <head>.

<meta http-equiv="Content-Security-Policy" content="default-src 'self';">

Directives: Controlling Your Content

  • default-src: The "fallback" if a more specific directive is not defined.
  • script-src: Specifies which domains the browser can download scripts from.
  • style-src: Specifies which domains the browser can download styles from.
  • img-src: Controls where images can be loaded from.

Blocking Inline Scripts

By default, a strict CSP will **block all inline JavaScript** (like <script>alert(1)</script> or onclick="..."). This is a major security benefit because it stops an attacker from injecting their own malicious scripts into your page.

Nonces and Hashes

If you really need an inline script to work with a CSP, you can use a "nonce" (a unique, random string) or a cryptographic "hash" of the script's contents. Only scripts matching the nonce or hash will be allowed to run.

Reporting: CSP-Report-Only

Before enforcing a strict policy, you can use the Content-Security-Policy-Report-Only header. This will not block anything, but it will send a report to a specified URL every time a potential violation occurs, allowing you to test your policy before it breaks your site.

💡 Pro Tip: Use a tool like the **CSP Evaluator** (by Google) to check if your policy has any common holes or bypasses. A poorly-written CSP can give you a false sense of security while actually being easy to defeat.
