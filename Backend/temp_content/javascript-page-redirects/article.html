JavaScript Page Redirects: Navigating Programmatically

Sometimes you need to send a user to a different page automatically—for example, after they successfully log in or when they try to access a page they don't have permission to see. This is called a **Redirect**.

1. The location Object

This is the most common way to handle redirects. The window.location object contains information about the current page and its URL.

  • location.href: The most popular choice. It's like the user clicking a link. It adds the current page to the browser's "back" button history.
  • location.replace(): Similar to href, but it **removes** the current page from the history. This is often used for login pages so the user can't "Logout" and then click "Back" to see their private data.
  • location.assign(): The same as href, but it's a function.

2. Redirecting After a Delay

You can use setTimeout() to show the user a message (e.g., "Redirecting in 3 seconds...") before sending them to the next page.

setTimeout(() => {
  window.location.href = "https://eduscale.com/dashboard";
}, 3000);

3. Handling "Invalid" Pages

If a user tries to access a page that doesn't exist, you might want to redirect them to your 404 page or your homepage automatically.

Performance: Client vs. Server Redirects

A JavaScript redirect happens in the browser, meaning the "invalid" page has to load first. For the best performance (and SEO), you should always try to handle redirects on the **Server** (using HTTP status codes like 301 or 302) if possible.

💡 Pro Tip: Use **window.location.replace()** for your login and signup flows. It provides a much cleaner user experience by preventing the "Back Button Trap" where a user gets stuck in a loop of redirecting pages.
