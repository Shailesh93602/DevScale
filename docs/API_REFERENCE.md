# API Reference & Interception Playbook

Welcome to the **EduScale RESTful API** and WebSocket integration documentation.

Our API revolves around strict REST architecture where endpoints describe resources: Users, Roadmaps, Battles, and Portfolios. We expect clients to send properly formatted JSON objects containing Authorization schema where specified.

---

## 1. Authentication Mechanism
EduScale utilizes JWTs generated by the Supabase Auth Gateway, which are transmitted securely via HTTP-Only Cookies or Authorization headers. 

**Format:**
```http
Authorization: Bearer <Access_Token>
```
The token maps directly to backend permissions through our custom Rate Limiting & Auth middleware. Always verify `token_expiration` prior to launching intensive operations.

---

## 2. API Endpoints

### 2.1. Authorization
`POST /api/v1/auth/login` - Initialize session and establish Redis map.
`POST /api/v1/auth/register` - Create user metadata and associate identity providers.
`POST /api/v1/auth/refresh` - Swap an expired Access Token using your active Refresh Token.
`GET /api/v1/auth/me` - Validates bearer token and returns serialized user DTO.

### 2.2. Users & Portfolio
`GET /api/v1/users/:id` - Fetch public user profile and aggregated XP (Experience Points).
`PUT /api/v1/users/:id` - Update user bio, handles, and avatars (Cloudinary).
`GET /api/v1/users/:id/progress` - Returns multidimensional array of roadmap completions.

### 2.3. Learning Roadmaps
`GET /api/v1/roadmaps` - List available macro-roadmaps (e.g., Frontend, Backend, Machine Learning).
`GET /api/v1/roadmaps/:id` - Returns deep nested tree (Topics -> Modules -> Lessons -> Resources).
`POST /api/v1/progress/topic/:topic_id` - Mark module as complete and award user point increments.

### 2.4. Battle Zone (REST & WebSocket)
The Battle Zone operates hybrid. REST endpoints manage matchmaking lobbies, whereas WebSocket lines calculate latency-sensitive keystrokes.
`POST /api/v1/battles/create` - Initiate a matchmaking queue.
`GET /api/v1/battles/active` - List lobbies requiring opponents.
`GET /api/v1/battles/:battle_id/leaderboard` - Reconstruct post-battle results from PostgreSQL and Redis delta.

## 3. WebSocket Integration (`socket.io`)

The WebSocket line uses specific event nomenclatures heavily optimized for minimal packet payloads. Ensure the `ws[s]://` protocol targets the root endpoint.

**Connection Payload:**
```json
{
  "token": "<Access_Token>"
}
```

### Event: `battle:join`
Client connects to an active battle environment. Emitted by the Client (`socket.emit()`); Backend assigns Room Subscription and broadcasts `battle:participant_join` to peers inside `battle:id` room.

### Event: `battle:state_update`
Emitted asynchronously by Backend whenever an active countdown timer dictates changing application state (Lobby -> Preparation -> In-Game -> Review).

### Event: `battle:score_update`
Backend dispatches real-time rank matrices as competitors submit valid AST passes against test cases.

## 4. Rate Limiting Guidelines
We employ strict 15-minute sliding windows across global edge nodes using BullMQ structures:
- `Auth Actions`: 5 requests / 15-minute window / IP.
- `Standard Reads`: 1,000 requests / 15-minute window / IP.
- `Battle Submission`: 10 requests / 1-minute window / User ID (to prevent brute-force compilation payloads).

If breached, server will return HTTP `429 Too Many Requests`.
